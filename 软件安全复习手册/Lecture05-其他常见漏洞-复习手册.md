### 本讲主线

Lecture05 补充非典型缓冲区溢出的漏洞：格式化字符串、整数溢出、C++ 虚函数攻击，以及注入类、权限类等其他漏洞。

### 格式化字符串漏洞

安全写法：

```c
printf("%s", user_input);
```

危险写法：

```c
printf(user_input);
```

问题本质是“通道混淆”：程序员本来想把用户输入当作普通数据打印，却让它变成了格式控制字符串。

常见格式符：

- `%x`：读取栈上的十六进制数据，可造成信息泄漏。
- `%p`：按指针形式输出地址，可用于泄露地址信息。
- `%s`：把某个值当作地址，再从该地址读取字符串。
- `%n`：把已输出字符数写入指定地址，可造成内存写。

因此格式化字符串漏洞既可能读内存，也可能写内存。它甚至可用于泄露地址以绕过 ASLR。

课堂例子中，如果输入导致栈上出现 `0x41414141`，再使用 `%s`，就可能从地址 `0x41414141` 读取字符串。能写入数据的格式化符是 `%n`。

格式化字符串漏洞的关键不是“printf 这个函数危险”，而是“格式串被攻击者控制”。`printf("%s", user_input)` 中，格式串是程序员固定写死的，用户输入只是普通参数；`printf(user_input)` 中，用户输入变成解释规则，`printf` 会把 `%x`、`%s`、`%n` 当指令执行。

它常见的攻击能力分三层：

1. 读栈：输入 `%x %x %x`，让程序把本来不该输出的栈内容打印出来。
2. 读任意地址附近内容：利用 `%s` 把某个值当地址读取字符串。
3. 写内存：利用 `%n` 把已输出字符数写到指定地址。

所以格式化字符串漏洞往往先用于信息泄漏，再用于绕过 ASLR 或配合其他漏洞。它和缓冲区溢出的共同点是都来自错误信任外部输入；区别是缓冲区溢出通常是“写太多”，格式化字符串是“把数据当解释规则”。

#### `printf` 为什么会多读栈

`printf`、`fprintf`、`sprintf`、`snprintf` 等 `printf` 族函数支持可变参数。函数本身不知道调用者实际传了多少参数，它会根据格式字符串中的 `%x`、`%s`、`%d` 等格式符，决定接下来从参数区域取多少值、按什么类型解释。

如果程序写成：

```c
printf("%s %d %d %d %x\n", buf, a, b, c);
```

格式串里多了一个 `%x`，但后面没有提供对应参数。`printf` 仍会按照格式串继续取值，于是它会从调用上下文中取出后续栈内容并打印。PPT 中用这个例子说明：格式化字符串漏洞不是凭空“生成数据”，而是把本来在栈上的残留数据、保存的寄存器、旧栈数据、地址值等按格式解释出来。

这些泄漏很有价值，因为打印出来的值可能包含返回地址、栈地址、堆地址、libc 或模块地址、stack canary、PIE 基址等。它们可以帮助攻击者绕过 ASLR、PIE 或栈保护。

#### `AAAA%x...` 和 `0x41414141`

PPT 中的例子：

```c
char str[200];
fgets(str, 200, stdin);
printf(str);
```

输入：

```text
AAAA%x%x%x%x
```

输出中出现：

```text
AAAA18FE84BB40603041414141
```

`0x41` 是字符 `A` 的 ASCII 值，因此 `41414141` 对应四个 `A`。这说明攻击者输入的内容已经出现在 `printf` 后续取参数的位置附近。如果把最后一个 `%x` 改成 `%s`，`printf` 会把取到的 `0x41414141` 当作地址，再从地址 `0x41414141` 读取字符串。正确理解是：`%s` 不是读出数值 `0x41414141`，而是把这个值解释为地址。

#### `%n` 和写入字符数

`%n` 的作用是把“到目前为止已经输出的字符数”写入对应参数指向的整型变量。例如：

```c
printf("Jamsa%n", &first_count);
```

输出到 `%n` 前已经打印了 5 个字符，所以 `first_count` 会被写入 5。

PPT 中还有一个 `sprintf` 示例：

```c
int formatstring_func2(int argc, char *argv[])
{
    char buffer[100];
    sprintf(buffer, argv[1]);
}
```

如果命令行参数是：

```text
aaaabbbbcc%n
```

`sprintf` 会先把 `aaaabbbbcc` 写入 `buffer`，到 `%n` 时已经输出了 10 个字符。由于程序没有提供 `%n` 对应的参数，函数会继续从当前上下文中取值作为目标地址。课件示意中，前四个字节 `aaaa` 被解释成地址 `0x61616161`，于是数值 10 会被写入该地址。

为了写入更大的数，攻击者还可以利用打印宽度。例如：

```c
printf("%100d%n\n", num, &num);
```

`%100d` 会让输出宽度至少达到 100，于是 `%n` 写入的值可以变成 100。`%02333d` 这类形式则通过补零控制输出宽度。考试中不需要手工构造完整 exploit，但要知道“输出字符数可被格式宽度影响，进而影响 `%n` 写入的值”。

### 整数溢出

整数溢出是数值运算超出类型能表示的范围，发生回绕或符号变化。它通常不单独完成攻击，而是绕过长度检查、边界判断或内存分配判断，进而引发缓冲区溢出等漏洞。

典型模式：

- 长度相加后变小。
- 有符号/无符号转换导致判断失效。
- 乘法计算分配大小时溢出，实际分配空间小于后续写入大小。
- 大整数存入小类型导致高位截断。

整数溢出的危险在于：程序员以为自己检查了长度，但检查基于错误的计算结果。例如：

```c
int size = count * 4;
buf = malloc(size);
read(fd, buf, count * 4);
```

如果 `count * 4` 超过 `int` 能表示的范围，`size` 可能回绕成很小的正数。于是 `malloc` 分配很小空间，但后续逻辑仍按很大的数据量写入，最终造成堆溢出。

常见类型：

- 加法溢出：`len + header` 回绕变小。
- 乘法溢出：`count * element_size` 回绕。
- 符号转换：负数转成无符号数后变成很大的数。
- 截断：大整数赋给小整数类型，高位丢失。

考试里如果问“整数溢出是不是内存覆盖型溢出”，一般要区分：整数溢出本身是数值错误，不是直接写越界；但它常常导致后续缓冲区溢出、越界访问或权限判断绕过。

#### `0xFFFFFFFF + 1` 示例

PPT 中有一个典型代码：

```c
char* integer_overflow(int* data, unsigned int len)
{
    unsigned int size = len + 1;
    char *buffer = (char*)malloc(size);
    if (!buffer)
        return NULL;
    memcpy(buffer, data, len);
    buffer[len] = '\0';
    return buffer;
}
```

如果攻击者传入 `len = 0xFFFFFFFF`，在 32 位无符号整数中，`len + 1` 会回绕成 `0`。于是 `malloc(size)` 申请的可能是一个极小空间或大小为 0 的块，但后续 `memcpy(buffer, data, len)` 仍然按照非常大的 `len` 拷贝数据，最终造成堆溢出。

因果链可以这样记：

```text
整数回绕 -> 分配大小错误 -> 后续拷贝仍按原长度执行 -> 堆缓冲区被写爆
```

#### `short` 加法绕过长度检查

另一个课堂例子中，程序大致逻辑是：

```c
#define MAX_INFO 32767

void (*fuc_ptr)() = func;
char info[MAX_INFO];
char info1[30000];
char info2[30000];

short len1 = strlen(info1);
short len2 = strlen(info2);
short all_len = len1 + len2;

if (all_len < MAX_INFO) {
    strcpy(info, info1);
    strcat(info, info2);
}

fuc_ptr();
```

`short` 的表示范围有限。若 `len1 + len2` 超过 `short` 能表示的最大正数，`all_len` 可能变成负数。负数满足 `all_len < MAX_INFO`，于是程序进入分支，把两段很长的数据拼接进 `info`。这时前面的长度检查反而成了“被绕过的检查”，后续 `strcpy`/`strcat` 会造成缓冲区溢出。

如果 `info` 附近有函数指针 `fuc_ptr`，攻击者可能通过溢出覆盖它。后续执行 `fuc_ptr()` 时，程序会跳到被篡改的地址。这说明整数溢出往往不是终点，而是打开后续内存破坏的入口。

### C++ 虚函数攻击

C++ 对象中通常有虚表指针 vptr，指向虚函数表 vtable。调用虚函数时，程序通过 vptr 找到 vtable，再取出函数地址调用。

如果攻击者能通过溢出修改对象内存中的 vptr 或虚函数指针，就可能让虚函数调用跳到攻击者指定位置。

易考：虚表指针保存在对象的内存空间中。

C++ 虚函数攻击属于“间接控制流劫持”。它不一定覆盖返回地址，而是覆盖对象中的指针。

可以这样理解：

```text
C++ 对象
┌─────────────┐
│ vptr        │ -> vtable -> 虚函数1地址 / 虚函数2地址 / ...
├─────────────┤
│ 普通字段    │
└─────────────┘
```

当程序执行 `obj->virtual_func()` 时，机器层面不是直接调用固定地址，而是先从对象里取 vptr，再从 vtable 里取函数地址，最后间接跳转。如果溢出覆盖了 vptr，让它指向攻击者伪造的虚表，后续虚函数调用就可能跳到攻击者选择的位置。

这类攻击说明：控制流数据不只有返回地址。函数指针、回调指针、虚表指针、异常处理指针都可能成为目标。

PPT 示例中，`Failwest` 类包含 `char buf[200]` 和虚函数 `test()`。对象的虚表指针位于对象内存中，示例通过 `overflow.buf - 4` 定位虚表指针，再让虚表相关位置指向保存 shellcode 的 `overflow.buf`。当程序执行 `p->test()` 时，虚函数调用会通过被篡改的 vptr/vtable 间接跳转。

考试重点不是记某个具体地址，而是理解攻击策略：

1. 可控缓冲区中放置攻击者希望执行或跳转的数据。
2. 修改对象中的 vptr，让它指向攻击者布置的伪造虚表。
3. 修改伪造虚表中的函数指针，让虚函数调用跳到目标地址。
4. 触发虚函数调用，完成间接控制流劫持。

### 注入类漏洞

注入类攻击的共同点：外部输入被当作代码、命令或解释器语法的一部分执行。

常见类型：

- SQL 注入：输入被拼接进 SQL 查询。
- 操作系统命令注入：输入被拼接进系统命令。
- 动态代码执行：输入被拼接进可执行代码。
- 文件包含：用户控制包含哪个文件。
- XML/SOAP 注入：XML 是解释性结构，缺乏检查时可破坏消息结构。

SOAP 协议存在注入风险的原因包括 XML 是解释性语言，以及缺乏对 XML 消息恶意输入的检查。

按应用工作方式，课件把代码注入分成两类：

| 类型 | 含义 | 例子 |
|---|---|---|
| 二进制代码注入 | 外部二进制代码获得执行机会 | 缓冲区溢出后执行 shellcode |
| 脚本注入 | 提交可被解释执行的数据 | SQL 注入、脚本语言动态执行、SOAP/XML 注入 |

SQL 注入的典型路径是：Web 页面输入参数、表单域或数据包字段被拼接成 SQL 语句，传给 Web 服务器，再传给数据库服务器执行。问题的根源仍然是数据通道和控制通道混淆：用户输入本应是数据，却进入了 SQL 语法结构，改变了查询逻辑，例如构造永真条件绕过认证。

操作系统命令注入常出现在 Web 应用调用系统命令时。PHP 中的 `system`、`exec`，ASP 中的 `wscript` 等能力本身可以用于正常功能；但如果应用把用户输入拼接进命令行，又缺少校验，就可能让攻击者执行非预期命令。这类问题在设备管理后台、防火墙、打印机、路由器管理界面等场景中尤其需要警惕。

Web 脚本语言注入主要有两条路线：

- 动态执行：把用户输入合并进运行时生成的代码，再交给解释器执行。
- 动态包含：根据用户输入决定包含哪个代码文件，若路径或文件可控，就可能执行攻击者代码。

SOAP 是 Simple Object Access Protocol，简单对象访问协议，基于 XML，常用于 Web 服务组件之间通信。XML 通过 `<`、`>`、`/` 等元字符表达结构；如果用户输入未经处理直接进入 SOAP 消息，就可能破坏 XML 结构，改变应用逻辑。

注入类漏洞可以用统一模型理解：

```text
程序原本想构造一段“命令/查询/页面/路径”
攻击者输入被拼进去
解释器无法区分“程序员写的语法”和“用户输入的数据”
最终语义被攻击者改变
```

比如 SQL 注入中，程序本来想查某个用户：

```sql
select * from users where name='输入'
```

如果输入直接拼接，攻击者可以让它变成恒真条件、追加查询或注释掉后续语句。命令注入、模板注入、文件包含也是同一思想：用户输入进入了某种解释器。

防御的核心是让代码和数据分离：

- SQL 用参数化查询。
- 命令执行避免拼接字符串，使用参数数组或白名单。
- HTML 输出按上下文编码。
- XML 解析关闭危险外部实体和不必要特性。
- 文件包含使用固定映射，不让用户直接控制路径。

### 权限类漏洞

权限漏洞是访问控制逻辑出错。常见两类：

- 垂直越权：普通用户获得管理员功能。
- 水平越权：用户 A 访问或修改用户 B 的资源。

水平越权常见原因是服务端过度相信客户端传入的 `userid`、订单号、对象 ID，而没有检查资源归属。

权限漏洞和内存漏洞不同，它不一定导致崩溃，也不一定有明显报错。它的本质是“该不该允许这个用户做这件事”的判断错了。

垂直越权：权限等级向上突破。普通用户访问管理员接口，或者普通员工调用财务审批接口。

水平越权：同一权限等级内访问别人的资源。用户 A 把 `/order?id=1001` 改成 `/order?id=1002`，如果能看到用户 B 的订单，就是水平越权。

垂直越权还可以分成向上越权和向下越权：

- 向上越权：低权限用户或未授权用户执行高权限用户的操作。
- 向下越权：高权限用户访问低权限用户信息，但业务规则本来不允许这样访问。

水平越权常出现在“一个用户对象关联多个其他对象”的业务中，例如个人资料、修改密码、订单信息等 CRUD 操作。如果服务端根据用户提交的 `userid` 或对象 ID 查询资源，却没有检查资源归属，攻击者就可以修改参数访问别人的数据。

常见根因：

- 只在前端隐藏按钮，后端没有鉴权。
- 服务端相信客户端传来的 `userid`、`role`、`is_admin`。
- 只检查“是否登录”，不检查“是否有权访问这个对象”。
- 接口设计缺少统一权限中间件。

防御重点是服务端强制鉴权，尤其是对象级权限检查。每次访问订单、文件、用户资料、审批记录，都要确认资源属于当前用户或当前角色有权访问。

### 易考点

- `printf(str)` 风险大于 `printf("%s", str)`。
- `%n` 可写内存。
- 整数溢出常用于绕过检查。
- vptr 在对象内存中。
- 注入漏洞本质是输入被解释器当作代码/语法执行。
- `%x` 常用于泄露栈数据，`%s` 可能把值当地址读字符串，`%n` 能写内存。
- `%n` 写入的是到当前位置为止已经输出的字符数，打印宽度会影响写入值。
- 整数溢出不是直接内存覆盖，但可导致缓冲区溢出。
- `0xFFFFFFFF + 1` 在 32 位无符号语境下可能回绕为 0。
- C++ 虚函数调用依赖对象中的 vptr 和虚表 vtable。
- 虚函数攻击的目标通常是篡改 vptr 或虚表项，让间接调用跳到目标地址。
- 水平越权是访问同级别其他用户资源，垂直越权是低权限访问高权限功能。
- 注入漏洞的首选防御通常是结构化 API 或参数化，而不是黑名单过滤。

### 格式化字符串漏洞的核心理解

格式化字符串漏洞和缓冲区溢出一样，都来自程序员对外部输入的错误信任。区别在于：缓冲区溢出通常是“写得太多”，格式化字符串漏洞是“把数据当成格式指令”。`printf` 族函数会根据格式串决定后面应该从栈或寄存器中取多少参数、如何解释这些参数。如果用户输入成为格式串，用户就间接控制了解释规则。

例如程序写成：

```c
char str[200];
fgets(str, 200, stdin);
printf(str);
```

输入普通字符串时看不出问题；输入 `%x %x %x` 时，`printf` 会以为调用者还提供了多个参数，于是从当前调用上下文里继续取值并打印，造成栈内容泄露。输入 `%s` 时，取出的值会被当作地址，再从该地址读取字符串，可能造成敏感内存读取或崩溃。输入 `%n` 时，函数会把“已经输出的字符数”写入对应地址，因此可形成内存写。

安全写法是固定格式串：

```c
printf("%s", str);
```

这里用户输入只作为一个普通字符串参数，不再控制格式解析规则。这个例子背后的通用经验是：解释器或格式化函数的“控制通道”必须由程序自己掌握，不能交给用户输入。

### 格式化字符串的攻击能力

格式化字符串漏洞可能产生三类能力：

1. 信息泄漏：通过 `%x`、`%p`、`%s` 等读取栈、地址或字符串。
2. 地址定位：泄露模块地址、栈地址、canary 等，为绕过 ASLR 和栈保护提供条件。
3. 内存写入：通过 `%n` 系列格式符写指定地址，进一步修改返回地址、函数指针或安全检查变量。

很多时候格式化字符串漏洞先用来“看见内存”，再配合其他漏洞完成控制流劫持。它的危险不只在于崩溃，而在于泄露和写入能力。

### 整数溢出的核心理解

整数溢出不是“内存写越界”，而是“计算结果错了”。程序经常先计算长度，再根据长度分配内存或检查边界。如果长度计算发生溢出，后续所有看似合理的判断都会建立在错误结果上。

典型场景：

```c
int size = count * sizeof(item);
buf = malloc(size);
read(fd, buf, count * sizeof(item));
```

如果 `count * sizeof(item)` 超过整数范围，`size` 可能回绕成很小的值，导致 `malloc` 分配很小空间，而后续读取却按原始逻辑写入大量数据，最终引发堆溢出。

常见类型包括：

- 加法溢出：`len + header_size` 回绕。
- 乘法溢出：`count * element_size` 回绕。
- 有符号/无符号转换：负数被当成很大的无符号数。
- 截断：大整数赋给小类型后高位丢失。

整数溢出通常是“前置漏洞”：它绕过检查或造成错误分配，再引出缓冲区溢出、越界访问、权限绕过等后果。

### C++ 虚函数攻击的核心理解

C++ 的动态派发依赖虚表。对象中保存虚表指针 vptr，vptr 指向虚表 vtable，虚表里保存各个虚函数的真实地址。调用虚函数时，程序不是直接跳到固定函数，而是通过对象里的 vptr 间接找到函数地址。

这带来一个安全风险：如果攻击者通过溢出覆盖对象中的 vptr，让它指向攻击者布置的伪造虚表，那么后续虚函数调用就可能跳到攻击者选择的地址。虚函数攻击的关键不是“虚函数本身危险”，而是“间接调用依赖对象内存中的指针，而对象内存可能被破坏”。

理解虚函数攻击要抓住三步：

1. 对象内存被覆盖。
2. vptr 或虚表项被改写。
3. 程序调用虚函数时发生间接跳转。

这和覆盖返回地址属于同一类思想：攻击者不一定直接控制 `ret`，也可以控制函数指针、虚表指针、回调指针等间接控制流数据。

### 注入类漏洞的统一模型

SQL 注入、命令注入、XML/SOAP 注入、文件包含等看起来形式不同，本质相同：程序把外部输入拼进某种“解释器语言”里，解释器无法区分哪些是程序员本来写的语法，哪些是用户恶意输入。

统一模型：

1. 程序接收外部输入。
2. 程序把输入拼接进 SQL、命令、HTML、XML、脚本或文件路径。
3. 后端解释器解析拼接后的内容。
4. 输入改变了原本语义。

防御的共同原则是把“代码”和“数据”分开。SQL 用参数化查询，HTML 用输出编码，命令执行避免拼接字符串，文件包含使用白名单映射，XML 解析关闭危险特性并做结构化校验。

### 权限漏洞的核心理解

权限漏洞不是内存错误，而是业务逻辑和访问控制错误。系统不能只相信“用户请求里写了什么”，必须在服务端确认“这个用户是否有权操作这个对象”。

水平越权例子：用户 A 请求 `/order?id=1002`，把 `id` 改成用户 B 的订单号后也能查看。问题在于服务端只按订单号查询，没有检查订单是否属于当前登录用户。

垂直越权例子：普通用户访问管理员接口成功。问题可能是接口没有鉴权，或只在前端隐藏按钮，后端没有检查角色。

权限漏洞的防御重点：

- 所有敏感操作都在服务端鉴权。
- 对象级权限检查不能省。
- 不信任客户端传来的用户 ID、角色、价格、权限标志。
- 日志记录越权尝试。
