## 一、选择题答案

### A. 概念辨析

1. A。CVE 给具体漏洞实例编号（像身份证号），CWE 给漏洞类型分类（像病种分类）。B 顺序颠倒；C 二者都不是评分系统（CVSS 才是）；D CWE 不是数据库。

2. B。补丁已公开但大量系统未修补，属于 1day 阶段。A 0day 是漏洞尚未公开；C 已普遍修复则风险降低；D 评分结果是 CVSS 的职责。

3. A。肉鸡通常依赖"软件漏洞 + 木马"形成：漏洞提供入侵方式，木马提供长期控制。B 肉鸡可以是任何联网设备；C 肉鸡不一定要依赖蠕虫；D 肉鸡可以通过清除木马和修复漏洞来处置。

4. A。渗透测试是授权范围内的风险验证，漏洞扫描是自动化检测。B 渗透测试不只是扫描；C 扫描结果不能替代渗透测试报告；D 渗透测试必须有授权。

5. A。黑箱测试从外部视角开始，隐秘测试强调测试存在不被发现。B 黑箱测试不需要源码；C 隐秘测试也关注检测响应能力；D 二者是不同的测试方式。

### B. 堆栈与溢出

6. A。buf 占 12 字节，saved EBP 占 4 字节，返回地址占 4 字节。20 字节输入 = 12(buf) + 4(saved EBP) + 4(返回地址)，刚好覆盖全部。B 错误：返回地址被完整覆盖，不是"低 4 字节"；C 错误：saved EBP 会被覆盖；D 错误：20 > 12，会越界。

7. B。`malloc` 返回的指针指向块身，供程序直接使用。A 块首是堆管理器使用的管理信息；C 堆表入口是索引结构；D 页起始地址是操作系统管理粒度。

8. A。栈溢出常威胁返回地址（控制流），堆溢出常威胁相邻对象或堆管理元数据。B 栈溢出也能覆盖返回地址；C 栈和堆的利用方式不同；D 堆溢出通常比栈溢出更复杂。

9. A。`call` 压入返回地址并跳转，被调用函数建立栈帧，`ret` 从栈中取返回地址并跳回。B `call` 不建立栈帧；C `call` 和 `ret` 都涉及栈操作；D 返回地址由 `call` 自动压入。

10. C。`xor eax, eax` 将 EAX 与自身异或，结果为 0，不依赖其他寄存器。A 依赖 EBX 的值；B 依赖 EBX 的值；D `cmp eax, eax` 只影响标志位，不修改 EAX。

11. D。`[EBX+ESI+1000h]` 是基址 + 变址 + 位移的形式，属于相对基址变址寻址。A 立即数寻址操作数直接在指令里；B 直接寻址是 `[3100h]`；C 基址变址寻址是 `[EBX+ESI]`，没有位移。

### C. PE与调试

12. B。`.idata` 是 import data，存放导入表信息。A `.text` 存放机器代码；C `.rsrc` 存放资源；D `.reloc` 存放重定位信息。

13. A。IAT 在运行时由 Loader 填入导入函数的真实地址。B 编译时函数地址未知；C 返回地址在栈帧中；D IAT 记录外部 DLL 函数地址。

14. B。VA = ImageBase + RVA = 0x00400000 + 0x00011000 = 0x00411000。A 计算错误；C 这是 RVA 本身；D 这是 ImageBase。

15. A。F7 是单步步入（遇到 CALL 进入子函数），F8 是单步步过（遇到 CALL 不进入）。B 顺序颠倒；C F9 是运行；D F9 是运行。

16. B。按 F5 可生成 C/C++ 风格伪代码。A F2 是设置断点；C F7 是单步步入；D F9 是运行。

### D. 漏洞类型与利用

17. B。`%n` 会把已输出字符数写入指定地址，可实现任意地址写入。A `%x` 读取栈上数据；C 整数溢出是计算错误；D 虚函数攻击依赖 vptr/vtable。

18. A。整数溢出本身是计算错误，但可能作为前置条件引发缓冲区溢出。B 整数溢出可发生在任何位宽；C 整数溢出可导致缓冲区溢出；D 整数溢出可影响内存分配。

19. A。攻击者通过篡改 vptr 或 vtable 来劫持虚函数调用。B 虚函数攻击也可通过栈溢出实现；C 虚函数调用依赖 vptr；D vtable 通常在代码段，不在栈上。

20. B。DEP 阻止数据区代码执行，ROP 通过复用已有可执行代码片段绕过。A DEP 会阻止栈上代码执行；C DEP 不能被攻击者关闭；D 整数溢出不能绕过 DEP。

21. A。ASLR 随机化关键地址布局，但可通过信息泄露绕过。B ASLR 不能阻止所有利用；C 程序逻辑不会被随机化；D ASLR 也保护堆和模块地址。

22. A。ROP 通过复用已有代码段中的 gadget 绕过 DEP。B ROP 执行的是已有代码，不是新注入代码；C ROP 也可用于 32 位；D ROP 依赖栈上的返回地址。

23. A。污点源是不可信输入，Sink 是危险操作点。B 污点分析可发现多种漏洞；C 污点分析可以是静态或动态；D Source 和 Sink 的定义颠倒了。

24. A。符号执行适合求解深路径约束，模糊测试适合快速探索大量输入。B 二者是不同技术；C 符号执行更精确；D 符号执行需要约束求解器。

### E. Web安全

25. A。XSS 让脚本在用户浏览器中执行，SQL 注入改变后端数据库查询语义。B XSS 不攻击数据库；C 二者有本质区别；D 存储式和反射式是 XSS 的类型。

26. A。存储式 XSS 风险通常更高，因为恶意脚本保存在服务器端，用户访问正常页面时触发。B 反射式需要用户点击链接；C 存储式风险更高；D 存储式可影响所有访问者。

27. A。参数化查询把 SQL 结构和用户输入分离，黑名单容易被编码、注释等方式绕过。B 速度不是主要优势；C 黑名单能防御部分攻击；D 参数化查询需要数据库支持。

28. A。LFI 包含本地文件，RFI 需要 allow_url_fopen 和 allow_url_include 开启。B 文件包含可包含多种文件类型；C 文件包含也可执行代码；D RFI 需要特定配置。

29. A。用户可控数据直接进入 unserialize()，触发魔术方法执行恶意操作。B PHP 支持序列化；C 反序列化可影响多种操作；D 魔术方法可执行多种操作。

30. A。CSRF 利用用户已登录身份发起请求，XSS 注入脚本在浏览器执行。B CSRF 不攻击数据库；C 二者有区别；D CSRF 也可通过 POST 实现。

## 三、简答题答案

41. 程序崩溃通常只是意味着程序触发了内存违例或异常，但要达到“可利用”的程度，必须满足更苛刻的条件。从**可控性**角度来看，崩溃可能只是由于向非法地址写入了随机数据，而可利用性要求攻击者必须能够精确控制关键寄存器（如 EIP/RIP）的值，或者控制整个执行流的走向。从**覆盖目标**角度来看，普通的崩溃可能只是破坏了无意义的临时变量或触发了未捕获的断言，而可利用的漏洞则要求溢出或篡改的数据能够精准覆盖到关键的控制结构，例如栈上的返回地址、虚函数表指针（vptr）或结构化异常处理（SEH）记录。从**防护机制**角度来看，现代操作系统部署了诸如 ASLR（地址空间布局随机化）等防御手段，即使攻击者成功覆盖了控制目标，如果无法绕过随机化来获取准确的跳转地址，程序也只会因为访问到错误地址而异常崩溃，并不能执行攻击者的恶意代码。因此，崩溃只是发现漏洞的起点，而可利用性则是一场围绕控制流夺取与对抗的深度博弈。
42. 堆溢出和栈溢出由于所处的内存区域和管理机制不同，导致它们的攻击目标存在显著差异。栈溢出发生在连续分配的局部变量区域，其攻击目标通常非常直接和固定，主要瞄准当前函数栈帧中的关键控制数据，例如调用者保存的基址指针（saved EBP/RBP）和最核心的函数返回地址（Return Address），攻击者企图通过直接覆盖返回地址来在函数返回时瞬间劫持控制流。相比之下，堆溢出发生在由程序员动态申请和释放的动态内存区，这里没有固定的返回地址可供直接覆盖，因此堆溢出的攻击目标更加多元且间接。堆溢出的常见目标包括堆块自身管理信息中的前向和后向指针（用以在堆块合并或释放时触发解链漏洞），或者位于堆上的动态对象中的虚函数表指针（vptr）、函数指针成员以及邻近堆块中的业务关键数据。简而言之，栈溢出倾向于“一击必杀”地直接劫持控制流返回，而堆溢出则更倾向于通过破坏内存布局、篡改对象指针或引发复杂的堆管理错误来逐步蚕食系统控制权。
43. 操作系统加载器（Loader）将一个 PE 文件加载进内存并运行需要经过几个核心步骤。首先是**映射文件与创建进程环境**，Loader 会读取 PE 文件的头部信息，验证其合法性（如 MZ 和 PE 签名），并根据文件头中的镜像基址（ImageBase）和映像大小（SizeOfImage）在虚拟内存中为新进程开辟空间，随后将 PE 文件的各个节区（如代码节、数据节）按照内存对齐要求从磁盘映射到对应的虚拟内存地址。接着是**基址重定位处理**，如果程序开启了 ASLR，或者预设的基址已被其他模块占用，Loader 就会根据 PE 文件中的重定位表（Relocation Table），将代码中所有硬编码的绝对地址进行修正。然后是**加载依赖模块与修复输入表**，Loader 会解析 PE 的导入表（Import Table），查找程序运行所需的动态链接库（DLL），将这些 DLL 加载到进程空间中，并通过获取这些 DLL 中导出函数的实际内存地址，逐一填入程序的输入地址表（IAT）中。最后，Loader 会进行**权限设置与跳转执行**，它根据节区属性为不同的内存区域分配恰当的保护属性（如代码段设为只读可执行，数据段设为读写不可执行），初始化主线程的堆栈环境，最终将 CPU 的指令指针（EIP/RIP）跳转到 PE 头中指定的程序入口点（OEP），正式交由用户代码执行。
44. ROP（返回导向编程）之所以能够完美绕过 DEP/NX（数据执行保护）的限制，是因为它彻底改变了恶意代码的执行方式，将传统的“注入新代码并执行”转变为“重用已有合法代码”。在 DEP/NX 的保护下，系统强制规定栈和堆等数据区域只具有读写权限而没有执行权限，这意味着攻击者如果像以前一样把 Shellcode 注入到栈上并跳转过去，CPU 会直接拦截并报错。而 ROP 的核心原理在于，攻击者并不向内存中注入任何新的可执行代码，而是利用程序自身或已加载的依赖 DLL 中原本就存在的高合法性代码片段。这些代码片段通常以 `ret`（返回）指令结尾，被称为 Gadget。攻击者通过精心构造栈上的数据，将一连串精心挑选的 Gadget 地址伪造成函数调用链压入栈中。当程序触发溢出并执行 `ret` 时，CPU 会顺着攻击者布置好的栈数据，一个接一个地跳转到这些原本就拥有“可执行”权限的合法内存地址中执行。通过这些零散 Gadget 的拼接，攻击者可以在完全合法的代码执行流中组合出任意逻辑（如调用 API 动态修改栈内存权限，或直接调用系统命令），从而在不违背 DEP 规则的前提下实现恶意控制。
45. 参数化查询比黑名单过滤在防御 SQL 注入上展现出压倒性的优势，其本质原因在于“彻底隔离了代码与数据”与“被动修补绕过”之间的设计哲学差异。黑名单过滤属于一种典型的“围堵”策略，它试图通过正则表达式或字符串替换，找出输入中包含的单引号、双写注释符、UNION、SELECT 等潜在危险字符并将其拦截或剔除。然而，这种方法面临着无法穷举的致命弱点，黑客经常可以通过编码变形、大小写混淆、特定数据库特性或利用宽字节等意想不到的技巧实现黑名单的完美绕过。相比之下，参数化查询（预编译）采用的是“根本性安全”策略。在参数化查询中，数据库引擎首先对带有占位符的 SQL 语句模板进行编译，在这个阶段，SQL 的语法结构、查询逻辑和执行计划就已经完全确定并固定下来。随后传入的用户输入数据只会被当作纯粹的字面值（Literal Value）填入占位符中，无论攻击者在输入中构造了多么复杂的 SQL 命令、单引号或注释符，数据库引擎都绝对不会把这些数据重新当作 SQL 代码去解析和执行。这种机制从根本上剥夺了输入数据改变 SQL 语句语义的能力，从而实现了对 SQL 注入的绝对免疫。
46. 符号执行在发现数组越界漏洞时，展示了静态深度路径分析的强大能力。其核心方法是抛弃传统测试中使用具体数值作为输入的做法，转而使用抽象的**符号值**（例如变量 $X$）来代表外部输入。当符号执行引擎在虚拟环境中推进程序控制流时，每当遇到分支语句（如 `if` 条件），它不会只走一条路，而是分叉成两条路径，并同时收集该路径必须满足的控制流约束，形成所谓的**路径约束（Path Constraints）**。当分析推进到数组访问语句（假设为 `array[index]`）时，引擎会自动在当前路径上插入一个安全断言，即要求索引值必须在合法范围内，表达式通常为 $0 \le \text{index} < \text{array\_size}$。随后，符号执行引擎会把当前的路径约束与这个断言的取反条件（即约束加上 $\text{index} < 0 \lor \text{index} \ge \text{array\_size}$）打包送入后端强大的 SMT 约束求解器（如 Z3）。如果求解器反馈该条件“可解”（Satisfiable），并给出了一组对应的具体边界数值，就意味着存在一条切实可行的执行路径能够打破安全边界，从而精准、无误报地证实了数组越界漏洞的存在。
47. 存储式 XSS 与反射式 XSS 无论在破坏危害、风险扩散范围还是攻击者的利用手法上，都有着本质的区别。**反射式 XSS** 属于“非持久型”攻击，它的恶意脚本并不保存在服务器上，而是包含在攻击者精心构造的恶意 URL 中。利用方式上，攻击者需要通过社会工程学手段（如钓鱼邮件、聊天群组）诱骗目标用户主动点击这个特制的链接，当用户的浏览器向服务器发送请求后，服务器将 URL 中的恶意脚本直接“反射”回响应页面中触发执行。因此，反射式 XSS 的风险通常是“点对点”的，每次攻击往往只能定向针对点击了链接的特定受害者。而**存储式 XSS** 则是“持久型”攻击，它的恶意脚本会被直接提交并长久保存在服务器的数据库、文件系统或日志中。在利用方式上，攻击者通常将脚本注入到具有交互功能的公共区域（如留言板、评论区或用户个人资料中）。一旦注入成功，任何其他正常用户只要浏览了包含该内容的页面，服务器就会自动将数据库中的恶意脚本加载并渲染给用户，导致脚本在用户的浏览器中被动执行。因此，存储式 XSS 具有极高的传染性和隐蔽性，无需依靠诱骗链接即可实现“大范围撒网”式的无差别攻击，其潜在危害和风险级别远高于反射式 XSS。
48. 在真实的现代网络攻击中，安全团队很少把病毒、蠕虫和木马视为孤立的存在，而是将它们整合进一条完整的攻击链条中，这是因为现代高级可持续威胁（APT）和网络犯罪往往需要同时满足“隐蔽潜伏”、“快速扩散”和“精准控制”等多重战术目标，而没有任何一种单一的恶意软件形式能够独自满足这些复杂的需求。在实际的实战链条中，它们各自扮演着不可或缺的专属角色：木马（Trojan）通常充当攻击的“敲门砖”与“潜伏者”，它伪装成合法软件欺骗用户下载运行，或者作为持久化后门（RAT）静默潜伏在受害者系统中，负责窃取凭证并接受攻击者的远程控制指令；蠕虫（Worm）**则在木马站稳脚跟后被激活，扮演“扩张先锋”，它利用局域网内的漏洞（如永恒之蓝）在无需人工干预的情况下实现全网自动化、指数级的横向移动与快速扩散；而**病毒（Virus）的技术特性则常用于“宿主感染与顽固宿留”，它将自身代码注入到系统核心进程或常用可执行文件中，确保即使某些独立的恶意文件被安全软件清除，恶意代码依然能借由其他受感染的宿主程序不断复活。因此，现代恶意软件早已走向高度的功能融合与模块化组合，将三者串联成链，是攻击者突破复杂纵深防御、最大化攻击效益的必然演进结果。

## 四、综合题答案及解析

**49. 基础内存破坏与漏洞利用：栈溢出与防护绕过**
* **（1）内存布局分析**：
  * **排列顺序**：从低地址到高地址依次为：`buf`（局部变量） -> `saved EBP`（保存的旧栈底指针） -> `Return Address`（返回地址）。
* **（2）溢出点计算**：
  * **计算方法**：在 32 位程序中，`buf` 占 32 字节，`saved EBP` 占 4 字节。因此，攻击者至少需要输入 `32 + 4 = 36` 个字节的填充数据，才能将这两个区域完全填满。第 37 个字节开始就会刚好覆盖栈上的返回地址。
* **（3）控制流劫持**：
  * **参数布置顺序**：在 `__cdecl` 调用约定下，通过覆盖返回地址跳转到 `system()` 后，栈上的布置顺序从低到高必须为：`system() 函数地址` -> `伪造的返回地址`（如 `exit()` 地址或任意 4 字节） -> `参数指针`（如指向 `"cmd.exe"` 字符串的地址）。
* **（4）防御机制对抗**：
  * **对抗思路**：攻击者必须提前利用其他漏洞（如格式化字符串信息泄露），从内存中窃取到当前函数栈帧中的 **Canary 的真实魔术值**。在构造溢出数据时，将真实的 Canary 值原样填回原处，从而绕过校验机制。

**50. 恶意代码植入与逆向工程：PE 注入与调试分析**
* **（1）PE 结构解析**：
  * **RVA 计算**：`RVA = VA - ImageBase`，即 `0x00405000 - 0x00400000 = 0x00005000`。
  * **入口点修改**：PE 头中的 `AddressOfEntryPoint` 填写的是 RVA，因此需要修改为 **`0x00005000`**。
* **（2）底层调用规约**：
  * **压栈顺序**：`__stdcall` 调用约定的参数压栈顺序是**从右到左**。
* **（3）隐蔽性控制**：
  * **操作机制**：注入的恶意代码执行完毕后，必须使用 `JMP` 或 `CALL` 等跳转指令，将指令指针强制跳转回程序的**原始入口点（OEP，即 `0x00403000`）**。这样原程序才能正常初始化并运行，使受害者难以察觉程序已被劫持。
* **（4）动态调试验证**：
  * **验证思路**：可以在系统库（如 `user32.dll`）导出的 `MessageBoxA` 函数的入口处下达 **API 断点（如通过 INT 3 的软件断点或硬件断点）**。如果程序运行后命中了该断点，并且调用栈（Call Stack）显示是从注入代码所在的地址区域调用的，即可证明注入代码被成功执行。

**51. 经典 Web 漏洞串联：从注入到任意代码执行**
* **（1）身份劫持**：
  * **绕过手法**：在 `user` 字段输入类似 `admin' #` 或者 `admin' OR '1'='1` 的万能密码 Payload。前者利用注释符 `#` 或 `-- ` 屏蔽了后续的密码校验逻辑，后者则通过构造逻辑恒真式使整条 SQL 语句判断为真，从而绕过验证。
* **（2）恶意载荷投递**：
  * **利用原理**：这种手段主要利用了 HTTP 协议头部中的 **`Content-Type`** 字段。由于后端只校验该字段的值，攻击者使用抓包工具将其修改为 `image/jpeg` 即可欺骗服务器，使其认为上传的是合法图片文件。
* **（3）攻击链闭环**：
  * **执行原理**：PHP 的 `include` 函数有一个核心特性，即无论被包含文件的后缀名是什么（哪怕是 `.jpg`），只要文件内容中包含合法的 `<?php ... ?>` 标签，PHP 解释器都会将其中的代码作为脚本进行解析和执行。因此，利用 LFI 漏洞包含该伪装图片即可成功激活木马，实现 RCE。
* **（4）安全防御建设**：
  * **防御 SQL 注入**：放弃简单的字符串拼接，全面改用 **预编译语句（参数化查询，Parameterized Queries）**。
  * **防御文件上传**：后端应强制重命名上传文件（禁止保留原文件名），并且不应该将上传目录置于可执行的 Web 目录中，或者在 Web 服务器（如 Nginx/Apache）层面配置该目录禁止解析 PHP 脚本。
  * **防御文件包含**：采用**白名单校验机制**，只允许包含预先定义好的几个安全文件路径，拒绝包含任何由用户不可信参数控制的未知路径。

**52. 漏洞挖掘与自动化分析：高级测试技术论证**
* **（1）快速发现崩溃**：**模糊测试（Fuzzing）**。适用于快速探索和大批量喂入变异输入，寻找触发崩溃的浅层漏洞。
* **（2）数据流向追踪**：**污点分析（Taint Analysis）**。专门用于追踪标记的不可信数据（污点源 Source）是否能够到达敏感的操作节点（Sink）。
* **（3）深层约束求解**：**符号执行（Symbolic Execution）**。将特定的路径条件转化为数学方程交由求解器求解，能够精准计算出突破复杂分支或校验墙的具体输入。
* **（4）剥离冗余代码**：**程序切片（Program Slicing）**。通过分析程序的控制流和数据流，将与特定感兴趣变量无关的代码剔除，极大地缩减逆向和安全审计的范围。
