# 软件安全进阶综合实战练习题

> 本手册为《软件安全》课程的进阶训练题库，侧重于深度概念辨析、多重机制对抗以及真实场景的综合利用。

---

## 一、 单项选择题（20 题）

**1. 在 32 位 x86 架构下进行软件调试时，关于函数调用的栈帧建立与销毁过程，以下描述更为准确的是：**
A. caller 负责将局部变量压入栈中，随后使用 call 指令跳转至 callee 的入口地址。
B. callee 执行 ret 指令时，处理器会首先恢复 EBP 寄存器，然后再弹出返回地址。
C. call 指令不仅修改 EIP 指向目标函数，同时也会自动将下一条指令的地址压栈。
D. leave 指令的作用等价于 pop ebp，它用于在函数即将结束时直接恢复栈基址。

**2. 针对 Windows 平台下 PE 文件格式的解析与执行机制，以下描述更为准确的是：**
A. ImageBase 指定了 PE 文件在内存中的绝对基址，程序每次运行必然加载于此。
B. AddressOfEntryPoint 存储的是相对虚拟地址（RVA），指示程序执行的第一条指令。
C. `.reloc` 节区包含重定位信息，当程序未开启 ASLR 时，加载器仍然会强制进行重定位。
D. `.idata` 节区中记录了导入函数信息，其在编译阶段就已经被静态填入了真实物理地址。

**3. 在经典的栈溢出漏洞中，攻击者企图劫持程序控制流。关于覆盖目标与利用条件的描述，更为准确的是：**
A. 将局部变量缓冲区填满后，溢出的数据通常会直接触发内存违例导致程序在当前上下文中崩溃。
B. strcpy 函数在遇到换行符（\x0a）时会停止复制，因此 Payload 中无法包含换行符。
C. 成功覆盖栈上的 saved EBP 后，程序会在当前函数立即跳转到攻击者指定的地址。
D. 覆盖返回地址的目的是控制 EIP 的指向，但这通常需要在当前函数执行到 ret 时才生效。

**4. 在现代操作系统中，关于各类漏洞利用与缓解机制的博弈，以下描述更为准确的是：**
A. 开启 DEP（数据执行保护）后，攻击者通常会转向利用 ret2libc 技术，通过伪造栈帧调用系统库函数。
B. ASLR（地址随机化）开启后，程序每次加载时各模块内的函数相对偏移地址也会发生动态改变。
C. Stack Canary 主要通过校验栈底附近魔术值的完整性来防御攻击，这在机制上天然隔离了格式化字符串漏洞的威胁。
D. ROP（返回导向编程）的核心思想是通过将恶意的 Shellcode 分解成短小的片段，注入到数据区执行。

**5. 关于 C/C++ 程序中的整数溢出（Integer Overflow）漏洞，以下描述更为准确的是：**
A. 整数溢出本身往往不会直接导致控制流被劫持，而是经常作为前置条件引发堆栈缓冲区溢出。
B. 无符号整数在发生向下溢出（Underflow）时，其值会被截断为零，从而导致程序逻辑进入异常分支。
C. 在 64 位系统中，由于寄存器宽度的增加，使用 32 位整型变量进行内存分配计算不再存在溢出风险。
D. 整数溢出主要影响涉及数学运算的业务逻辑结果，对关键数据进行高强度的加密处理是防御此类漏洞的核心手段。

**6. 在 32 位程序的格式化字符串漏洞利用中，关于读写内存的操作机制，以下描述更为准确的是：**
A. 使用 `%s` 格式控制符时，printf 会直接读取并打印参数列表所在栈地址上的具体数值内容。
B. 格式化字符串漏洞主要局限于读取调用栈上的数据内容，通常很难跨越内存段去读取堆区或全局数据段的信息。
C. 借助 `%n` 格式控制符，攻击者可以将已输出的字符总数写入由参数指针所指向的特定内存地址中。
D. 构造特定数量的 `%x` 可以精准修改函数的返回地址，从而在不引发崩溃的前提下实现代码执行。

**7. 关于堆内存破坏中的 Use-After-Free (UAF) 漏洞，以下描述更为准确的是：**
A. UAF 漏洞的触发前提是堆管理器在释放内存后，未将其标记为空闲状态，导致其他申请无法使用。
B. 攻击者必须通过连续的 malloc 操作耗尽可用堆空间，才能迫使程序重新分配已被释放的关键对象。
C. 程序中如果存在对已释放内存的读操作，通常会导致程序的执行流被直接劫持，进而获取系统的控制权。
D. 配合堆风水（Heap Fengshui）技术，攻击者可以利用特定大小的新对象去精确占位已被释放的内存块。

**8. 在编写用于漏洞利用的 Shellcode 时，攻击者需要考虑多种执行环境的限制，以下描述更为准确的是：**
A. Shellcode 在执行系统调用时，往往依赖于预先确定的绝对地址，这使其在应对 ASLR 机制时具有显著的稳定性优势。
B. 为了避免被 strcpy 等字符串处理函数提前截断，Shellcode 的字节序列中通常需要规避空字符（\x00）。
C. 注入到堆区的 Shellcode 相比于注入到栈区的 Shellcode，在执行效率上会有显著的提升与优化。
D. 目标程序在赋予缓冲区可读写权限后，注入其中的 Shellcode 序列即可顺利地被 CPU 解析并执行。

**9. 关于自动化漏洞挖掘技术中的模糊测试（Fuzzing），以下描述更为准确的是：**
A. 基于覆盖率引导的灰盒 Fuzzing 能够利用程序执行时的路径反馈，动态调整变异策略以触及更深层的代码。
B. Fuzzing 的核心优势在于能够利用形式化方法的推导过程，从理论层面上证明目标程序不存在越界访问的隐患。
C. 黑盒 Fuzzing 通常依赖于目标程序的源代码，以便在编译阶段插入专用的探针与监测代码来捕获深层异常。
D. 在处理极其复杂的加密算法或哈希校验分支时，Fuzzing 技术往往比符号执行能够更高效地求解出合法输入。

**10. 在进行程序安全分析时，关于污点分析（Taint Analysis）与符号执行（Symbolic Execution），描述更为准确的是：**
A. 污点分析主要用于求解复杂的数学约束条件，从而生成能够触发程序崩溃的精确测试用例。
B. 动态污点分析通过跟踪不可信输入在内存和寄存器中的传播路径，检测其是否到达危险的敏感操作点。
C. 符号执行在分析循环结构和复杂分支时，能够通过路径剪枝技术有效控制状态空间爆炸，广泛适用于各类大型软件。
D. 静态污点分析通过在目标程序实际运行的过程中，动态拦截并追踪内存与寄存器的状态转移来发现危险操作。

**11. 关于 Web 安全中的 SQL 注入防御，以下描述更为准确的是：**
A. 对用户输入中的单引号、双引号及反斜杠进行全面的转义处理，是目前阻断各类 SQL 注入攻击的最通用方案。
B. 采用基于黑名单的 Web 应用防火墙（WAF）拦截常见的 SQL 关键字，是修复应用程序底层逻辑漏洞的首选实践。
C. 参数化查询（预编译）通过将 SQL 语句结构与传入数据严格分离，从解析机制上剥夺了输入数据改变查询语义的能力。
D. 报错注入主要依赖于数据库返回的错误信息，关闭前端页面的错误回显可以直接切断 SQL 注入的攻击路径。

**12. 在跨站脚本攻击（XSS）的分类与利用场景中，以下描述更为准确的是：**
A. 反射式 XSS 的恶意代码直接被持久化存储在服务器的数据库中，每次用户访问相关页面时均会自动触发。
B. DOM 型 XSS 的执行过程严格依赖于服务器端对恶意请求的解析和响应，其攻击载荷往往出现在后端日志中。
C. 在设置 Cookie 时开启 HttpOnly 属性，能够有效地从浏览器层面阻止 XSS 脚本对当前用户登录态的劫持与利用。
D. 存储式 XSS 具有更广的波及范围和更高的隐蔽性，因为正常用户在浏览包含恶意负载的公共页面时即可能中招。

**13. 关于文件上传漏洞及其防护策略，结合服务器解析机制，以下描述更为准确的是：**
A. 前端 JavaScript 对文件后缀的校验是防止任意文件上传最核心的防线，因为它可以显著减轻服务器的计算压力。
B. 即使严格校验了文件的 MIME 类型和 Magic Byte，攻击者仍可将代码隐藏在正常的图片数据区中构造图片马。
C. 强制将用户上传的文件重命名并抹除其原始扩展名，是防御 Web 服务器恶意脚本解析漏洞的标准规范。
D. Apache 容器的解析漏洞主要由于其总是从左向右严格匹配单个扩展名，导致未知后缀名的文件被拒绝解析。

**14. 在 PHP 文件包含漏洞（File Inclusion）的利用中，关于本地（LFI）与远程（RFI）包含的特性，描述更为准确的是：**
A. LFI 主要用于读取服务器上的静态配置文件，若包含的文件中存在 PHP 代码，解析器通常会将其当作纯文本原样输出。
B. RFI 的触发具有极高的独立性，程序中若存在用户可控的 include 语句，攻击者即可直接引入外部的恶意脚本负载。
C. 攻击者常利用 PHP 的伪协议（如 php://filter）结合 LFI 读取敏感源码，或利用 php://input 提交恶意流数据实现代码执行。
D. 在 LFI 和 RFI 的利用场景中，若被包含的目标文件未获得操作系统的可执行权限，则无法触发其内部代码的执行。

**15. 关于 PHP 反序列化漏洞的产生机制与防范，以下描述更为准确的是：**
A. 反序列化漏洞的根源在于 serialize() 函数在处理复杂对象时存在解析缺陷，导致生成的字符串携带恶意指令。
B. 对传输的序列化字符串进行 Base64 编码，能够有效规避不可信数据在反序列化过程中对危险魔术方法的触发调用。
C. 魔术方法（如 __destruct, __wakeup）通常需要在程序显式地通过对象实例去调用它们时，才会真正进入执行流程。
D. 当不可信数据直接传递给 unserialize() 时，攻击者可控制对象的内部属性，进而利用程序中已有的魔术方法链（POP 链）执行操作。

**16. 在 CSRF（跨站请求伪造）与 SSRF（服务器端请求伪造）的对比中，以下描述更为准确的是：**
A. CSRF 主要利用受害者浏览器中已有的身份凭证，诱导其向存在漏洞的站点发起状态改变请求；而 SSRF 则是利用服务器自身的权限向内网发起请求。
B. CSRF 的攻击目标通常是后端服务器的内部服务接口，而 SSRF 的攻击目标则是其他普通用户的个人敏感数据。
C. 在 HTTP 响应头中配置严格的 CORS（跨域资源共享）策略，是同时防御 CSRF 和 SSRF 两种请求伪造攻击的有效机制。
D. SSRF 漏洞的形成源于目标服务器具备对外部 URL 的拉取功能，这导致其难以越权访问与自身处于同一局域网内的内部资产。

**17. 关于逆向工程中常用的动静态分析工具与技术，以下描述更为准确的是：**
A. IDA Pro 作为强大的静态反汇编工具，能够在不加载任何插件的情况下，直接实时调试并修改内核级驱动程序的运行逻辑。
B. 在 OllyDbg 等用户态调试器中，软件断点（INT 3）的本质是将目标指令的第一个字节临时替换为特定的机器码（0xCC）。
C. 动态分析相比于静态分析，最大的优势在于其能够瞬间覆盖程序的所有可能执行路径，而不受特定输入数据的限制。
D. 反编译（Decompilation）过程能够完美地还原出程序在编译前包含的所有局部变量名称和详细的开发者注释信息。

**18. 在深入理解栈的结构和函数返回地址劫持时，关于寄存器与内存的关系，描述更为准确的是：**
A. EBP 寄存器主要用于动态跟踪栈顶的位置，随着栈内数据的压入和弹出，它的值会频繁发生改变。
B. ESP 寄存器指向当前函数的栈底基址，在整个函数的生命周期内，它的值通常保持相对稳定不变。
C. 当被调用函数执行 leave 指令时，处理器会将 EBP 的值赋给 ESP，随后将栈中保存的旧 EBP 弹出并恢复。
D. 攻击者在成功覆盖了 ESP 寄存器中的数据后，程序随后执行的下一条指令通常会跳转至攻击者所预设的恶意地址。

**19. 在复杂的 Web 业务逻辑中，经常会出现多漏洞组合的场景。关于这些漏洞的关联与转化，描述更为准确的是：**
A. XSS 漏洞的作用域主要停留在窃取用户 Cookie 层面，通常难以借此在后台进一步触发如文件上传等深层次漏洞。
B. SQL 注入的威胁主体集中在读取数据库中的敏感数据，即使拥有 DBA 权限，也难以利用数据库引擎向服务器物理写入恶意文件。
C. 文件包含漏洞（LFI）在面临无法直接上传脚本文件的限制时，攻击者通常难以寻找到其他途径将其转化为远程代码执行（RCE）。
D. 通过存储式 XSS 窃取到高权限管理员的 Session 后，攻击者可以利用该登录态进入后台，从而接触并利用更深层次的系统漏洞。

**20. 结合现代软件安全体系的整体视角，从漏洞产生、挖掘到防护的整个生命周期中，描述更为准确的是：**
A. 许多 Web 层面的业务逻辑缺陷最终可能导致服务器端的原生命令被非法调用，这种跨层的漏洞组合往往具有极大的破坏性。
B. 在开发生命周期的早期引入静态代码扫描（SAST）工具进行全面排查，是消除软件上线后深层次内存破坏隐患的核心手段。
C. 渗透测试的主要目标是验证系统的网络带宽承受能力和抗 DDoS 攻击能力，以确保业务在极端流量下不中断。
D. CWE（常见缺陷枚举）主要用于对已发现的具体漏洞实例进行统一编号与评级，而 CVE 则侧重于总结漏洞的成因模式。

## 二、 判断题（20 题）

21. 即使一个软件 Bug 导致了程序在运行时发生崩溃，若该 Bug 缺乏被外部不可信输入稳定触发并打破安全边界的条件，则不应被归类为安全漏洞（Vulnerability）。

22. 在 x86 汇编中，`call` 指令的执行实际上是由 `push eip` 和 `jmp` 两个微操作复合而成的，因此它不可避免地会修改栈顶指针 ESP 的值。

23. 如果在调试器中单步步入（Step Into）一个调用操作系统的底层 API 函数，即使本地没有该系统库的源码或符号表，EIP 依然会跳转到系统模块（如 ntdll.dll）的虚拟地址空间中去执行其内部的汇编指令。

24. 当目标程序未开启如 Canary、DEP、ASLR 等现代防护机制时，被成功触发的栈溢出漏洞总是能够导致程序的执行流被劫持并执行相应的 Shellcode。

25. 在利用格式化字符串漏洞时，攻击者可以通过精心构造参数的对齐长度，使得 `%n` 格式符向内存写入极其精确的单字节或双字节十六进制数值，从而实现对目标地址的精准篡改。

26. 在进行加法或乘法等敏感运算之前，将变量强制类型转换为位宽更长的类型（如将 32 位 int 转为 64 位 long long），是跨平台编译下消除整数溢出风险的标准规范。

27. 为了保证 Shellcode 在不同版本或不同语言的同一操作系统上都能稳定执行，攻击者往往需要在 Shellcode 开头动态解析 PEB（进程环境块）来获取系统 API 的真实内存基址，而不是依赖硬编码地址。

28. 由于 DEP（数据执行保护）主要作用于限制栈区的执行权限，攻击者通过将恶意代码注入到具有可读写属性的堆区中，即可达成绕过该防御机制的目的。

29. 在 64 位 Linux 环境下构造 ROP 链时，由于函数调用的前几个参数是通过寄存器（如 rdi, rsi）而非栈进行传递的，因此攻击者必须在代码段中寻找到能对这些特定寄存器进行赋值的 Gadget 才能完成函数调用。

30. ASLR（地址空间布局随机化）的核心安全价值在于它能够在程序每次运行时，动态改变函数内部控制流图（CFG）的相对拓扑结构，从而使得基于特定执行路径的攻击载荷彻底失效。

31. 灰盒模糊测试（Grey-box Fuzzing）为了实现高吞吐量的变异生成，完全摒弃了对目标程序内部状态的关注，因此它在求解极其复杂的魔术字节（Magic Byte）校验时，效率通常远高于符号执行。

32. 在动态污点分析中，如果被标记为污染（Tainted）的外部输入数据，在传播过程中经过了系统内置的安全过滤函数或严格的边界截断处理，分析引擎通常会将该数据的污染标记清除（Sanitization）。

33. 为了在架构层面构建针对 XSS 攻击的系统性防御，后端开发者通常需要在将用户输入存入数据库之前，对原始数据执行严格的 URL 编码处理。

34. 即使应用程序使用了预编译语句（参数化查询），如果在编写 SQL 语句时，将用户可控的输入直接拼接为了表名（Table Name）或列名（Column Name），依然会存在 SQL 注入的严重风险。

35. 在处理用户头像上传时，利用后端的图像处理库（如 GD 库）在写入磁盘前对文件进行二次重绘与压缩，是彻底阻断恶意后门文件植入的有效隔离手段。

36. 当 Web 应用程序强制要求所有涉及状态变更的敏感操作（如转账、修改密码）统一采用 POST 请求方式进行提交时，该应用便具备了防御跨站请求伪造（CSRF）的基础屏障。
