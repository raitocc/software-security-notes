## 总体学习路线

这门课可以按一条主线理解：

1. 先理解恶意代码、Bug、漏洞、漏洞库和渗透测试。
2. 再补底层基础：内存布局、栈帧、寄存器、汇编、PE 文件、调试。
3. 然后进入典型漏洞：缓冲区溢出、格式化字符串、整数溢出、C++ 虚函数、注入、权限漏洞等。
4. 接着学习漏洞利用：exploit、payload、shellcode、控制流劫持。
5. 再看防护与绕过：ASLR、GS、DEP/NX、SafeSEH、SEHOP、ROP。
6. 最后学习漏洞挖掘、Web 安全和渗透测试流程。

一句话版：软件安全就是理解“输入如何变成错误状态，错误状态如何变成控制权，控制权如何被防御或检测”。

# Lecture01 课程概要和基本概念 - 复习手册

> 本文件基于课程 PDF 整理，用于考前复习和人工复核。  
> 本版目标不是压缩成答案卡，而是把概念讲清楚：先讲是什么，再讲怎么发生、表现是什么、和相邻概念怎么区分。

### 本讲主线

Lecture01 是软件安全课程的入口。它不急着讲某一种攻击细节，而是先建立整门课反复使用的基本语言：

- 恶意代码：病毒、蠕虫、木马。
- 软件问题：Bug、漏洞、内存安全问题。
- 攻击对象：肉鸡、服务器、客户端、Web 应用。
- 漏洞时间线：0day、1day、已公开漏洞。
- 漏洞知识组织：CVE、CWE、NVD、CNNVD、CNVD。
- 安全评估方式：渗透测试、黑箱、白箱、隐秘测试。

可以用一句话串起来：软件由人编写，程序中会有 Bug；其中一部分 Bug 能被攻击者利用，就变成漏洞；漏洞可能被做成 exploit，用来植入木马、形成肉鸡、传播蠕虫；防御者通过漏洞库、补丁管理和渗透测试来理解、修复和验证这些风险。

### 病毒、蠕虫、木马

病毒、蠕虫、木马都属于恶意程序或恶意代码，但它们强调的能力不同。考试常考它们的区别，理解时不要只背定义，要抓住传播方式和攻击目标。

| 类型 | 核心特征 | 是否自我复制 | 典型传播方式 | 主要目的 |
|---|---|---|---|---|
| 病毒 Virus | 感染宿主程序或文件，把自身代码嵌入别人的执行流程 | 是 | 依附可执行文件、文档宏、脚本等传播 | 传播、破坏、隐藏、执行恶意逻辑 |
| 蠕虫 Worm | 自包含程序，利用网络独立扩散 | 是 | 扫描网络、利用远程漏洞、自动复制到其他主机 | 大规模扩散、攻击、投放载荷 |
| 木马 Trojan Horse | 伪装成正常程序，诱导用户运行，建立未授权控制 | 通常不强调自我复制 | 软件捆绑、钓鱼附件、伪装工具、破解软件 | 隐蔽控制、窃密、远程操作、持久化 |

#### 计算机病毒怎么实现

病毒的关键不是“它很坏”，而是它能把自己放进其他程序或文件的执行路径里。常见思路包括：

1. 找到宿主文件，例如可执行程序、脚本文件或带宏的文档。
2. 修改宿主文件，把病毒代码插入进去，或修改入口点让程序先执行病毒代码。
3. 病毒代码执行后，再把控制权交还给原程序，让用户不容易察觉。
4. 继续寻找新的宿主文件并感染。

病毒通常具有感染性、潜伏性、触发性和破坏性。感染性指能复制或嵌入其他文件；潜伏性指不一定立刻发作；触发性指可能在特定日期、次数、文件名、用户行为下执行破坏逻辑；破坏性指删除文件、篡改数据、降低系统稳定性或加载其他恶意模块。

表现上，病毒可能导致文件体积异常变化、程序入口被修改、系统变慢、文件损坏、杀毒软件报警、异常进程或启动项出现。但现代恶意代码越来越强调隐藏，不能只靠肉眼观察判断。

#### 蠕虫怎么实现

蠕虫与病毒最大的区别是：蠕虫不一定依附宿主文件，它更像一个能自己到处跑的网络程序。它通常会：

1. 扫描网络，寻找开放端口、易受攻击服务或弱口令主机。
2. 利用远程漏洞或配置错误进入目标。
3. 把自身复制到目标主机并运行。
4. 在新主机上继续扫描和传播。

蠕虫的危险在于自动化和规模。一个漏洞如果存在于大量联网系统中，蠕虫可以在短时间内指数级扩散。典型案例包括 Stuxnet 和 WannaCry。WannaCry 的传播与 SMBv1 相关漏洞有关，它在进入系统后会加密文件并勒索；EternalBlue 相关利用链说明，蠕虫常常把“远程漏洞利用”和“恶意载荷执行”组合起来。

蠕虫与病毒都可以自我复制，但病毒强调感染宿主，蠕虫强调网络独立传播。考试题里如果看到“通过网络自传播、自包含程序、不依赖感染文件”，优先想到蠕虫。

#### 木马怎么实现

木马的名字来自“特洛伊木马”：外表看起来是正常、有用或无害的东西，实际隐藏恶意目的。木马通常不靠自己大规模复制，而靠欺骗用户、捆绑软件或漏洞投放进入系统。

木马进入系统后常做几类事：

- 建立远程控制通道，让攻击者能执行命令或操作文件。
- 窃取账号、密码、Cookie、聊天记录、浏览器数据等。
- 记录键盘输入或截屏。
- 修改启动项、服务、计划任务，实现开机自启动。
- 下载更多恶意模块，例如挖矿程序、勒索软件、代理模块。
- 隐藏自身，例如伪装进程名、修改注册表、绕过安全软件。

木马的关键词是伪装、诱导、隐蔽控制。它不强调“感染其他文件”，而强调“让用户或系统运行它之后，攻击者获得非授权能力”。

#### 三者之间的关系

病毒、蠕虫、木马不是互斥标签。真实攻击中经常混合：

- 蠕虫利用漏洞进入系统后，可能投放木马用于长期控制。
- 木马也可能带有自传播模块，表现出蠕虫特征。
- 病毒感染宿主程序后，也可能下载木马组件。

所以分类的目的不是给每个恶意程序贴唯一名字，而是理解它主要靠什么传播、靠什么控制、造成什么后果。

### Bug 与漏洞

Bug 是软件缺陷，漏洞 Vulnerability 是可被攻击者利用并造成安全影响的软件缺陷。可以粗略理解为：

```text
软件错误/缺陷 >= Bug
Bug 中可被攻击者利用并造成安全影响的部分 = 漏洞
```

不是所有 Bug 都是漏洞。比如按钮颜色显示错误、普通页面排版错乱、某个本地工具偶尔崩溃，通常只是功能或稳定性问题。只有当缺陷能被外部输入触发，并导致越权、信息泄露、代码执行、拒绝服务、权限提升等安全后果时，才更接近漏洞。

#### Bug 怎么来的

Bug 可能来自很多环节：

- 需求理解错误：程序实现了错误的业务规则。
- 设计缺陷：认证、授权、权限边界设计不清。
- 编码错误：数组越界、空指针、格式化字符串、整数溢出。
- 输入处理不当：没有校验长度、类型、格式、范围。
- 依赖问题：使用有漏洞的第三方库或旧组件。
- 配置错误：调试接口暴露、默认密码、权限过大。
- 并发和状态错误：竞态条件、状态机绕过。

安全课程更关心那些能被攻击者稳定触发和利用的 Bug。一个 Bug 如果只在非常偶然的本地环境中出现，攻击价值可能不高；如果攻击者能通过网络包、文件、URL、表单参数、文档内容精确触发，它就更危险。

#### Bug 怎么升级成漏洞

一个普通 Bug 升级成漏洞，通常需要满足几个条件：

1. 攻击者能控制输入或环境，例如请求参数、文件内容、网络数据、用户名、Cookie。
2. 程序对这些输入处理错误，例如越界写、拼接 SQL、错误鉴权。
3. 错误结果影响安全边界，例如读取了不该读的数据，执行了不该执行的代码。
4. 攻击者能重复触发或构造利用链，而不是只让程序随机崩溃。

例如，程序崩溃不一定是漏洞。如果崩溃只能影响用户自己的本地程序，可能只是稳定性问题；如果远程攻击者发送一个包就能让服务器崩溃，就可能是拒绝服务漏洞；如果崩溃位置可控，能覆盖返回地址或函数指针，就可能进一步发展成远程代码执行漏洞。

#### 为什么点 URL 或打开文档也会中木马

“用户只是点击 URL，怎么会执行代码？”关键在于浏览器、插件、文档解析器本身也是程序。它们要解析 HTML、JavaScript、图片、字体、PDF、Office 文档等复杂格式。只要解析器存在漏洞，攻击者就可以构造特殊数据，让解析过程出现越界、类型混淆或脚本执行。

Office 文档也类似。文档本身不是传统 exe，但打开文档的软件可能支持宏、脚本、嵌入对象，也可能在解析格式时有内存漏洞。攻击者不一定让用户主动运行木马 exe，而是让正常软件在解析恶意内容时被劫持，再下载或执行木马。

### 肉鸡

肉鸡是被攻击者控制的电脑、服务器或其他联网设备。它的重点不是设备类型，而是“控制权不在真正所有者手里”。

一台机器变成肉鸡，通常经历两步：

1. 入侵：利用漏洞、弱口令、钓鱼、恶意附件、供应链问题等进入系统。
2. 控制：植入木马、后门、远控程序或持久化脚本，让攻击者能长期操作。

肉鸡可以被用来：

- 发送垃圾邮件、钓鱼邮件。
- 发起 DDoS 攻击。
- 作为跳板隐藏攻击者真实位置。
- 挖矿或消耗资源。
- 扫描更多目标，继续扩散。
- 窃取本机数据或内网数据。

大量服务器沦为肉鸡，通常依赖“软件漏洞 + 木马”这两个要素：漏洞提供进入方式，木马提供长期控制。考试里如果问“肉鸡形成依赖什么”，这个组合很常见。

### 漏洞生命周期：0day、1day、已公开漏洞

漏洞不是一出现就被所有人知道。它有一个从未知到公开、从无补丁到有补丁、从高风险到逐渐被修复的生命周期。

| 类型 | 含义 | 攻防状态 | 风险 |
|---|---|---|---|
| 0day / Zero-day | 漏洞尚未公开，厂商和用户通常不知道或没有补丁 | 攻击者如果掌握，防御者很被动 | 最高 |
| 1day / One-day | 漏洞已公开或补丁刚发布，但大量系统尚未修复 | 攻击者可逆向补丁、快速开发利用 | 仍然很高 |
| 已公开漏洞 | 已有编号、描述、补丁或缓解方案 | 防御重点是资产排查和补丁管理 | 取决于修复情况 |

0day 的危险在于信息不对称。防御者不知道漏洞存在，没有补丁、没有检测规则、也没有经验；攻击者却可以利用它绕过常规防线。

1day 不一定严格指“过了一天”。课程语境中更接近“补丁刚出来不久、尚未被普遍修复的漏洞”。补丁发布后，攻击者可以通过比较补丁前后代码差异，推测漏洞位置，写出 exploit。很多真实攻击不是靠神秘 0day，而是靠大家没及时修补的 1day 或老漏洞。

已公开漏洞也不等于没风险。只要系统没有打补丁、资产暴露在公网、配置不当，几年以前的漏洞仍然可能造成入侵。

### 内存安全

内存安全是后续课程的主线之一。简单说，内存安全就是程序访问内存时必须访问“该访问的地方”、在“该访问的时间”、用“正确的类型和长度”。一旦程序读写了不属于自己的内存，或使用已经释放的对象，就可能出现安全问题。

常见内存安全问题包括：

- 栈溢出：写入局部缓冲区的数据超过边界，覆盖栈帧中的相邻数据、旧 EBP、返回地址等。
- 堆溢出：写入堆缓冲区的数据超过边界，覆盖相邻堆块、对象字段或堆管理元数据。
- 越界读：读取数组或缓冲区外的数据，可能泄露敏感信息、地址、canary。
- 越界写：写到数组或缓冲区外，可能破坏对象、函数指针、返回地址。
- Use After Free：释放后继续使用对象，攻击者可能重新占用该内存并控制对象内容。
- Double Free：同一块内存释放两次，可能破坏分配器状态。
- 悬空指针：指针指向的对象生命周期已结束，但指针仍被使用。
- 类型混淆：把对象当成错误类型解释，导致错误字段访问或错误函数调用。
- 整数溢出：长度计算回绕，导致分配空间过小、检查绕过，再引发内存破坏。
- 未初始化内存使用：使用未设置的内存内容，可能泄露数据或导致不可预测行为。

内存安全问题在 C/C++ 程序中尤其常见，因为程序员需要自己管理内存边界和对象生命周期。Java、Python、Go、Rust 等语言在不同程度上通过运行时检查、垃圾回收、所有权模型等方式减少这类问题，但它们仍可能有逻辑漏洞、依赖漏洞、反序列化漏洞、注入漏洞等。

PPT 中的统计图强调：在很多大型软件项目中，内存安全长期占据漏洞的重要比例。后续 Lecture02 到 Lecture07 会把它展开成栈帧、缓冲区溢出、shellcode、ASLR、DEP、ROP 等内容。

### 漏洞库

漏洞库的作用是把分散的漏洞信息组织起来，让厂商、安全人员、扫描器、报告和补丁管理系统能说同一种语言。只背缩写很难记，必须知道全称和分工。

| 缩写 | 全称 | 中文理解 | 重点 |
|---|---|---|---|
| CVE | Common Vulnerabilities and Exposures | 通用漏洞与暴露 | 给“具体漏洞实例”统一编号 |
| CWE | Common Weakness Enumeration | 通用弱点枚举 | 给“漏洞类型/弱点模式”分类 |
| NVD | National Vulnerability Database | 美国国家漏洞数据库 | 基于 CVE 补充 CVSS、影响版本、参考链接等 |
| CVSS | Common Vulnerability Scoring System | 通用漏洞评分系统 | 用分值描述漏洞严重程度 |
| CNNVD | China National Vulnerability Database of Information Security | 中国国家信息安全漏洞库 | 中国的国家级漏洞信息库 |
| CNVD | China National Vulnerability Database / 国家信息安全漏洞共享平台 | 国家信息安全漏洞共享平台 | 由 CNCERT 联合多方建设的漏洞共享平台 |
| CNCERT | National Computer Network Emergency Response Technical Team/Coordination Center of China | 国家计算机网络应急技术处理协调中心 | 网络安全应急协调机构 |

CVE 和 CWE 最容易混：

- CVE 讲“哪一个具体漏洞”。例如某产品某版本的某个远程代码执行漏洞，有一个 CVE 编号。
- CWE 讲“这类错误是什么类型”。例如缓冲区溢出、SQL 注入、跨站脚本、Use After Free，都属于弱点类型。

可以这样记：CVE 像身份证号，标识某个具体漏洞；CWE 像病种分类，描述漏洞属于哪一类问题。

读 CVE/NVD 条目时，重点看：

- CVE ID：漏洞编号。
- Description：漏洞描述。
- CVSS：严重程度评分和向量。
- CWE：漏洞类型。
- Affected Products：受影响产品和版本。
- References：厂商公告、补丁、分析文章。
- Published / Modified：发布时间和修改时间。

漏洞库的价值不是“背编号”，而是帮助防御者定位资产、判断风险、安排补丁、写报告、配置扫描器。

### 渗透测试

渗透测试 Penetration Testing 是在授权范围内，模拟攻击者方法，对系统进行安全评估。它不是“随便扫网站”，也不是“炫工具”，而是一种有边界、有流程、有交付的安全验证活动。

渗透测试通常关注：

- 攻击面在哪里：域名、端口、服务、接口、后台、人员入口。
- 漏洞是否真实存在：扫描器报出的风险能否复现。
- 漏洞能造成什么影响：能否越权、泄露数据、拿到权限、横向移动。
- 防御是否有效：监控是否发现、权限是否隔离、补丁是否到位。
- 如何修复：给出可执行的加固建议。

它的特点是渐进、逐步深入，并尽量选择不影响业务正常运行的方法。真正的渗透测试必须有授权。没有授权的扫描、利用、入侵，即使技术动作类似，也不是渗透测试。

#### 黑箱、白箱、隐秘测试

| 方法 | 测试者掌握的信息 | 适合模拟什么 | 特点 |
|---|---|---|---|
| 黑箱测试 Black-box Testing | 几乎不了解系统内部信息 | 外部攻击者 | 从公开信息、端口、页面、接口开始 |
| 白箱测试 White-box Testing | 能获得源码、拓扑、账号、架构、人员信息等 | 内部审计或深度安全评估 | 覆盖更深，效率更高 |
| 灰箱测试 Gray-box Testing | 掌握部分内部信息 | 介于黑箱和白箱之间 | 更接近有一定权限的攻击者 |
| 隐秘测试 Covert Testing | 只有极少数人知道测试存在 | 检验监控、响应和应急能力 | 重点不只是漏洞，也包括检测响应 |

黑箱测试不是“乱猜”，而是从公开信息开始逐步缩小范围。白箱测试不是“不真实”，它能更系统地发现深层问题。隐秘测试的重点是看安全团队能不能发现、响应和恢复。

### 易考点

- 病毒通常依附宿主文件传播；蠕虫更强调网络独立扩散；木马更强调伪装和远程控制。
- 木马通常不强调自我复制，而强调欺骗性、隐蔽性和非授权控制。
- Bug 不一定是漏洞；能被攻击者利用并造成安全影响的 Bug 才是漏洞。
- 肉鸡通常依赖“软件漏洞 + 木马”：漏洞负责入侵，木马负责控制。
- 0day 是未知或未修复漏洞，1day 是补丁/信息刚公开但大量系统未修补的漏洞。
- CVE 是具体漏洞编号，CWE 是漏洞类型分类。
- NVD 全称是 National Vulnerability Database。
- CVSS 是漏洞评分系统，不是漏洞编号系统。
- 渗透测试必须获得授权；无授权测试就是攻击行为。
- 授权扫描一个开放网站、从外部入口开始，通常属于黑箱测试。

### 本讲应该形成的安全观

这门课不是孤立背名词，而是建立因果链：

```text
程序设计/实现错误
-> Bug
-> 可被外部输入触发并造成安全后果
-> 漏洞
-> exploit 触发漏洞
-> payload/shellcode/木马执行
-> 主机被控制成为肉鸡
-> 攻击者继续扩散、窃密、横向移动
-> 防御者通过漏洞库、补丁、检测、渗透测试应对
```

你复习时要养成一个习惯：看到任何安全概念，都问四个问题：

1. 它是什么？
2. 它怎么发生或怎么实现？
3. 它造成什么后果？
4. 它和相邻概念怎么区分？

只背“病毒会复制、木马会伪装、CVE 是漏洞编号”是不够的。考试和真实安全分析都会更关心：为什么会这样、链条怎么走、哪个环节被防御打断。

### 复习自查

1. 病毒、蠕虫、木马分别靠什么传播或建立控制？
2. 为什么木马通常比病毒更强调欺骗性和隐蔽性？
3. 为什么点 URL 或打开 Word 文档也可能触发恶意代码？
4. Bug 和漏洞是什么包含关系？普通 Bug 变成漏洞需要哪些条件？
5. 肉鸡通常怎样形成？攻击者拿肉鸡做什么？
6. 0day 和 1day 的本质区别是什么？为什么 1day 仍然危险？
7. CVE、CWE、NVD、CVSS 分别解决什么问题？
8. 黑箱、白箱、隐秘测试的差别是什么？
9. 为什么说渗透测试的核心不是工具，而是授权范围内的风险验证？

# Lecture02 堆栈基础和汇编语言 - 复习手册

> 本文件基于课程 PDF 整理，用于考前复习和人工复核。

### 本讲主线

Lecture02 是漏洞利用的底层基础。后面所有栈溢出、返回地址覆盖、shellcode、ROP 都建立在“内存怎么分区、函数怎么调用、寄存器怎么保存状态、汇编指令怎么改变执行流”之上。

### 进程内存布局

程序运行起来以后，不再只是磁盘上的一个 exe 文件，而是变成一个进程。进程拥有自己的虚拟地址空间，代码、全局变量、动态申请的对象、函数调用信息都会放在不同区域。理解这些区域，是后面分析“溢出覆盖了谁”的基础。

常见内存区域：

- 代码区：存放程序执行代码，CPU 从这里取指令执行。
- 数据区：存放已初始化的全局变量和静态变量。
- BSS 区：存放未初始化的全局变量和静态变量。
- 堆区：程序运行时动态申请和释放的内存。
- 栈区：函数调用时自动维护的内存区域，保存参数、返回地址、旧栈帧、局部变量等。

可以用变量位置来记：

```c
int g1 = 1;        // 已初始化全局变量，通常在数据区
int g2;            // 未初始化全局变量，通常在 BSS 区

void f() {
    int a = 1;     // 局部变量，通常在栈上
    static int s;  // 静态局部变量，不在栈上，通常在数据区/BSS
    int *p = malloc(100); // p 这个指针变量在栈上，malloc 申请的 100 字节在堆上
}
```

进程内存的精确组织依赖操作系统、编译器、链接器和加载器。课程中使用的模型主要服务于理解攻击，不必把每张图当成所有平台的唯一布局。考试里如果问“某变量在哪个区”，按经典模型判断即可；如果分析真实程序，要结合编译器优化、平台 ABI 和调试信息。

### 堆与栈

堆和栈都能放数据，但它们的管理方式完全不同。栈像函数调用自动带来的临时工作区，函数进来就分配，函数返回就释放；堆像程序运行时向内存管理器申请的一块长期区域，用完必须释放。

| 对比点 | 栈 | 堆 |
|---|---|---|
| 分配方式 | 系统/编译器自动分配 | 程序员或运行库动态申请 |
| 生命周期 | 通常随函数进入/返回自动创建和释放 | 由 malloc/new 申请，free/delete 释放 |
| 速度 | 通常更快 | 通常更慢 |
| 空间 | 较小，受栈大小限制 | 较大，但需管理 |
| 增长方向 | 课程中通常按向低地址增长理解 | 通常向高地址或由分配器管理 |

常见判断：

- 局部变量一般在栈上。
- `new int[200]` 这类动态申请的数组在堆上，指针变量本身可能在栈上。
- 栈先压入的局部变量与后压入变量的地址高低，要结合“栈向低地址增长”的模型判断。

安全上要抓两个点：

1. 栈上经常同时放局部缓冲区、保存的旧 `EBP/RBP`、返回地址，所以栈溢出容易直接威胁控制流。
2. 堆上经常放动态对象、结构体、字符串、链表节点、虚表指针等，所以堆溢出常通过破坏对象字段、函数指针或分配器元数据来利用。

不要只背“栈快堆慢”。对软件安全来说，更关键的是：谁管理生命周期、相邻位置可能是什么、越界后会覆盖什么。

### 堆结构：从操作系统到堆块

堆内存管理要从两层关系理解：操作系统负责粗粒度的内存供给，堆管理器负责细粒度的堆块分配。

```text
操作系统内核
-> 以页为单位管理虚拟内存和物理内存映射
-> 用户态堆管理器向操作系统申请一批内存
-> 堆管理器把这批内存切成堆块
-> 程序通过 malloc/new 得到某个堆块的块身指针
```

操作系统通常不会为每一次 `malloc(8)` 或 `malloc(16)` 单独分配内存。内核管理粒度更粗，常以页为单位，一页常见大小是 4096 字节。频繁进入内核做系统调用开销很大，所以用户态会有一层堆管理器：它先向操作系统申请较大的一片内存，再在用户态内部把这片内存切成许多小块，按需分给程序。

| 层次 | 管理者 | 管理粒度 | 主要任务 |
|---|---|---|---|
| 操作系统内核 | 内核内存管理器 | 页级别 | 分配/提交虚拟内存页，维护页表和权限 |
| 用户态堆管理器 | 运行库或系统提供的堆管理代码 | 堆块级别 | 切分、分配、释放、合并堆块 |

堆管理器不是神秘硬件，也不是每次都由内核直接完成的小分配。它是程序运行时的一部分，运行在用户态。它维护的堆表、链表指针、块首等，都是普通内存数据。堆漏洞之所以危险，正是因为攻击者如果破坏这些普通数据，堆管理器后续分配和释放时就可能按错误的“账本”行动。

#### 堆区、堆头和堆表

进程启动后，操作系统为它建立虚拟地址空间，堆区就是其中用于动态内存分配的一片区域。刚开始时，这片区域可能只是虚拟地址范围；真正物理内存映射和提交会按需发生。

当堆管理器从操作系统获得一批内存后，会在堆区中维护自己的管理信息。课程把这类索引和管理结构称为堆表。堆表用于回答几个问题：

- 哪些堆块空闲？
- 空闲块分别有多大？
- 下一次申请某个大小时，去哪里找合适的空闲块？
- 释放后是否需要和相邻空闲块合并？

Windows 堆中常见堆表包括：

- 空闲双向链表 `freelist`，简称空表。
- 快速单向链表 `lookaside`，简称快表。

`lookaside` 用于加速小块分配，特点是快，但一般不发生堆块合并。课程复习堆溢出时，重点通常放在空表 `freelist`。

#### 堆块：块首和块身

堆区中的内存会被堆管理器切成一个个堆块。堆块是堆管理的基本单位。

```text
堆块 = 块首 + 块身
```

- 块首：堆管理器使用的管理信息，记录块大小、前一块大小、占用/空闲状态等。
- 块身：返回给程序使用的数据区。

程序调用 `malloc(32)` 时，申请的是 32 字节可用数据区。堆管理器实际分配时，还要额外放置块首。课程以 32 位系统为例，块首可能是 8 字节；因此申请 32 字节时，实际堆块可能是：

```text
8 字节块首 + 32 字节块身 = 40 字节堆块
```

堆管理器返回给程序的指针，通常指向块身开头，而不是块首开头。

```text
堆块起始地址
↓
[块首 8 字节][块身：用户可读写数据]
             ↑
             malloc 返回的指针通常指向这里
```

这样设计的目的很直接：程序员拿到指针后，可以直接读写申请到的数据区，不需要知道前面还有块首。反过来，也正因为块首紧挨着用户数据，堆溢出一旦写出块身边界，就可能破坏相邻堆块的块首或其他管理信息。

课件还提到堆块按 8 字节粒度组织，不足 8 字节的部分会向上对齐。比如申请 30 字节时，堆管理器不会真的只切 30 字节，而会考虑块首和对齐，分配成更适合管理的大小。

#### 占用态和空闲态

堆块有两种基本状态：

| 状态 | 含义 | 管理方式 |
|---|---|---|
| 占用态 | 已经分配给程序使用 | 程序通过块身指针读写，之后用 `free/delete` 归还 |
| 空闲态 | 当前没有分配给程序 | 堆管理器把它挂入空表或快表，等待再次分配 |

占用态堆块的块身属于用户数据。空闲态堆块的块身暂时没人使用，堆管理器会复用这片空间保存链表指针。课程中提到空闲块中有两个 4 字节指针：

- `Flink`：Forward Link，指向空闲链表中的一个相邻节点。
- `Blink`：Backward Link，指向空闲链表中的另一个相邻节点。

也就是说，空闲块不只是“空着”。它会被堆管理器挂到某条空闲链表里。用于链接的指针就存在空闲块自身的空间中。一旦这个块重新分配给程序，这些链表指针所在位置会重新变成用户数据区。

这个点对安全非常关键：堆管理器的链表指针、块大小、状态位都在普通内存里。如果堆溢出覆盖了这些字段，后续脱链、合并、再分配等操作就可能被攻击者影响。

#### 空表索引：free[0] 到 free[127]

空表可以理解为“按大小分类的空闲块目录”。课程里提到空表索引数组，也就是 `freelist array`，常见讲法是 `free[0]` 到 `free[127]`。

这些索引项不是把堆内存本身切成 128 份，而是 128 个链表入口。每个入口负责挂一类大小的空闲块。

按课件口径：

- `free[1]` 对应大小为 8 字节的空闲堆块。
- 后续索引项对应的块大小依次增加 8 字节。
- `free[0]` 是特殊的零号空表，挂较大的空闲块，例如大于等于 1024 字节、小于 512KB 的空闲块，并按大小升序排列。

可以这样记：

```text
free[1]   -> 8 字节空闲块链表
free[2]   -> 16 字节空闲块链表
free[3]   -> 24 字节空闲块链表
...
free[127] -> 较小固定粒度空闲块链表
free[0]   -> 大空闲块链表，即零号空表
```

这样做的好处是查找快。程序申请 24 字节时，堆管理器不需要遍历整个堆区，只要先去对应大小的空表项查找；没有合适块时，再查找更大的空表项或零号空表。

#### 堆块分配：查找、脱链、切割

当程序申请一块内存时，堆管理器大致按以下流程处理：

1. 根据申请大小，加上块首，并按 8 字节等粒度向上对齐。
2. 根据对齐后的大小，定位到对应的空表索引项。
3. 如果该空表中有合适空闲块，就把它从链表中取下，这个动作常叫脱链。
4. 修改块首，把状态改成占用态。
5. 返回块身地址给程序。

如果对应空表项没有合适的块，堆管理器会继续找更大的空闲块。找到大块后，可能发生“找零钱”：

```text
较大的空闲块
-> 切出前一部分作为本次申请的占用块
-> 剩余部分重新写块首
-> 把剩余空闲块挂回合适的空表
```

零号空表用于管理较大的空闲块。课件提到的策略是：先反向查看最后一个最大块能否满足；如果能满足，再正向搜索能够满足请求的最小空闲块，然后进行分配或切割。复习时不必把它背成完整算法，但要理解它的目的：在大块里找尽量合适的块，减少浪费。

#### 堆块释放：回收、入链、合并

释放过程和分配过程相反。程序调用 `free/delete` 时，传入的是块身指针。堆管理器会：

1. 根据块身指针向前找到块首。
2. 读取块大小、状态等管理信息。
3. 把堆块状态改为空闲态。
4. 把该空闲块链入对应的空表，课件中提到通常链入相应表尾。

释放后还可能发生堆块合并。堆长期使用后会产生碎片：许多小空闲块散落在占用块之间。如果两个物理上相邻的堆块都空闲，堆管理器可以把它们合并成一个更大的空闲块。

合并大致包括：

1. 检查当前块前后相邻堆块是否空闲。
2. 如果相邻块已经在空表中，先把它们从原链表中脱链。
3. 把相邻空闲块合并成一个物理连续的大块。
4. 重新计算并写入新块首。
5. 把合并后的大空闲块挂入对应空表。

所以释放不是简单标记一下结束，而是可能入链、可能合并、可能重新归类到新的空表项中。

#### 为什么堆结构对安全重要

堆结构的安全意义在于：用户数据和堆管理数据在内存中非常接近。堆管理器依赖块首、大小字段、状态位、`Flink/Blink` 等信息来完成分配、释放和合并。如果攻击者通过堆溢出、释放后使用或双重释放破坏这些信息，堆管理器可能会在后续操作中执行错误的链表修改或错误的内存写入。

常见风险包括：

- 堆溢出覆盖相邻块块首，破坏大小或状态。
- 堆溢出覆盖对象字段、函数指针、虚表指针。
- 覆盖空闲块的 `Flink/Blink`，影响空表脱链。
- Use After Free 中，程序继续使用已经释放且可能被重新分配的块。
- Double Free 让同一块内存重复进入空闲结构，破坏堆管理状态。

因此，学习堆结构不是为了死背 Windows 堆的每个实现细节，而是为了理解后续堆溢出利用的根本原因：攻击者不是直接控制操作系统内核，而是先破坏用户态堆管理器依赖的普通内存数据，再借分配、释放、合并这些正常动作放大影响。

### 函数调用与栈帧

高级语言里的函数调用，在机器层面必须解决三件事：参数怎么传、函数结束后回到哪里、局部变量放在哪里。栈帧就是为一次函数调用组织这些信息的结构。

函数调用时通常发生：

1. 调用者准备参数。
2. `call` 指令把返回地址压栈，并跳转到被调用函数。
3. 被调用函数保存旧 `EBP/RBP`。
4. 建立新的栈帧，让 `EBP/RBP` 作为当前函数栈帧基准。
5. 为局部变量分配栈空间。
6. 函数返回时恢复旧栈帧，并通过返回地址回到调用者。

栈帧通常包括：

- 函数参数。
- 返回地址。
- 保存的旧帧指针。
- 局部变量。

不包括全局变量。

以 32 位 x86 经典模型为例，函数开头常见：

```asm
push ebp
mov ebp, esp
sub esp, 20h
```

含义是：

1. `push ebp`：保存调用者的栈帧基址。
2. `mov ebp, esp`：建立当前函数自己的栈帧基准。
3. `sub esp, 20h`：给局部变量留出空间。

函数结尾常见：

```asm
mov esp, ebp
pop ebp
ret
```

或者用 `leave; ret` 简写。`ret` 的本质是从栈顶取出返回地址，放进 `EIP/RIP`，让 CPU 去那个地址继续执行。所以栈溢出如果覆盖了返回地址，程序返回时就可能跳到攻击者控制的位置。

手推栈帧时，可以按这个顺序画：

```text
高地址
函数参数
返回地址
保存的旧 EBP
局部变量 / 缓冲区
低地址
```

课程中的很多代码题，本质都是问：输入从哪个缓冲区开始写，写多少字节会越过缓冲区，接下来覆盖的是相邻变量、旧 EBP，还是返回地址。

### 关键寄存器

寄存器是 CPU 内部速度最快的小容量存储位置。漏洞分析不需要记住所有寄存器，但必须知道几个“控制程序运行状态”的寄存器。

| 寄存器 | 作用 |
|---|---|
| EIP/RIP | 指令指针，保存下一条将执行指令的地址 |
| ESP/RSP | 栈顶指针 |
| EBP/RBP | 栈帧基址 |
| EAX/RAX | 常用于返回值和通用计算 |
| ECX/RCX | 常用于计数 |
| EDX/RDX | 常用于数据和乘除运算辅助 |

漏洞利用里最关键的是 EIP/RIP，因为控制它就可能控制程序下一步执行位置。

可以这样理解：

- `EIP/RIP` 像书签，告诉 CPU 下一句读哪里。
- `ESP/RSP` 像栈顶指针，告诉 CPU 当前栈顶在哪里。
- `EBP/RBP` 像本函数的参照尺，方便访问参数和局部变量。
- `EAX/RAX` 经常放函数返回值，所以破解、补丁分析常盯它。

如果题目说“劫持进程控制权”，很多时候就是让 `EIP/RIP` 变成攻击者想要的地址。后续 ROP、shellcode、返回地址覆盖都围绕这个目标展开。

### 常见汇编指令

汇编不要当作另一门完整编程语言死背。复习时只要能看懂数据怎么流动、栈怎么变化、控制流怎么跳转，就够支撑本课程。

- `mov dst, src`：赋值/传送。
- `push x`：压栈，栈顶移动。
- `pop x`：出栈。
- `call addr`：压入返回地址并跳转。
- `ret`：从栈中取返回地址并跳转。
- `cmp a, b`：比较，实质是影响标志位。
- `jmp/je/jne`：无条件/条件跳转。
- `and/xor/or`：位运算。

`xor eax, eax` 常用于把 `EAX` 清零。`cmp eax, eax` 后零标志 `ZF=1`。

几个容易混的点：

- `mov eax, ebx`：把 `EBX` 里的值复制到 `EAX`。
- `mov eax, [ebx]`：把 `EBX` 当地址，读取该地址处的内存内容到 `EAX`。
- `call addr`：不仅跳转，还会自动把返回地址压栈。
- `ret`：不仅返回，还会从栈里取返回地址并跳过去。
- `cmp a, b`：不保存相减结果，但会影响标志位，后面的 `je/jne/jg/jl` 根据标志位决定是否跳转。

很多破解题和调试题就是观察：比较指令设置了什么标志位，条件跳转有没有跳，返回值是否被改成 0 或 1。

### 寻址与标志位

寻址方式回答“操作数从哪里来”。方括号通常表示访问内存，这一点特别重要。

常见寻址：

- 立即数寻址：操作数直接在指令里。
- 寄存器寻址：操作数在寄存器中。
- 直接寻址：直接给出内存地址。
- 间接寻址：地址存放在寄存器中。
- 基址/变址/偏移/相对寻址：如 `[BX+1200H]`。

常见标志位：

- ZF：零标志，结果为 0 时置位。
- CF：进位/借位。
- SF：符号标志。
- OF：溢出标志。

例子：

```asm
mov al, 12h       ; 立即数寻址，把 12h 放入 AL
mov al, bl        ; 寄存器寻址，把 BL 放入 AL
mov al, [3100h]   ; 直接寻址，读取内存 3100h 处的数据
mov al, [bx]      ; 间接寻址，BX 中保存地址
mov al, [bx+4]    ; 基址加偏移
```

第一份课后题里的 `MOV AL, [3100H]` 就是直接寻址：把内存地址 `3100H` 处的数据读入 `AL`，不是把数字 `3100H` 本身放入 `AL`。

### 易考点

- 函数栈帧不包括全局变量。
- 函数返回依赖把保存的返回地址装入 EIP/RIP。
- 寄存器访问通常快于缓存。
- `push ebp; mov ebp, esp` 执行后，`ESP` 与 `EBP` 相等。
- `call` 会把返回地址压栈，`ret` 会从栈中取返回地址。
- `mov eax, [ebx]` 中方括号表示访问内存。
- 立即寻址的操作数直接写在指令里，通常不需要额外访问内存取操作数。
- 栈帧里局部变量、保存的 EBP、返回地址相邻，是栈溢出可利用的基础。
- 堆块由块首和块身组成，程序拿到的指针通常指向块身，不指向块首。
- 堆块有占用态和空闲态，空闲态堆块会被堆管理器链入空闲结构。
- Windows 堆表中常见空表 `freelist` 和快表 `lookaside`，堆溢出复习重点通常放在空表。
- 空表按大小组织空闲堆块，`free[0]` 是较大的零号空表。
- 堆块分配可能从大空闲块中切出一块，剩余部分重新链回空表。
- 堆块释放后可能与相邻空闲块合并，以减少碎片。

### 为什么软件安全必须学栈

栈是理解二进制漏洞的入口。高级语言里一次函数调用看起来只是 `f(a, b)`，但 CPU 实际执行时需要知道参数在哪里、函数结束后回到哪里、局部变量放在哪里。栈帧就是把这些信息组织起来的一段内存。

在 32 位 x86 经典模型中，`call` 指令会把返回地址压入栈，然后跳到被调用函数。函数开头常见序言是：

```asm
push ebp
mov ebp, esp
sub esp, N
```

第一句保存调用者的栈帧基址；第二句让 `EBP` 成为当前函数栈帧的稳定参考点；第三句为局部变量留空间。函数结尾会恢复栈帧，再执行 `ret`，`ret` 从栈顶取出返回地址放入 `EIP`。因此，只要攻击者能把栈上的返回地址改成别的值，程序就会跳到别处执行。

栈和堆的区别不要只背“栈快堆慢”。对安全来说，更重要的是生命周期和控制权：栈由函数调用自动管理，所以返回地址、旧 `EBP`、局部变量经常紧挨着；堆由分配器管理，适合长期对象和动态大小对象。栈溢出常直接威胁返回地址，堆溢出常威胁相邻对象、堆元数据、函数指针或虚表指针。

寄存器可以理解为 CPU 手边的小工作台。内存很大但慢，寄存器很少但快。`EIP/RIP` 最关键，因为它决定下一条指令；`ESP/RSP` 决定当前栈顶；`EBP/RBP` 让函数稳定访问参数和局部变量。漏洞利用里常说“控制 EIP/RIP”，意思就是让程序下一步去执行攻击者选择的位置。

汇编指令的学习目标不是写大型程序，而是能读懂控制流和数据流。`mov` 改数据，`push/pop` 改栈，`call/ret/jmp` 改执行路径，`cmp/test` 配合条件跳转决定分支。看到一段反汇编时，先问三个问题：数据从哪里来，写到哪里去，下一步跳到哪里。

寻址方式决定“方括号里到底是谁”。`mov eax, ebx` 是把 `EBX` 的值给 `EAX`；`mov eax, [ebx]` 是把 `EBX` 指向的内存内容读到 `EAX`。安全分析中，很多越界读写、野指针、任意地址读写，都和“把值当地址”有关。

### 手推栈帧的小方法

1. 先画出高地址和低地址。
2. 标出栈增长方向，课程中通常按向低地址增长理解。
3. 遇到 `push`，栈顶移动并写入数据。
4. 遇到 `call`，记得返回地址会自动入栈。
5. 以 `EBP` 为基准，参数通常在 `EBP+偏移`，局部变量通常在 `EBP-偏移`。
6. 看到 `ret`，就看当前栈顶保存的返回地址是什么。

### 知识点一：PE 文件格式

PE，Portable Executable，是 Windows 平台常见可执行文件格式。`.exe` 和 `.dll` 都是典型 PE 文件。ELF，Executable and Linkable Format，是 Unix/Linux 系统常见格式。

可执行文件不是只有机器码。一个 PE 文件还会包含：

- 程序代码。
- 已初始化数据。
- 资源，例如图标、菜单、位图、字体。
- 导入表，记录依赖哪些 DLL 和函数。
- 导出表，记录自己提供哪些函数。
- 重定位信息。
- TLS 等初始化相关信息。

PE 格式的意义是给操作系统 Loader 一个标准说明书：文件头在哪里，入口点在哪里，各个 section 在文件中和内存中分别在哪里，依赖哪些外部库，哪些地址需要重定位。

#### PE 与 ELF 的共同点

PE 和 ELF 是不同系统上的可执行文件格式，但思想相似：

- 前面的 Header 提供元数据。
- 后面的 section/segment 才是真正的代码、数据和资源。
- Loader 根据格式信息把文件映射到进程虚拟地址空间。
- 不同 section 映射到内存后可以有不同权限，例如代码节可读可执行，数据节可读可写。

PE 是本课程重点，ELF 只需要知道它在 Unix/Linux 中承担类似角色。

#### PE 典型结构

PE 常见结构：

- DOS Header 和 DOS Stub：历史兼容。DOS Header 会告诉 Loader 真正 PE Header 在哪里；DOS Stub 对现代 Windows Loader 基本无实际执行意义。
- PE Signature：`PE\0\0`，标识真正 PE 头开始。
- COFF File Header：记录 CPU 架构、section 数量、时间戳、Optional Header 大小、文件属性等。
- Optional Header：名字叫 Optional，但对可执行文件非常重要，记录如何加载，包括 `AddressOfEntryPoint`、`ImageBase`、`SectionAlignment`、`FileAlignment`、`SizeOfImage`、Data Directory 等。
- Section Table：记录各个 section 的名称、文件偏移、虚拟地址、大小、属性。
- Section 数据：真正的代码、数据、资源等内容。

Optional Header 中几个字段很重要：

| 字段 | 含义 |
|---|---|
| `ImageBase` | 模块希望被装入内存的基址 |
| `AddressOfEntryPoint` | 程序入口点的 RVA |
| `SectionAlignment` | section 在内存中的对齐粒度 |
| `FileAlignment` | section 在磁盘文件中的对齐粒度 |
| `DataDirectory` | 各种重要表的位置，如导入表、导出表、资源表、重定位表、TLS 表 |

如果对齐不正确，可能导致加载效率低、内存保护无法正确设置，甚至 Loader 无法直接映射。

#### 常见 PE 节

PE 文件被分成若干 section，不同内容放在不同节中：

| 节名 | 常见内容 | 复习记忆 |
|---|---|---|
| `.text` | 编译器产生的机器代码 | 反汇编和调试主要看这里 |
| `.data` | 已初始化数据，如全局变量、静态变量 | 可读写数据 |
| `.rdata` | 只读数据 | 字符串常量、只读表 |
| `.bss` | 未初始化数据 | 文件中不一定真正占空间，加载后内存中要分配 |
| `.rsrc` | 资源，如图标、菜单、位图 | Resource |
| `.idata` | 导入表，记录 DLL 和外来函数信息 | Import data，易考 |
| `.edata` | 导出表 | Export data |
| `.reloc` | 重定位信息 | Relocation |
| `.tls` | 线程局部存储和 TLS 回调相关信息 | Thread Local Storage |

易考：存放动态链接库等外来函数与文件信息的是 `.idata` 或导入表。

#### Loader 加载 PE 的核心步骤

PPT 中给出的加载核心步骤是：

1. 解析可执行文件结构。
2. 构建进程内存映像。
3. 地址重定位。
4. 解析外部依赖。
5. 运行初始化代码。
6. 启动程序入口。

展开理解：

1. Loader 根据 DOS Header 找到 PE Header，读取 `PE\0\0`，确认文件合法。
2. 读取 COFF File Header，知道 CPU 架构、section 数量等。
3. 读取 Optional Header，得到 `ImageBase`、入口点、对齐方式、Data Directory。
4. 为整个 Image 分配虚拟地址空间。
5. 读取 Section Table，把 `.text`、`.data`、`.rsrc` 等映射到内存。
6. 按 `ImageBase + VirtualAddress` 计算 section 在内存中的位置。
7. 查找 Import Table，加载所需 DLL，解析函数地址，填入 IAT。
8. 如果首选 `ImageBase` 被占用或 ASLR 生效，进行 relocation 修正。
9. 处理 TLS 等初始化逻辑。
10. 最终跳转到入口点执行。

这里的重点是：PE 文件不是直接“从第一字节开始执行”。操作系统先按 PE 结构加载和修正，最后才跳到入口点。

#### 导入表和 IAT

动态库相关内容是第三章重点之一。程序调用 `printf`、`MessageBoxA` 等函数时，这些函数代码通常不在主程序自己的 `.text` 节里，而在外部 DLL 中。

编译/链接阶段：

1. 程序记录自己依赖哪些动态库，例如 `kernel32.dll`、`msvcrt.dll`、`user32.dll`。
2. 程序为导入函数建立跳转表，即 IAT，Import Address Table，输入函数地址表。
3. 此时函数真实地址还未知。

运行时阶段：

1. Loader 加载所有需要的 DLL。
2. 为每个 DLL 分配加载地址。
3. 解析导入函数符号，例如 `printf = msvcrt.dll + 某个偏移`。
4. 把真实函数地址写入 IAT，例如 `IAT[printf] = 真实地址`。
5. 程序执行 `call [IAT printf]` 时，就能跳到正确 DLL 函数。

IAT 的安全意义：

- 程序兼容不同系统/DLL 版本，因为真实地址运行时才填。
- 分析者可以通过 IAT 看程序调用了哪些外部 API。
- 攻击者可能通过 IAT Hook 修改函数跳转目标。
- PE 注入实验中调用 `MessageBoxA` 能成功，是因为目标 PE 的导入表/IAT 已经包含相关 API。

### 加壳与脱壳

PPT 在 PE 文件格式后讲了加壳。加壳不是修改程序功能，而是改变可执行代码在磁盘文件中的形态。

加壳，简单说，是对 EXE/DLL 的代码和资源做压缩或加密，再附加一段壳代码 loader stub。加壳后的程序仍然能运行，但磁盘上的原始代码常常变成压缩或加密形式，静态反汇编不容易直接看到真实逻辑。

加壳常见目的：

- 保护软件版权。
- 增加逆向分析难度。
- 降低程序体积。
- 攻击者隐藏代码，躲避安全检测。

加壳的大致步骤：

1. 解析原始 PE 文件。
2. 压缩或加密 `.text`、`.rdata`、`.data` 等 section。
3. 插入加载器桩代码 loader stub。
4. 改变程序入口点，让壳代码先执行。

运行时过程：

```text
Windows Loader 加载加壳程序
-> 先执行壳代码 stub
-> stub 在内存中解密/解压原始程序
-> 修复必要结构
-> 跳回原始入口点 OEP
-> 原程序正常执行
```

压缩壳侧重减小体积；加密壳侧重保护和反调试，不同壳还可能加入注册限制、次数限制、时间限制等功能。反病毒和 Crack 分析中经常遇到节名古怪、入口点异常、导入表异常的加壳程序。

### 知识点二：虚拟内存

Windows 使用用户模式和内核模式隔离普通程序与操作系统核心。

| 模式 | 运行内容 | 权限 |
|---|---|---|
| 用户模式 User Mode | 普通用户程序 | 权限低，不能直接访问硬件和关键内核内存 |
| 内核模式 Kernel Mode | 操作系统内核、系统服务、设备驱动 | 权限高，可访问所有内存和硬件，执行特权指令 |

系统调用是用户态和内核态之间的桥。普通程序需要文件、网络、内存、进程等系统资源时，不能直接操作硬件，而是通过系统调用请求内核完成。

#### 虚拟地址与物理地址

Windows 内存可以分为两个层面：

- 物理内存：真实 RAM，对应物理地址。通常需要内核级别才能直接观察。
- 虚拟内存：进程看到的逻辑地址空间。用户态调试器看到的地址通常是虚拟地址。

用户程序使用的地址称为虚拟地址或逻辑地址。计算机物理内存访问地址称为物理地址或实地址。虚拟地址到物理地址的转换由 MMU，Memory Management Unit，结合页表完成。

为什么要区分虚拟内存和物理内存：

- 地址空间隔离：每个进程有独立虚拟地址空间，程序 A 不会直接改到程序 B 的数据。
- 内存保护：不同页可设置不同权限，例如只读、可写、可执行。
- 装载方便：程序不必关心自己最终被放到哪个物理地址。
- 按需加载：只把正在用的页加载到 RAM。
- 支持交换空间和共享内存。

32 位 Windows 经典语境下，一个进程可拥有 4GB 虚拟地址空间。现在 x64 已突破 4GB 限制，但“每个进程拥有独立虚拟地址空间”的机制仍然成立。

易考：用户模式下，用调试器看到的内存地址通常是虚拟内存地址，不是物理地址。

#### ASLR 与 relocation

ASLR，Address Space Layout Randomization，地址空间布局随机化。它随机化关键内存区域，例如 stack、heap、libc/DLL、code 的加载位置。

为什么需要 ASLR：

- 如果地址固定，攻击者更容易预测 shellcode、库函数或 gadget 地址。
- 覆盖返回地址时，攻击者可以写死跳转目标。

ASLR 生效后，Loader 会随机选择地址映射程序或 DLL，并通过 relocation 修正需要绝对地址的位置。CPU 不“理解 ASLR”，它只执行当前进程里的虚拟地址。随机化和修正工作由 Loader 和操作系统完成。

ASLR 不是绝对安全，常见绕过包括：

- 信息泄漏。
- 暴力尝试。
- 部分地址覆盖。

### PE 文件与虚拟内存的映射

调试漏洞时，经常需要在“文件位置”和“内存位置”之间转换。

两种常见需求：

1. 静态反汇编工具看到某条指令在 PE 文件中的位置，即文件偏移。我们想知道它运行时在内存中的虚拟地址。
2. 动态调试器看到某条指令的虚拟地址。我们想回到 PE 文件中找到对应机器码。

#### VA、RVA、文件偏移、ImageBase

| 概念 | 全称 | 含义 |
|---|---|---|
| VA | Virtual Address | 虚拟地址，PE 装入内存后的地址 |
| RVA | Relative Virtual Address | 相对虚拟地址，相对于 ImageBase 的偏移 |
| File Offset | 文件偏移 | 数据在磁盘 PE 文件中相对于文件开头的位置 |
| ImageBase | 装载基址 | PE 希望被装入内存的基地址 |

核心公式：

```text
VA = ImageBase + RVA
RVA = VA - ImageBase
```

默认情况下，EXE 的常见 ImageBase 是 `0x00400000`，DLL 常见 ImageBase 是 `0x10000000`。这些位置可以通过编译选项更改，实际运行时也可能因 ASLR 或地址占用而变化。

#### 文件对齐与内存对齐

文件偏移和 RVA 有一定一致性，但不能直接等同。原因是磁盘文件和内存中的 section 对齐粒度不同。

PPT 中给出的典型值：

- PE 文件中的数据按磁盘标准存放，常以 `0x200` 字节为基本单位组织。section 不足 `0x200` 会用 `0x00` 填充。
- 代码装入内存后，按内存标准存放，常以 `0x1000` 字节为基本单位组织。

为什么内存中的 section 更大：

- `.bss` 等未初始化数据在文件中不需要实际存储，但内存中需要分配空间。
- Loader 把 section 加载到内存时需要按页对齐。
- 文件中 section 可以较紧凑，内存中必须满足页和权限管理要求。

所以，不能简单把文件偏移当运行时地址。真实换算要结合 Section Table。

#### LordPE 例子

PPT 中用 LordPE 查看节信息：

- `VOffset` 是 RVA。
- `ROffset` 是文件偏移。

例子：`.text` 节的 `VOffset = 0x11000`，ImageBase 为 `0x400000`，则该节在内存中的虚拟地址为：

```text
0x400000 + 0x11000 = 0x411000
```

如果同一节在文件中的 `ROffset = 0x1000`，则可以用二进制编辑器在文件偏移 `0x1000` 附近找到对应机器码。这个例子就是为了说明：同一段代码，在文件中有文件偏移，在内存中有 VA/RVA，两者需要通过节表关联。

LordPE 还能查看：

- Section 信息。
- 导入表。
- 导出表。
- 地址换算。

导入表例子：PPT 中导入表在文件里的 `ROffset = 0x24000`，RVA 是 `0x25000`。打开目录表可以看到输入表 RVA 确实是 `0x25000`，再通过左侧按钮查看具体输入表内容。

常见 PE/二进制分析工具：

- LordPE：Windows 下 PE 信息查看、修改、脱壳辅助。
- PEView：直观显示 PE 文件结构。
- `readelf`：Linux 下查看 ELF。
- `objdump`：查看 section、反汇编。
- Cutter：跨平台 GUI 逆向工具。
- ImHex：逆向工程可视化和十六进制分析。
- IDA Pro：反汇编、反编译、分析多种可执行文件。

### 知识点三：调试分析工具

调试工具的目标是把程序运行时状态看见。静态工具看文件结构和代码逻辑，动态调试器看寄存器、内存、栈、指令执行和分支跳转。

#### OllyDbg

OllyDbg 是 32 位可视化汇编分析调试器，适合动态调试。它可以直接打开一个 exe 调试，也可以附加到已经运行的进程。

两种载入方式：

- 文件 -> 打开，快捷键 `F3`，打开可执行文件进行调试。
- 文件 -> 附加，附加到已经运行的进程。

常用快捷键：

| 快捷键 | 作用 |
|---|---|
| `F2` | 设置断点 |
| `F8` | 单步步过，遇到 `CALL` 不进入子函数 |
| `F7` | 单步步入，遇到 `CALL` 会进入子函数 |
| `F4` | 运行到选定位置 |
| `F9` | 运行 |
| `Alt+F9` | 执行到用户代码，用于从系统代码区域返回到被调程序区域 |
| `Ctrl+F9` | 执行到返回，在遇到 `ret` 时暂停 |

Trace 跟踪功能用于记录调试过程中执行过的指令，有助于分析断点之前发生了什么。Trace 可选择是否记录寄存器值。

#### IDA Pro

IDA，Interactive Disassembler，交互式反汇编工具，是逆向分析主流工具。它偏静态分析，也有调试能力。打开文件后，IDA 会分析函数、字符串、导入导出、控制流等。

IDA 常见视图和窗口：

- Proximity View：展示函数或代码块之间的关系。
- IDA View：主要反汇编窗口。
- Graph View：图形视图，把一个函数分成基本块，显示控制流。
- Text View：文本视图，显示完整反汇编清单，也能看数据部分。
- Names Window：全局名称窗口。
- Strings Window：字符串窗口。
- Functions Window：函数列表。
- Function Call Window：函数调用关系窗口。
- Pseudocode Window：反编译伪代码窗口。

Graph View 中：

- 函数被分成多个基本块。
- Yes 分支箭头通常为绿色。
- No 分支箭头通常为红色。
- 蓝色箭头表示指向下一个即将执行的块。

Text View 中：

- 虚拟地址常以 `[区域名称]:[虚拟地址]` 显示，例如 `.text:0040110C0`。
- 实线箭头通常表示非条件跳转。
- 虚线箭头通常表示条件跳转。
- 粗线跳转往往提示控制流回到某个地址，常见于循环。

Names Window 列举二进制文件的全局名称。名称是对虚拟地址的符号描述。课后题提到：Names 窗口中名称前的 `I` 通常表示 imported name，即导入名称。

Strings Window 显示二进制中提取的字符串以及地址。双击字符串会跳转到该字符串位置；结合交叉引用，可以追踪代码中哪些位置引用了该字符串。这在破解和漏洞分析里很常用，例如通过 “wrong password” 定位错误分支。

Functions Window 显示所有函数。条目可显示函数地址、所在 section、函数长度。例如在虚拟地址 `00401040` 的 `.text` 部分找到 `_main` 函数，该函数长度为 `0x50` 字节。

IDA 新版本支持反编译。选中汇编代码后按 `F5`，IDA 会在 Pseudocode 窗口中显示 C/C++ 风格伪代码。这有助于快速理解逻辑，但伪代码不是源码，仍需结合汇编判断。

### 知识点四：PE 文件代码注入示例

PPT 演示的是授权实验环境下对 Windows XP 扫雷程序 `winmine.exe` 做 PE 注入。目的不是让你背步骤，而是理解代码注入的三个关键动作：

1. 把新代码写入 PE 文件某个可用位置。
2. 让程序执行流先跳到新代码。
3. 新代码执行后跳回原始入口点，保证原程序继续运行。

实验目标：让扫雷程序运行前先弹出一个 `MessageBoxA` 对话框，显示注入提示，然后再正常启动扫雷。

#### 找入口点

用 OllyDbg 打开扫雷程序，程序停在入口点。PPT 中：

- LordPE 查看入口点 RVA 是 `0x00003E21`。
- 装载基址是 `0x01000000`。
- 因此入口点 VA 是 `0x01000000 + 0x00003E21 = 0x01003E21`。
- OllyDbg 中 EIP 也显示 `0x01003E21`，注释提示 `ModuleEntryPoint`。

这就是 VA/RVA 公式在实际调试中的使用。

#### 写入注入代码

实验在代码空白区域写入数据和汇编。计划注入代码功能：

- 构造标题字符串，例如 `PE Inject`。
- 构造内容字符串，例如 `You are Injected!`。
- 调用 `MessageBoxA`。
- 跳回原始入口点 `0x01003E21`。

`MessageBoxA` 函数原型：

```c
int MessageBoxA(
    HWND hWnd,
    LPCSTR lpText,
    LPCSTR lpCaption,
    UINT uType
);
```

参数含义：

- `hWnd`：所属窗口句柄，`NULL` 表示不属于任何窗口。
- `lpText`：消息框内容字符串指针。
- `lpCaption`：消息框标题字符串指针。
- `uType`：消息框风格，`NULL` 表示默认风格。

Windows API 中 `MessageBox` 会根据字符串类型选择 ASCII 版本 `MessageBoxA` 或 Unicode 版本 `MessageBoxW`。实验使用 `MessageBoxA`。

32 位常见调用约定下，参数从右到左压栈。因此调用代码形态是：

```asm
push 0              ; uType，默认风格
push 标题字符串地址 ; lpCaption
push 内容字符串地址 ; lpText
push 0              ; hWnd
call MessageBoxA
jmp  原始入口点
```

PPT 中示例地址：

```asm
push 0
push 0x01004AA7    ; 标题字符串地址
push 0x01004A9C    ; 内容字符串地址
push 0
call MessageBoxA
jmp 0x01003E21     ; 回到原始入口点
```

字符串后面要保留 `0x00` 结束符。`call MessageBoxA` 能成功，是因为 PE 输入表里已经有这个函数的入口地址或可通过导入机制定位。

#### 挂接入口点

只把代码写到空白区域还不够。代码必须被执行，才算真正注入。实验中通过 LordPE 修改程序入口点，让程序启动时先执行注入代码。

PPT 中注入代码起始 VA 是 `0x01004ABA`，ImageBase 是 `0x01000000`，所以新入口点 RVA 是：

```text
0x01004ABA - 0x01000000 = 0x00004ABA
```

因此只需要把 PE 入口点 RVA 改成 `0x00004ABA`。运行后流程变成：

```text
程序启动
-> 跳到注入代码
-> 弹出 MessageBoxA
-> jmp 回原始入口点 0x01003E21
-> 扫雷程序继续正常运行
```

这说明代码注入的本质不是“凭空多了代码”，而是修改了程序控制流。

### 知识点五：软件破解示例

PPT 的破解示例是一个简单密码验证程序。源代码逻辑大意：

```c
#define password "12345678"

bool verifyPwd(char *pwd) {
    int flag;
    flag = strcmp(password, pwd);
    return flag == 0;
}

void main() {
    char pwd[1024];
    while (1) {
        scanf("%s", pwd);
        if (verifyPwd(pwd)) {
            printf("passed\n");
            break;
        } else {
            printf("wrong password, please input again:\n");
        }
    }
}
```

程序的安全逻辑是：`strcmp` 返回 0 表示密码相等，`verifyPwd` 返回 true，进入 `passed` 分支。

PPT 讲了两种破解思路，本质都是修改控制流或返回值。

#### 破解方式一：改条件跳转

OllyDbg 中可以通过字符串快速定位关键分支：

1. 运行程序，输入错误密码，看到输出 `wrong password`。
2. 在 OllyDbg 反汇编窗口中，右键选择“查找 -> 所有引用的字符串”。
3. 用 `Ctrl+F` 搜索 `wrong`。
4. 双击定位到引用该字符串的代码位置。
5. 观察附近条件跳转，找到决定进入成功分支还是失败分支的指令。

PPT 中核心分支是 `jz`。如果 `jz` 条件成立，会跳到显示错误密码的分支。把 `jz` 改为 `jnz` 后，逻辑反过来：错误密码也会进入成功分支。

这个例子的本质：

```text
原程序：条件成立 -> 失败分支
修改后：条件不成立 -> 失败分支
结果：原本失败的输入可能进入成功分支
```

修改当前汇编代码后，只是在调试器内存中修改。如果要真正修改 exe 文件，需要把修改复制到可执行文件并保存。

#### 破解方式二：改函数返回值

第二种方式是进入 `verifyPwd` 函数，直接修改返回值。

关键点：

- `strcmp` 的返回值在 `EAX` 中。
- 程序把 `strcmp` 结果保存到局部变量 `flag`。
- 比较 `flag` 是否等于 0。
- `sete al` 根据比较结果设置 `AL`。相等则 `AL=1`，否则 `AL=0`。
- 函数返回值常通过 `EAX/AL` 表达。

典型汇编含义：

```asm
; strcmp 返回值在 eax
; flag = eax
xor eax, eax              ; eax 清零
cmp dword ptr [ebp-8], 0  ; 比较 flag 和 0
sete al                   ; 如果相等，则 al = 1
```

如果想强制函数返回 true，可以把相关代码改成：

```asm
mov al, 01
nop
```

这样不管 `strcmp` 结果如何，函数都返回真。

易考指令：

- `XOR EAX, EAX` 执行后，`EAX = 0`。
- `CMP EAX, EAX` 会设置相等标志。
- `CMP EAX, EAX; SETE AL` 后，因为比较相等，`AL = 1`。

### 易考点

- PE 是 Windows 平台 Portable Executable，ELF 是 Unix/Linux 常见 Executable and Linkable Format。
- `.idata` / 导入表保存外部 DLL 和函数信息。
- Loader 加载 PE 要解析头部、映射 section、处理导入表、重定位、TLS，最后跳到入口点。
- IAT 是 Import Address Table，运行时填入导入函数真实地址。
- 加壳会插入 loader stub，并改变入口点，让壳代码先执行。
- 用户态调试器看到的通常是虚拟地址。
- `VA = ImageBase + RVA`。
- 文件偏移和 RVA 不同，因为文件对齐和内存对齐不同。
- LordPE 中 `VOffset` 是 RVA，`ROffset` 是文件偏移。
- OllyDbg：`F2` 断点，`F7` 步入，`F8` 步过，`F9` 运行，`Ctrl+F9` 执行到返回。
- IDA 文本视图中实线箭头通常表示非条件跳转，虚线箭头表示条件跳转。
- IDA Names 窗口中 `I` 通常表示导入名称。
- IDA 按 `F5` 可生成伪代码。
- PE 注入实验中，修改入口点只是改 RVA，不是直接填 VA。
- `MessageBoxA(hWnd, lpText, lpCaption, uType)` 压栈顺序是 `uType`、`lpCaption`、`lpText`、`hWnd`。
- 软件破解常通过修改条件跳转或强制修改返回值实现。
- `XOR EAX, EAX` 后 `EAX=0`。
- `CMP EAX, EAX; SETE AL` 后 `AL=1`。

### 常见分析问题

7. 函数返回值放在哪个寄存器？
8. 如果修改了代码，是否已经保存回可执行文件？

# Lecture04 缓冲区溢出漏洞 - 复习手册

> 本文件基于课程 PDF 整理，用于考前复习和人工复核。

### 本讲主线

Lecture04 从攻击流程讲到漏洞类型，再重点讲栈溢出、堆溢出和其他溢出。它是后续 exploit、shellcode、DEP/ASLR/ROP 的核心基础。

### 攻击生命周期

课程中把攻击流程拆成两种视角：

业务/攻击生命周期：

- 收集信息。
- 制作攻击工具。
- 发送钓鱼邮件或投放恶意文件。
- 利用漏洞。
- 执行攻击。
- 建立长期控制，例如后门或木马。

技术流程：

1. 信息收集：软件、版本、服务、端口、Web 架构和库。
2. 漏洞发现：找出可利用弱点。
3. 利用开发：控制 EIP/RIP、任意读写、绕过 ASLR/DEP/canary。
4. 初始入侵：获取入口。
5. 后渗透：提权、持久化、横向移动等。
6. 清理痕迹。
7. 武器化。

这里的重点不是背顺序，而是理解“攻击不是一个动作，而是一条链”。比如针对某个服务器：

1. 先收集它运行什么系统、什么服务、什么版本。
2. 再判断这些版本是否存在公开漏洞。
3. 如果有漏洞，构造能触发漏洞的输入。
4. 触发后观察能否崩溃、泄露地址、覆盖返回地址或函数指针。
5. 如果能控制执行流，再考虑 payload、权限、持久化。
6. 最后攻击者可能清理日志或把这次利用做成自动化工具。

缓冲区溢出只是这条链中的“利用漏洞”环节。考试和实际分析都要避免一个误区：能崩溃不等于能利用，能利用也不等于一定能稳定拿权限。

### 漏洞类型

课程列出的漏洞大类包括：

- 内存安全漏洞。
- 注入漏洞。
- 授权漏洞。
- 加密类漏洞。
- 业务逻辑漏洞。
- 信息泄漏漏洞。
- 反序列化与类型混淆漏洞。
- 拒绝服务漏洞。

缓冲区溢出属于内存安全漏洞，是因为程序对内存边界管理错误，导致数据写出预期范围。

“溢出漏洞”这个名字容易让人以为只有字符串太长才算。实际上，只要程序把数据写到预期缓冲区之外，都可以形成越界写。触发入口可以是：

- 用户输入的字符串。
- 网络包字段。
- 文件格式字段。
- 图片、压缩包、字体、文档等复杂格式。
- 环境变量、命令行参数。
- Web 表单参数或上传文件内容。

溢出后的结果也不止一种：

- 覆盖普通变量，改变程序逻辑。
- 覆盖返回地址，改变函数返回后的执行位置。
- 覆盖函数指针、虚表指针、回调指针，改变间接调用目标。
- 覆盖长度字段、权限字段、路径字段，造成后续更大影响。
- 破坏堆元数据，诱导分配器产生任意写能力。

### 栈溢出

栈溢出的典型原因是把过长输入写入固定长度局部缓冲区。若没有边界检查，输入会覆盖栈帧中相邻数据，严重时覆盖返回地址。

攻击者关注：

- 能否控制写入内容。
- 能否覆盖返回地址或函数指针。
- 能否让 EIP/RIP 跳到攻击者想要的位置。
- 能否绕过 canary、DEP/NX、ASLR 等防护。

理解栈溢出时最重要的是：溢出不是“程序崩溃”这么简单，而是“越界写导致控制数据被改写”。

经典栈溢出的形态可以这样画：

```text
高地址
函数参数
返回地址
保存的旧 EBP
局部变量 auth/check
char buf[40]
低地址
```

如果程序把超过 40 字节的数据写入 `buf`，第 41 字节开始就会写到 `buf` 之外。具体先覆盖谁，取决于编译器安排的局部变量顺序和平台栈方向。课程和课后题通常会给出布局，按题图推即可。

栈溢出常见利用层次：

1. 覆盖相邻变量：例如把 `authenticated` 从 0 改成 1。
2. 覆盖保存的 `EBP`：影响函数返回时恢复栈帧。
3. 覆盖返回地址：函数 `ret` 时跳到攻击者指定地址。
4. 配合 shellcode/ROP：进一步执行 payload 或复用已有代码。

代码题手推方法：

1. 先看缓冲区大小，例如 `char buf[8]`。
2. 再看输入函数是否限制长度，例如 `strcpy` 不限制，`fgets` 可限制。
3. 按题目给定布局计算偏移：填满缓冲区需要多少字节，覆盖目标变量需要多少字节。
4. 注意字符串结束符 `\0` 也会写入内存，某些题会专门考这个。
5. 最后看被覆盖变量什么时候被使用，是立即判断、函数返回，还是异常处理触发。

### 堆溢出

堆溢出发生在动态分配内存区域。它可能覆盖：

- 相邻堆块的数据。
- 堆管理元数据。
- 函数指针或对象指针。
- 虚表指针等间接控制流数据。

堆溢出比栈溢出更依赖具体堆分配器、对象布局和运行时状态。

堆溢出的难点是“相邻关系不如栈固定”。栈帧通常由编译器按函数局部变量组织，而堆块由内存分配器根据申请顺序、大小、释放情况安排。攻击者如果想利用堆溢出，往往要先想办法让目标对象和溢出缓冲区相邻。

例如：

```c
char *buf1 = malloc(8);
char *buf2 = malloc(8);
strcpy(buf1, "AAAAAAAAhacked");
```

如果题目说明 `buf1` 后面紧接 `buf2`，那么前 8 个 `A` 填满 `buf1`，后面的 `hacked` 就会写入 `buf2` 的开头。真实系统中还要考虑堆块头、对齐、分配器策略和 ASLR，但考试题通常会把布局条件给清楚。

堆溢出可利用的对象包括：

- 相邻对象中的权限标志。
- C++ 对象的虚表指针 vptr。
- 函数指针或回调指针。
- 链表前后指针。
- 堆管理结构中的 freelist 指针。

所以堆溢出不一定直接覆盖返回地址，它更常见的路线是破坏对象或分配器状态，间接获得控制流或任意写能力。

### 任意读写与 DWORD SHOOT

课程提到的 DWORD SHOOT 可以理解为攻击者获得“向任意位置写入任意数据”的能力。任意写非常危险，因为它可能修改返回地址、函数指针、GOT/IAT 项、权限标志或其他关键控制数据。任意读则可用于泄露地址、绕过 ASLR、读取敏感信息。

“任意写”比普通越界写更抽象。普通溢出可能只能从某个缓冲区开始连续往后写，而任意写意味着攻击者可以控制：

- 写到哪里。
- 写什么值。

这就像从“水漫出来碰巧淹到旁边东西”，升级成“我能指定把某个 4 字节值打到某个地址”。DWORD SHOOT 中 DWORD 通常指 4 字节数据，Shoot 强调把数据写到目标地址。

任意读同样重要。现代系统有 ASLR、Canary、DEP 等防护，攻击者常常需要先读出：

- 模块基址，用来计算函数或 gadget 地址。
- 栈地址或堆地址，用来定位可控数据。
- Canary 值，用来绕过栈保护。
- 密钥、Token、Cookie 等敏感数据。

所以很多现代利用链是“先信息泄漏，再控制写入，最后劫持控制流”。

### SEH

SEH 是 Windows 的结构化异常处理机制。SEH 结构在栈上，历史上常被栈溢出攻击利用：攻击者覆盖异常处理链，触发异常后让控制流跳向恶意代码。后续 Lecture07 的 SafeSEH、SEHOP 就是针对这类攻击的防护。

SEH 攻击的关键在于：异常发生时，系统会沿异常处理链寻找处理函数。如果攻击者能通过栈溢出覆盖异常处理记录，再故意触发异常，程序就可能跳到被篡改的异常处理地址。

它和普通返回地址覆盖的区别是：

- 返回地址覆盖依赖函数执行到 `ret`。
- SEH 覆盖依赖触发异常，让系统走异常处理流程。

因此 SEH 是另一种控制流劫持入口。后来 Windows 引入 SafeSEH 和 SEHOP，就是为了验证异常处理函数是否合法、异常链是否完整。

### 易考点

- 漏洞也称 Vulnerability。
- 栈溢出的危险点是覆盖控制数据，尤其返回地址。
- SEH 结构存在于栈区。
- 任意写能力比单纯崩溃危险得多。
- 缓冲区溢出可发生在栈区，也可发生在堆区。
- C/C++ 中 `strcpy`、`strcat` 等函数如果不限制长度，容易造成溢出。
- 溢出代码或 payload 通常以原程序权限运行，不会凭空获得更高权限。
- 整数溢出本身是数值计算错误，不属于“内存覆盖型溢出”，但可诱发缓冲区溢出。
- 栈溢出代码题要注意偏移、变量顺序和字符串结束符。

### 缓冲区溢出的本质

缓冲区是程序为了存放数据预留的一段内存，比如字符数组、网络包缓存、文件读取缓存。溢出发生在“写入的数据超过缓冲区边界，而程序没有正确检查”。这个错误本身可能只是覆盖相邻数据；一旦相邻数据是返回地址、函数指针、对象指针、长度字段、权限标志等关键内容，就可能变成安全漏洞。

理解缓冲区溢出要分三层：

1. 内存破坏：越界写改变了不该改变的内存。
2. 状态破坏：被改的内存影响程序判断、权限、长度或对象结构。
3. 控制流破坏：被改的内存最终影响下一条执行指令的位置。

栈溢出最经典，是因为局部数组、保存的 `EBP`、返回地址经常在同一个栈帧附近。攻击者输入过长字符串时，可能先填满缓冲区，再覆盖保存的 `EBP`，最后覆盖返回地址。函数返回时，`ret` 会把被覆盖的值当作返回地址，程序就跳到攻击者指定位置。

堆溢出的难点在于堆布局不如栈帧固定。攻击者往往要理解分配器如何安排堆块，如何让目标对象与溢出缓冲区相邻，以及覆盖哪个字段最有效。堆攻击可能覆盖对象中的函数指针、虚表指针，也可能破坏堆管理结构，造成更复杂的任意写。

并非所有溢出都能利用。可利用性取决于几个问题：输入是否可控，写入长度是否可控，覆盖内容是否可控，目标内存是否包含有价值的控制数据，程序崩溃前是否能走到利用点，防护机制是否阻断。安全分析时要从“会不会崩”进一步追问“能不能控”。

SEH 攻击是 Windows 历史上的典型控制流劫持方式。异常处理链在栈上，如果溢出能覆盖异常处理记录，攻击者触发异常后，系统会沿异常链寻找处理函数。若处理函数地址已被篡改，就可能跳向攻击者控制的位置。后来的 SafeSEH 和 SEHOP 正是为打断这条链。

任意读和任意写是比普通溢出更抽象的能力。任意读可以泄露地址、密钥、cookie、canary，帮助绕过 ASLR 或窃取信息。任意写可以修改控制数据或安全检查结果。很多复杂漏洞利用会先通过信息泄漏获得地址，再通过写能力改控制流。

### 漏洞分析时的推理流程

1. 找输入入口：网络、文件、命令行、环境变量、表单字段。
2. 找缓冲区大小：固定数组、动态分配长度、结构体字段。
3. 找边界检查：是否检查长度，检查值是否可能整数溢出。
4. 找相邻目标：返回地址、函数指针、对象、长度字段。
5. 找触发点：被覆盖内容什么时候被使用。
6. 找防护：canary、DEP、ASLR、SafeSEH、编译器检查。

可以再加一个考试版口诀：入口、长度、布局、目标、触发、防护。

- 入口：攻击者能不能控制输入。
- 长度：输入能不能超过缓冲区。
- 布局：超过后会写到哪里。
- 目标：那里有没有有价值的变量或控制数据。
- 触发：被覆盖的数据什么时候生效。
- 防护：栈保护、DEP、ASLR 是否阻断。

看到代码题时，不要先想 shellcode，先问最朴素的问题：写了几个字节？目标变量距离缓冲区起点几个字节？最终判断条件用的是什么值？

### 复习自查

1. 为什么“程序崩溃”不等于“漏洞可利用”？
2. 栈溢出为什么常常和返回地址有关？
3. 堆溢出为什么更依赖具体内存布局？
4. 任意读如何帮助绕过 ASLR？
5. SEH 攻击为什么需要触发异常？

### 本讲主线

Lecture05 补充非典型缓冲区溢出的漏洞：格式化字符串、整数溢出、C++ 虚函数攻击，以及注入类、权限类等其他漏洞。

### 格式化字符串漏洞

安全写法：

```c
printf("%s", user_input);
```

危险写法：

```c
printf(user_input);
```

问题本质是“通道混淆”：程序员本来想把用户输入当作普通数据打印，却让它变成了格式控制字符串。

常见格式符：

- `%x`：读取栈上的十六进制数据，可造成信息泄漏。
- `%s`：把某个值当作地址，再从该地址读取字符串。
- `%n`：把已输出字符数写入指定地址，可造成内存写。

因此格式化字符串漏洞既可能读内存，也可能写内存。它甚至可用于泄露地址以绕过 ASLR。

课堂例子中，如果输入导致栈上出现 `0x41414141`，再使用 `%s`，就可能从地址 `0x41414141` 读取字符串。能写入数据的格式化符是 `%n`。

格式化字符串漏洞的关键不是“printf 这个函数危险”，而是“格式串被攻击者控制”。`printf("%s", user_input)` 中，格式串是程序员固定写死的，用户输入只是普通参数；`printf(user_input)` 中，用户输入变成解释规则，`printf` 会把 `%x`、`%s`、`%n` 当指令执行。

它常见的攻击能力分三层：

1. 读栈：输入 `%x %x %x`，让程序把本来不该输出的栈内容打印出来。
2. 读任意地址附近内容：利用 `%s` 把某个值当地址读取字符串。
3. 写内存：利用 `%n` 把已输出字符数写到指定地址。

所以格式化字符串漏洞往往先用于信息泄漏，再用于绕过 ASLR 或配合其他漏洞。它和缓冲区溢出的共同点是都来自错误信任外部输入；区别是缓冲区溢出通常是“写太多”，格式化字符串是“把数据当解释规则”。

### 整数溢出

整数溢出是数值运算超出类型能表示的范围，发生回绕或符号变化。它通常不单独完成攻击，而是绕过长度检查、边界判断或内存分配判断，进而引发缓冲区溢出等漏洞。

典型模式：

- 长度相加后变小。
- 有符号/无符号转换导致判断失效。
- 乘法计算分配大小时溢出，实际分配空间小于后续写入大小。

整数溢出的危险在于：程序员以为自己检查了长度，但检查基于错误的计算结果。例如：

```c
int size = count * 4;
buf = malloc(size);
read(fd, buf, count * 4);
```

如果 `count * 4` 超过 `int` 能表示的范围，`size` 可能回绕成很小的正数。于是 `malloc` 分配很小空间，但后续逻辑仍按很大的数据量写入，最终造成堆溢出。

常见类型：

- 加法溢出：`len + header` 回绕变小。
- 乘法溢出：`count * element_size` 回绕。
- 符号转换：负数转成无符号数后变成很大的数。
- 截断：大整数赋给小整数类型，高位丢失。

考试里如果问“整数溢出是不是内存覆盖型溢出”，一般要区分：整数溢出本身是数值错误，不是直接写越界；但它常常导致后续缓冲区溢出、越界访问或权限判断绕过。

### C++ 虚函数攻击

C++ 对象中通常有虚表指针 vptr，指向虚函数表 vtable。调用虚函数时，程序通过 vptr 找到 vtable，再取出函数地址调用。

如果攻击者能通过溢出修改对象内存中的 vptr 或虚函数指针，就可能让虚函数调用跳到攻击者指定位置。

易考：虚表指针保存在对象的内存空间中。

C++ 虚函数攻击属于“间接控制流劫持”。它不一定覆盖返回地址，而是覆盖对象中的指针。

可以这样理解：

```text
C++ 对象
┌─────────────┐
│ vptr        │ -> vtable -> 虚函数1地址 / 虚函数2地址 / ...
├─────────────┤
│ 普通字段    │
└─────────────┘
```

当程序执行 `obj->virtual_func()` 时，机器层面不是直接调用固定地址，而是先从对象里取 vptr，再从 vtable 里取函数地址，最后间接跳转。如果溢出覆盖了 vptr，让它指向攻击者伪造的虚表，后续虚函数调用就可能跳到攻击者选择的位置。

这类攻击说明：控制流数据不只有返回地址。函数指针、回调指针、虚表指针、异常处理指针都可能成为目标。

### 注入类漏洞

注入类攻击的共同点：外部输入被当作代码、命令或解释器语法的一部分执行。

常见类型：

- SQL 注入：输入被拼接进 SQL 查询。
- 操作系统命令注入：输入被拼接进系统命令。
- 动态代码执行：输入被拼接进可执行代码。
- 文件包含：用户控制包含哪个文件。
- XML/SOAP 注入：XML 是解释性结构，缺乏检查时可破坏消息结构。

SOAP 协议存在注入风险的原因包括 XML 是解释性语言，以及缺乏对 XML 消息恶意输入的检查。

注入类漏洞可以用统一模型理解：

```text
程序原本想构造一段“命令/查询/页面/路径”
攻击者输入被拼进去
解释器无法区分“程序员写的语法”和“用户输入的数据”
最终语义被攻击者改变
```

比如 SQL 注入中，程序本来想查某个用户：

```sql
select * from users where name='输入'
```

如果输入直接拼接，攻击者可以让它变成恒真条件、追加查询或注释掉后续语句。命令注入、模板注入、文件包含也是同一思想：用户输入进入了某种解释器。

防御的核心是让代码和数据分离：

- SQL 用参数化查询。
- 命令执行避免拼接字符串，使用参数数组或白名单。
- HTML 输出按上下文编码。
- XML 解析关闭危险外部实体和不必要特性。
- 文件包含使用固定映射，不让用户直接控制路径。

### 权限类漏洞

权限漏洞是访问控制逻辑出错。常见两类：

- 垂直越权：普通用户获得管理员功能。
- 水平越权：用户 A 访问或修改用户 B 的资源。

水平越权常见原因是服务端过度相信客户端传入的 `userid`、订单号、对象 ID，而没有检查资源归属。

权限漏洞和内存漏洞不同，它不一定导致崩溃，也不一定有明显报错。它的本质是“该不该允许这个用户做这件事”的判断错了。

垂直越权：权限等级向上突破。普通用户访问管理员接口，或者普通员工调用财务审批接口。

水平越权：同一权限等级内访问别人的资源。用户 A 把 `/order?id=1001` 改成 `/order?id=1002`，如果能看到用户 B 的订单，就是水平越权。

常见根因：

- 只在前端隐藏按钮，后端没有鉴权。
- 服务端相信客户端传来的 `userid`、`role`、`is_admin`。
- 只检查“是否登录”，不检查“是否有权访问这个对象”。
- 接口设计缺少统一权限中间件。

防御重点是服务端强制鉴权，尤其是对象级权限检查。每次访问订单、文件、用户资料、审批记录，都要确认资源属于当前用户或当前角色有权访问。

### 易考点

- `printf(str)` 风险大于 `printf("%s", str)`。
- `%n` 可写内存。
- 整数溢出常用于绕过检查。
- vptr 在对象内存中。
- 注入漏洞本质是输入被解释器当作代码/语法执行。
- `%x` 常用于泄露栈数据，`%s` 可能把值当地址读字符串，`%n` 能写内存。
- 整数溢出不是直接内存覆盖，但可导致缓冲区溢出。
- C++ 虚函数调用依赖对象中的 vptr 和虚表 vtable。
- 水平越权是访问同级别其他用户资源，垂直越权是低权限访问高权限功能。
- 注入漏洞的首选防御通常是结构化 API 或参数化，而不是黑名单过滤。

### 格式化字符串漏洞的核心理解

格式化字符串漏洞和缓冲区溢出一样，都来自程序员对外部输入的错误信任。区别在于：缓冲区溢出通常是“写得太多”，格式化字符串漏洞是“把数据当成格式指令”。`printf` 族函数会根据格式串决定后面应该从栈或寄存器中取多少参数、如何解释这些参数。如果用户输入成为格式串，用户就间接控制了解释规则。

例如程序写成：

```c
char str[200];
fgets(str, 200, stdin);
printf(str);
```

输入普通字符串时看不出问题；输入 `%x %x %x` 时，`printf` 会以为调用者还提供了多个参数，于是从当前调用上下文里继续取值并打印，造成栈内容泄露。输入 `%s` 时，取出的值会被当作地址，再从该地址读取字符串，可能造成敏感内存读取或崩溃。输入 `%n` 时，函数会把“已经输出的字符数”写入对应地址，因此可形成内存写。

安全写法是固定格式串：

```c
printf("%s", str);
```

这里用户输入只作为一个普通字符串参数，不再控制格式解析规则。这个例子背后的通用经验是：解释器或格式化函数的“控制通道”必须由程序自己掌握，不能交给用户输入。

### 格式化字符串的攻击能力

格式化字符串漏洞可能产生三类能力：

1. 信息泄漏：通过 `%x`、`%p`、`%s` 等读取栈、地址或字符串。
2. 地址定位：泄露模块地址、栈地址、canary 等，为绕过 ASLR 和栈保护提供条件。
3. 内存写入：通过 `%n` 系列格式符写指定地址，进一步修改返回地址、函数指针或安全检查变量。

很多时候格式化字符串漏洞先用来“看见内存”，再配合其他漏洞完成控制流劫持。它的危险不只在于崩溃，而在于泄露和写入能力。

### 整数溢出的核心理解

整数溢出不是“内存写越界”，而是“计算结果错了”。程序经常先计算长度，再根据长度分配内存或检查边界。如果长度计算发生溢出，后续所有看似合理的判断都会建立在错误结果上。

典型场景：

```c
int size = count * sizeof(item);
buf = malloc(size);
read(fd, buf, count * sizeof(item));
```

如果 `count * sizeof(item)` 超过整数范围，`size` 可能回绕成很小的值，导致 `malloc` 分配很小空间，而后续读取却按原始逻辑写入大量数据，最终引发堆溢出。

常见类型包括：

- 加法溢出：`len + header_size` 回绕。
- 乘法溢出：`count * element_size` 回绕。
- 有符号/无符号转换：负数被当成很大的无符号数。
- 截断：大整数赋给小类型后高位丢失。

整数溢出通常是“前置漏洞”：它绕过检查或造成错误分配，再引出缓冲区溢出、越界访问、权限绕过等后果。

### C++ 虚函数攻击的核心理解

C++ 的动态派发依赖虚表。对象中保存虚表指针 vptr，vptr 指向虚表 vtable，虚表里保存各个虚函数的真实地址。调用虚函数时，程序不是直接跳到固定函数，而是通过对象里的 vptr 间接找到函数地址。

这带来一个安全风险：如果攻击者通过溢出覆盖对象中的 vptr，让它指向攻击者布置的伪造虚表，那么后续虚函数调用就可能跳到攻击者选择的地址。虚函数攻击的关键不是“虚函数本身危险”，而是“间接调用依赖对象内存中的指针，而对象内存可能被破坏”。

理解虚函数攻击要抓住三步：

1. 对象内存被覆盖。
2. vptr 或虚表项被改写。
3. 程序调用虚函数时发生间接跳转。

这和覆盖返回地址属于同一类思想：攻击者不一定直接控制 `ret`，也可以控制函数指针、虚表指针、回调指针等间接控制流数据。

### 注入类漏洞的统一模型

SQL 注入、命令注入、XML/SOAP 注入、文件包含等看起来形式不同，本质相同：程序把外部输入拼进某种“解释器语言”里，解释器无法区分哪些是程序员本来写的语法，哪些是用户恶意输入。

统一模型：

1. 程序接收外部输入。
2. 程序把输入拼接进 SQL、命令、HTML、XML、脚本或文件路径。
3. 后端解释器解析拼接后的内容。
4. 输入改变了原本语义。

防御的共同原则是把“代码”和“数据”分开。SQL 用参数化查询，HTML 用输出编码，命令执行避免拼接字符串，文件包含使用白名单映射，XML 解析关闭危险特性并做结构化校验。

### 权限漏洞的核心理解

权限漏洞不是内存错误，而是业务逻辑和访问控制错误。系统不能只相信“用户请求里写了什么”，必须在服务端确认“这个用户是否有权操作这个对象”。

水平越权例子：用户 A 请求 `/order?id=1002`，把 `id` 改成用户 B 的订单号后也能查看。问题在于服务端只按订单号查询，没有检查订单是否属于当前登录用户。

垂直越权例子：普通用户访问管理员接口成功。问题可能是接口没有鉴权，或只在前端隐藏按钮，后端没有检查角色。

权限漏洞的防御重点：

- 所有敏感操作都在服务端鉴权。
- 对象级权限检查不能省。
- 不信任客户端传来的用户 ID、角色、价格、权限标志。
- 日志记录越权尝试。

2. `%x`、`%s`、`%n` 分别可能造成什么后果？
3. 整数溢出为什么常常不是最终漏洞，而是漏洞触发条件？
4. C++ 虚函数调用为什么依赖对象内存中的 vptr？
5. SQL 注入、命令注入、文件包含的共同点是什么？
6. 水平越权和垂直越权的区别是什么？

# Lecture06 漏洞利用基础 - 复习手册

> 本文件基于课程 PDF 整理，用于考前复习和人工复核。

### 本讲主线

Lecture06 讲 exploit、payload、shellcode 的关系，以及覆盖相邻变量、代码植入、shellcode 编写和编码。它把“漏洞存在”推进到“漏洞如何被利用”。

### 漏洞、利用、破坏

有漏洞不一定有 exploit，但有 exploit 一定对应某种漏洞。

| 概念 | 含义 |
|---|---|
| Vulnerability | 可被利用的弱点 |
| Exploit | 触发漏洞并达成攻击目标的技术方案或动作 |
| Payload | 攻击载荷，完成攻击效果的整体内容 |
| Shellcode | payload 中可能包含的一段机器码，负责具体功能 |

触发漏洞、将控制权转移到目标程序/目标代码的是 exploit。

这几个词可以放到同一条攻击链里理解：程序中先有 Vulnerability；攻击者写出或使用 Exploit 去触发它；触发成功后执行 Payload；在二进制漏洞里，Payload 里常包含 Shellcode。不要把 exploit 和 shellcode 混成一个东西。SQL 注入、反序列化、逻辑越权也可以有 exploit，但它们不一定有传统 shellcode。

### Shellcode

Shellcode 原意与“获得 shell”有关，但现代语境中不一定真的开 shell。任何以注入方式获得非法权限或执行特定攻击功能的机器码，都可称为 shellcode。

漏洞利用的核心是控制流劫持：利用程序漏洞改变程序执行路径，让程序执行攻击者希望执行的代码或代码片段。

Shellcode 之所以难写，是因为它不像普通程序那样拥有完整运行环境。普通程序可以靠导入表、链接器和运行库找到 API；shellcode 常常只是一串被塞进缓冲区的机器码，必须尽量短、位置无关，还要避开会截断字符串或被过滤的坏字符。

### Exploit 的结构

Exploit 不只是 shellcode。它通常还要负责：

- 构造能触发漏洞的输入。
- 覆盖关键控制数据。
- 计算或猜测跳转地址。
- 绕过坏字符、长度限制、ASLR、DEP 等限制。
- 把控制流转移到 payload 或 shellcode。

导弹类比：

- exploit 像攻击方案。
- payload 像有效载荷。
- shellcode 像载荷里真正执行功能的一段机器码。

一个典型早期栈溢出 exploit 输入可以拆成：

```text
[填充数据] [覆盖旧 EBP] [覆盖返回地址] [NOP sled] [shellcode]
```

填充数据用于到达目标偏移；覆盖返回地址用于让 `ret` 跳到可控区域；NOP sled 用来提高跳转容错率；shellcode 才负责真正执行 payload。现代系统启用 DEP/ASLR/Canary 后，这个结构会变复杂，例如把 shellcode 换成 ROP chain，或先通过信息泄漏计算地址。

### 覆盖相邻变量

不是所有利用一开始都要覆盖返回地址。若缓冲区旁边有权限标志、认证结果、函数指针、路径变量等关键数据，溢出也可以通过覆盖相邻变量改变程序逻辑。

理解重点：内存越界写可以先破坏“数据流”，再进一步破坏“控制流”。

覆盖相邻变量的题通常不需要你想 shellcode，而是要你算偏移。例如 `char buffer[8]` 后面紧跟 `int authenticated`，那么写满 8 字节后，下一个字节就可能开始影响 `authenticated`。如果输入函数会自动写入字符串结束符 `\0`，这个结束符也算一次写入，课后题经常借它考“至少多少字节开始覆盖相邻变量”。

### 代码植入

早期典型思路：

1. 把 shellcode 放入缓冲区或可控内存。
2. 覆盖返回地址或控制数据。
3. 让程序跳转到 shellcode。

现代系统中，这条路会遇到 ASLR、NX/DEP、栈保护等防护。

代码植入要同时满足三个条件：攻击者能把代码写进进程内存，能让控制流跳过去，并且那块内存可执行。DEP/NX 正是卡住第三点：栈和堆即使能写入 shellcode，也不能直接执行。于是攻击者会转向 ret2libc、ROP，或者先调用系统 API 修改内存权限。

### Shellcode 编写难点

- 需要汇编和机器码知识。
- 代码长度受限制。
- 不能包含坏字符，例如字符串结束符 `0x00`。
- 地址可能随机化。
- 栈或堆可能不可执行。
- 安全软件可能检测特征。

Shellcode 编码的常见原因：

- 规避字符集限制。
- 绕过坏字符。
- 绕过安全检测。

“绕过返回地址”不是 shellcode 编码的主要原因。

课程中的 `MessageBoxA` 示例常用于说明 shellcode 如何调用系统 API。Windows 32 位常见调用约定下，函数参数通常从右向左压栈。`MessageBoxA(hWnd, lpText, lpCaption, uType)` 的压栈顺序是：

```text
uType -> lpCaption -> lpText -> hWnd
```

如果题目问“按压栈顺序写参数”，不要按函数声明从左到右写。这个点非常适合填空题。

API 地址也是 shellcode 的难点。ASLR 或系统版本差异会让 DLL 加载地址变化，shellcode 不能总是假设函数在固定地址。所谓 API 函数自搜索，就是 shellcode 自己遍历进程中的模块信息，定位 `kernel32.dll` 等模块，再从导出表中找到目标函数地址。

### 易考点

- exploit 负责触发漏洞并转移控制权。
- shellcode 不一定开 shell。
- payload 包含攻击效果，shellcode 可能是 payload 的一部分。
- shellcode 编码常为绕过坏字符、字符集和检测。
- `MessageBoxA(hWnd, lpText, lpCaption, uType)` 的压栈顺序是 `uType`、`lpCaption`、`lpText`、`hWnd`。
- NOP sled 用于提高跳转容错率。
- 传统代码植入需要可控内存、可控跳转和可执行权限。
- DEP/NX 阻止栈/堆 shellcode 直接执行，但不阻止复用已有代码。
- API 函数自搜索用于动态定位系统 DLL 和 API 地址。

### 从漏洞到利用的完整链条

一个漏洞只是“系统中存在可被利用的弱点”，并不自动等于攻击成功。漏洞利用要把弱点转化成可控效果。这个过程通常包含三个问题：

1. 如何触发漏洞？
2. 触发后能控制什么？
3. 控制后如何达成目标？

Exploit 负责回答前两个问题：构造输入、触发错误、覆盖关键数据、改变控制流。Payload 负责回答第三个问题：拿到控制流后要做什么。Shellcode 是 payload 中常见的一段机器码，用来执行具体动作。

可以把漏洞利用理解成一次“程序状态改造”：攻击者先把程序带到一个异常状态，再让这个异常状态变成可控状态，最后让可控状态产生目标效果。只让程序崩溃通常不够；能控制返回地址、函数指针、内存写入位置或解释器语义，才是利用的关键。

### Exploit、Payload、Shellcode 的关系

| 概念 | 关注点 | 例子 |
|---|---|---|
| Exploit | 怎么触发漏洞并转移控制权 | 构造超长输入覆盖返回地址 |
| Payload | 控制权到手后要实现什么效果 | 弹窗、建立会话、读取文件 |
| Shellcode | payload 中的机器码片段 | 调用系统 API 完成特定动作 |

一个 exploit 可以携带不同 payload，同一个 payload 也可能被不同 exploit 使用。Shellcode 不一定真的打开 shell，它只是历史名称。只要是注入并执行的短机器码，都可以被称为 shellcode。

### 覆盖相邻变量

早期学习漏洞利用时容易以为必须覆盖返回地址，其实不一定。程序里很多安全判断依赖普通变量，例如：

- `is_admin` 是否为真。
- `authenticated` 是否为真。
- `price` 或 `quota` 是否被修改。
- 指针是否指向安全对象。

如果溢出能覆盖这些相邻变量，攻击者可能不需要劫持控制流，也能改变程序逻辑。这类利用说明：安全影响不只来自“执行恶意代码”，也可能来自“改变关键数据”。

### 代码植入的基本思想

代码植入型利用的经典思路是：

1. 把 shellcode 放进可控输入或可控内存。
2. 通过漏洞覆盖返回地址、函数指针或其他控制数据。
3. 让程序跳到 shellcode 所在地址。
4. shellcode 调用系统函数完成 payload。

这条路线在早期系统上更直接，因为栈或堆上的数据可能可执行。现代系统启用 DEP/NX 后，栈和堆通常不可执行，直接跳到栈上 shellcode 会被阻止。于是攻击者转向 ROP 等代码复用技术，先修改内存权限或直接复用已有代码。

### Shellcode 编写为什么难

Shellcode 通常要短、小、位置无关，并且避开坏字符。它不像普通程序那样可以依赖完整运行环境和链接器。

常见限制：

- 不能包含 `0x00`，因为字符串函数会把它当作结尾。
- 输入字段长度有限。
- 地址随机化导致绝对地址不可直接使用。
- API 地址需要动态寻找或由 exploit 提供。
- 字符集过滤可能破坏机器码。
- 安全软件可能识别常见字节序列。

因此 shellcode 常写成位置无关代码，尽量通过相对寻址、栈操作和动态解析 API 来工作。课程中的 MessageBox 示例主要是帮助理解“机器码如何被当作数据写入，又如何被转成执行流”。

### Shellcode 编码

编码不是加密课程意义上的保密，而是把 shellcode 变成一种更适合通过输入过滤和检测的形式。运行时再由解码器还原。

编码常用于：

- 消除坏字符。
- 适配字符集限制。
- 绕过简单特征检测。
- 让 payload 能通过某些协议字段或文本字段。

编码后的 payload 通常由两部分组成：解码器 stub 和被编码的主体。程序跳到解码器后，解码器先还原真正 shellcode，再转去执行。

### 现代防护对利用链的影响

| 防护 | 对利用链的影响 |
|---|---|
| Canary | 覆盖返回地址前可能被检测 |
| ASLR | 难以预测 shellcode、库函数、gadget 地址 |
| DEP/NX | 栈/堆上的 shellcode 无法直接执行 |
| SafeSEH/SEHOP | 覆盖异常处理链更难 |
| CFI | 限制间接跳转目标 |

所以现代漏洞利用常常不是单一技巧，而是组合链：信息泄漏绕过 ASLR，内存破坏获得写能力，ROP 绕过 DEP，最后执行 payload。

# Lecture07 漏洞利用防护与漏洞挖掘 - 复习手册

> 本文件基于课程 PDF 整理，用于考前复习和人工复核。

### 本讲主线

Lecture07 合并了两部分：第一部分讲利用防护和绕过，第二部分讲漏洞挖掘方法。它的核心是攻防循环：防护不是修复漏洞本身，而是让漏洞更难利用；挖掘则是用程序分析方法更系统地发现漏洞。

## Part A 漏洞利用防护

### 防护目标

漏洞利用的典型链路是：

1. 覆盖返回地址。
2. 劫持 EIP/RIP。
3. 跳转到 shellcode。
4. 执行恶意代码。

防护技术的目标是打断链路中的某一步。直接修复漏洞当然是第一优先级，但现实中不可能所有漏洞都立刻发现并修复，所以还需要利用缓解技术。

### ASLR

ASLR 是地址空间布局随机化。它随机化关键地址，让攻击者难以预测跳转目标。

随机内容包括：

- 栈起始地址。
- 堆分配地址。
- DLL/共享库加载地址。
- 程序代码加载地址。

ASLR 防：

- ret2libc 地址跳转。
- ROP 链地址定位。
- shellcode 地址跳转。

ASLR 常见突破：

- 信息泄漏。
- 暴力尝试。
- 使用未启用 ASLR 的模块。

CPU 不需要关心 ASLR。ASLR 后由 Loader 修正映射和重定位，CPU 只执行当前进程里的虚拟地址。

ASLR 防的不是漏洞本身，而是防“攻击者知道该跳到哪里”。如果没有 ASLR，攻击者可以在 exploit 里写死某个库函数、gadget 或 shellcode 地址；有 ASLR 后，同一个地址在不同进程或不同运行中可能变化。

常见绕过方式：

- 信息泄漏：先泄露一个栈地址、堆地址或库函数地址，再计算其他地址。
- 未启用 ASLR 的模块：某些 DLL 地址固定，可作为跳板。
- 暴力尝试：在随机化熵较低或服务可反复重启时尝试。
- 部分返回地址覆盖：只覆盖地址低字节，利用高位不变或变化小的特点。

所以 ASLR 经常和信息泄漏联系在一起。没有地址信息，ROP 链难定位；泄露一个模块基址后，就能根据固定偏移找到函数和 gadget。

### GS / Stack Canary

GS Stack Protection 是缓冲区溢出检测防护。编译器在栈中放入 canary 值，函数返回前检查 canary 是否被破坏。如果栈溢出覆盖返回地址，通常会先覆盖 canary，从而被检测出来。

局限：

- 不是所有溢出都会覆盖 canary。
- 信息泄漏可能泄露 canary。
- 某些非返回地址类破坏不一定被拦截。

### DEP / NX

DEP 是数据执行保护，硬件层常对应 NX 位。它把栈、堆等数据区域标记为不可执行，阻止攻击者直接执行写入栈/堆的 shellcode。

易考：能有效阻止非代码区植入恶意代码执行的是 DEP。

DEP 防的是“执行数据区代码”，但不防“复用已有代码”。这就引出 ROP。

DEP 的边界要说清楚：它不能阻止缓冲区溢出发生，也不能阻止攻击者覆盖返回地址。它阻止的是“返回地址跳到栈/堆上的数据并把数据当代码执行”。如果攻击者不执行新写入的数据，而是跳到程序或 DLL 里原本就可执行的代码，DEP 就拦不住。这就是 ret2libc 和 ROP 的基本思路。

### SafeSEH 与 SEHOP

SEH 攻击通过覆盖异常处理链劫持控制流。

- SafeSEH：限制异常处理函数必须在安全表中。
- SEHOP：检测异常处理链完整性，防止伪造异常链。

SafeSEH 依赖模块在编译时生成合法异常处理函数表。异常发生时，系统检查处理函数地址是否在表中。如果攻击者把 SEH 处理函数改成任意地址，就会被拦截。

但如果进程中加载了未启用 SafeSEH 的模块，攻击者可能借用该模块中的地址作为跳板。因此课后题里“利用未开启 SafeSEH 的模块”常被列为绕过 SafeSEH 的思路。

SEHOP 更关注整条异常处理链是否完整，防止攻击者伪造链条。它和 SafeSEH 都是针对 SEH 覆盖攻击的缓解机制。

### ROP

ROP 是 Return-Oriented Programming，返回导向编程。它不注入新代码，而是复用程序或库中已有的短指令片段 gadget，通过一串返回地址把 gadget 串起来执行。

为什么重要：

- DEP 阻止栈/堆 shellcode 执行。
- ROP 复用代码段，代码段本来可执行。
- 因此 ROP 可绕过单纯 DEP。

关键词：

- Gadget：以 `ret` 等控制转移结尾的小代码片段。
- ROP chain：多个 gadget 地址和参数组成的链。
- 跳板指令：如 `jmp esp` 或等价控制转移，可把执行流引到可控位置。

ROP 的核心转变是：攻击者不再把“新代码”放到数据区执行，而是把已有代码片段重新组合。每个 gadget 做一点小事，例如 `pop eax; ret`、`mov [eax], ebx; ret`、`add esp, 8; ret`。攻击者把这些 gadget 地址按顺序放在栈上，每次 `ret` 都跳到下一个 gadget，最终拼出想要的逻辑。

ROP 常用于：

- 调用系统函数。
- 构造函数参数。
- 调用 `VirtualProtect` / `mprotect` 修改内存页权限。
- 绕过 DEP 后再执行 shellcode。

ROP 仍然害怕 ASLR，因为 gadget 地址需要知道。所以现代利用常见组合是：信息泄漏绕过 ASLR，ROP 绕过 DEP。

### 堆喷洒与 API 函数自搜索

堆喷洒 Heap Spray 是一种降低地址精度要求的技术。攻击者在堆中大量布置相似内容，通常是 NOP sled 加 shellcode。这样即使跳转地址不够精确，只要落入大片喷洒区域，也可能滑到 shellcode。

堆喷洒常和浏览器、脚本环境等场景联系在一起，因为脚本可以申请大量对象，把堆布置成攻击者希望的形态。它不是漏洞本身，而是利用技巧。

API 函数自搜索解决的是 shellcode 找 API 地址的问题。由于系统 DLL 地址可能随机化或版本不同，shellcode 不能总写死 API 地址。自搜索技术通常会：

1. 找到进程中已加载模块，例如 `kernel32.dll`。
2. 解析模块的 PE 结构和导出表。
3. 根据函数名或哈希找到目标 API。
4. 调用该 API 完成 payload。

课后题里如果问“绕过 Windows 防护的进阶利用技术”，ROP、堆喷洒、API 函数自搜索、利用未开启 SafeSEH 模块、部分返回地址覆盖都可以作为候选。

## Part B 漏洞挖掘

### 方法分类

| 方法 | 特点 | 优点 | 缺点 |
|---|---|---|---|
| 静态分析 | 不运行程序，分析代码或二进制 | 效率高、覆盖广、召回高 | 误报多、精度低 |
| 动态分析 | 运行程序观察行为 | 结果更真实 | 开销大、覆盖依赖输入 |
| 混合分析 | 结合静态和动态 | 互补 | 系统复杂 |

静态技术包括数据流分析、控制流分析、污点分析、符号执行、程序切片、抽象解释、模式匹配等。动态技术包括模糊测试、动态污点分析、Concolic execution、运行时监控、Sanitizer、DAST/IAST、沙箱执行、动态 exploit 验证等。

静态分析和动态分析不是谁绝对更好，而是取舍不同：

- 静态分析不运行程序，覆盖面大，容易快速扫危险函数和代码路径，但可能误报。
- 动态分析运行程序，观察真实行为，证据更强，但覆盖依赖输入和环境。
- 混合分析把两者结合，例如先静态找可疑点，再用 fuzzing 或调试复现。

课后题常见说法：“静态分析效率高但可能误报；动态分析准确率较高但依赖运行环境和输入覆盖。”这句话很值得记。

### 符号执行

符号执行用符号变量代替具体输入，沿程序路径推导约束条件，再通过约束求解器生成能触发目标路径的具体输入。

三个关键点：

- 变量符号化。
- 程序执行模拟。
- 约束求解。

应用：

- 生成高覆盖率测试用例。
- 判断漏洞监测点是否可达。
- 求出触发越界或特定路径的输入。
- 软件破解中寻找到达目标代码的输入。

局限：路径爆炸。程序路径数量随规模指数增长，约束求解可能不可承受。

易考：模拟程序运行、收集语义信息、探索可达路径、分析隐藏错误的方法是符号执行。

### 污点分析

污点分析把不可信输入标记为 tainted，跟踪它在程序中的传播，检查是否到达危险操作点 sink。

核心要素：

- Source：污点源，例如网络数据、文件内容、用户输入。
- Propagation：传播规则，包括扩散和清除。
- Sink：危险点，例如 SQL 执行、命令执行、内存访问、程序跳转。

适合检测：

- SQL 注入。
- XSS。
- 反序列化漏洞。
- RCE 利用链。
- 其他输入驱动型漏洞。

局限：

- 动态调用解析困难。
- 跨函数、跨库传播复杂。
- 指针和别名问题。
- 不擅长业务逻辑漏洞。

污点分析的关键词是 source、propagation、sink、sanitizer：

- Source：污点源，例如 `$_GET`、网络包、文件内容、命令行参数。
- Propagation：传播，例如赋值、拼接、计算、函数传参。
- Sink：危险点，例如 SQL 执行、命令执行、内存写、跳转目标。
- Sanitizer：清洗或约束，例如参数化查询、严格白名单、上下文编码。

它不只用于缓冲区溢出，也能用于 SQL 注入、XSS、命令注入、路径穿越、反序列化等输入驱动型问题。考试如果说“污点分析只能发现缓冲区溢出”，这是错的。

### 词法分析

词法分析通过文本或字符模式匹配危险函数、API 或简单语句组合。例如根据二进制特征、符号表、调试信息识别危险函数。

优点是简单、快；缺点是语义理解浅，容易漏报和误报。

### 数据流分析

数据流分析关注变量取值和数据沿程序路径的流动。流程通常是：

1. 代码建模，构造抽象语法树或控制流图。
2. 追踪变量变化信息。
3. 根据漏洞分析规则检测缺陷。

分类：

- 过程内分析：只分析一个函数内部。
- 过程间分析：跨函数跟踪数据。
- 流不敏感：不精确考虑语句顺序。
- 流敏感：考虑控制流顺序。
- 上下文不敏感：忽略不同调用点差异。
- 上下文敏感：区分不同调用点。

适合检查内存访问越界、常数传播、指针错误等，但复杂代码中可能误报高。

数据流分析关注“某个值从哪里来、经过哪里、到哪里去”。例如用户输入赋给变量 `x`，`x` 拼接进 SQL，最后传给数据库执行，这就是一条值得关注的数据流。

程序依赖图 PDG，Program Dependence Graph，通常包含两类依赖：

- 数据依赖：一个语句使用了另一个语句定义的数据。
- 控制依赖：一个语句是否执行，取决于某个条件分支。

PDG 常和程序切片一起出现，因为切片需要知道哪些语句和目标变量或目标语句有关。

### 程序切片

程序切片从某个变量或语句出发，只保留与它相关的代码片段。静态切片覆盖所有可能执行，动态切片针对某次具体执行。用途是缩小分析范围，让漏洞分析更聚焦。

切片方向非常容易考：

- 前向切片：从某个变量/语句出发，看它会影响哪些后续语句。
- 后向切片：从某个目标变量/语句出发，看哪些前面语句影响了它。
- 静态切片：不针对某次具体输入，覆盖所有可能路径，范围通常更大。
- 动态切片：针对某次具体执行和输入，只保留实际执行中相关的语句，范围通常更小。

如果要分析 `printf("%d", y)` 中的 `y` 从哪里来，用后向切片；如果要看某个用户输入会影响哪些危险点，用前向切片。

### 模糊测试

模糊测试通过向目标程序输入大量异常、边界或变异数据，观察崩溃、异常、Sanitizer 报告等行为来发现漏洞。

分类：

- 基于生成：按文件格式或协议规范生成测试用例。
- 基于变异：从合法样本出发做随机、边界值、位翻转等变异。
- 黑盒 fuzzing：不关心内部覆盖。
- 覆盖引导 fuzzing：根据代码覆盖反馈调整输入。

模糊测试适合发现崩溃类、内存错误类问题，但对深路径可能效率低。与符号执行、污点分析结合可以提高效果。

Fuzzing 的三个基本部件：

1. 种子输入：一批合法或半合法样本。
2. 变异/生成策略：位翻转、边界值、块插入、按格式生成等。
3. 反馈机制：崩溃、覆盖率、超时、Sanitizer 报告。

覆盖引导 fuzzing 会保留能探索新路径的输入，让测试逐渐深入。它的弱点是遇到复杂校验、魔数、深层协议状态时可能卡住。这时可以借助字典、符号执行、污点分析或手写 harness。

程序插桩 Instrumentation 是 fuzzing 和动态分析常用技术。它通过在程序中插入额外代码来收集运行时信息，例如覆盖率、函数调用、变量变化、内存访问。插桩可以在源码编译阶段做，也可以在二进制或运行时做；它通常会带来一定性能开销。

### 利用防护的整体图景

防护技术不是彼此孤立的。它们分别保护利用链上的不同环节：

1. Canary 保护返回地址附近的栈结构。
2. DEP/NX 阻止数据区直接执行代码。
3. ASLR 让攻击者难以知道应该跳到哪里。
4. SafeSEH/SEHOP 保护 Windows 异常处理链。
5. CFI 等更高级技术限制间接跳转目标。

攻击者如果要完成利用，往往需要绕过多层防护。例如传统栈溢出想执行 shellcode，会先遇到 canary 检测，再遇到 ASLR 地址不确定，再遇到 DEP 阻止栈执行。于是现代 exploit 常转向“信息泄漏 + ROP + 权限/内存属性修改”的组合路线。

### ASLR 的细节理解

ASLR 随机化的是地址布局，不是代码逻辑。程序仍然执行同一套代码，只是模块、堆、栈等位置变化了。攻击者真正困难的是：他想跳转到某个函数、gadget 或 shellcode，但不知道当前进程里它们的真实地址。

ASLR 的效果取决于随机化熵、模块是否全部启用随机化、是否存在信息泄漏。只要程序泄露了一个库函数地址，攻击者就可能根据库内偏移计算其他函数或 gadget 的位置。因此信息泄漏常被称为绕过 ASLR 的前置能力。

常见绕过思路：

- 泄露栈地址或库地址。
- 使用未启用 ASLR 的模块。
- 暴力尝试低熵地址空间。
- 利用部分覆盖，只修改地址低字节。

### DEP 与 ROP 的关系

DEP 解决的是“数据不能当代码执行”。如果攻击者把 shellcode 写入栈，然后让 `EIP/RIP` 跳过去，CPU 会因为该页不可执行而阻止。ROP 的关键转变是：不再执行攻击者写入的新代码，而是执行程序和库中本来就存在的代码片段。

一个 gadget 通常很短，例如完成一次寄存器赋值、一次内存写、一次栈调整，并以 `ret` 结束。攻击者把多个 gadget 的地址排在栈上，每次 `ret` 都跳到下一个 gadget，最终串成一段“用旧代码拼出来的新逻辑”。这就是代码复用攻击。

ROP 可以用来：

- 调用系统函数。
- 修改内存页权限，使 shellcode 所在页可执行。
- 构造函数参数。
- 直接完成 payload 逻辑。

因此 DEP 和 ROP 是一对必须一起理解的概念：DEP 让代码注入变难，ROP 用代码复用绕开“不可执行数据区”。

### Canary 的局限

Canary 能发现一类典型栈溢出，但不是万能。它主要保护返回地址附近连续覆盖的场景。如果攻击者通过格式化字符串直接写返回地址，或者覆盖函数指针、堆对象、相邻变量，不一定会破坏 canary。若存在信息泄漏，攻击者也可能先泄露 canary，再原样写回绕过检查。

复习时要记住：防护技术提高利用成本，不等于修复漏洞。真正的修复仍然是消除越界写、错误输入处理或不安全逻辑。

### 漏洞挖掘方法如何选择

不同分析方法适合不同问题：

| 问题 | 更适合的方法 |
|---|---|
| 想快速找危险 API | 词法分析、模式匹配 |
| 想知道数据是否从输入流向危险点 | 污点分析 |
| 想求出触发某条路径的输入 | 符号执行 |
| 想通过大量异常输入找崩溃 | 模糊测试 |
| 想缩小某个变量相关代码范围 | 程序切片 |
| 想理解变量取值变化 | 数据流分析 |

真实工具通常是混合的。例如 fuzzing 负责快速探索普通路径，覆盖卡住时用符号执行求解深层分支；污点分析指出哪些输入字段最可能影响 sink，再指导 fuzzing 重点变异这些字段；Sanitizer 负责在运行时更早发现越界和 use-after-free。

### 符号执行再理解

符号执行最像“带代数的程序执行”。普通执行给 `x=5`，程序只走一条路径；符号执行让 `x` 保持为符号变量，然后遇到 `if (x > 10)` 时分裂路径：一条带约束 `x > 10`，另一条带约束 `x <= 10`。如果目标是触发数组越界，分析器会把路径约束和越界条件一起交给求解器。如果有解，就能得到具体输入。

它的强项是深路径和精确条件，弱点是路径爆炸、复杂库函数建模困难、约束求解成本高。动态符号执行或 concolic execution 用真实执行推进状态，同时收集符号约束，是对纯符号执行的折中。

### 污点分析再理解

污点分析关心“数据有没有被污染，以及污染有没有流到危险点”。比如 Web 程序里 `$_GET['id']` 是 source，SQL 执行函数是 sink。如果 `id` 未经过有效清洗就拼进 SQL，污点到达 sink，就报告 SQL 注入风险。

污点分析的关键不是只标记 source 和 sink，还要定义传播和清洗规则。普通赋值会传播污点，字符串拼接会传播污点，严格的参数化查询可能阻断污点，安全编码可能清洗污点。规则写得太宽会误报，写得太窄会漏报。

### 模糊测试再理解

模糊测试的价值在于“让程序自己暴露异常”。它不需要完全理解程序语义，而是不断输入变异数据，观察崩溃、超时、断言失败、Sanitizer 报告。覆盖引导 fuzzing 会优先保留能探索新路径的输入，让测试逐渐深入程序。

Fuzzing 的三个关键部件：

1. 种子输入：初始合法样本。
2. 变异策略：随机翻转、边界值、块插入、结构生成等。
3. 反馈机制：崩溃、覆盖率、Sanitizer 报告。

它不擅长需要复杂校验、深层协议状态或精确魔数的路径，因此常和符号执行、字典、协议语法、污点分析结合。

# Lecture08 Web 安全 - 复习手册

> 本文件基于课程 PDF 整理，用于考前复习和人工复核。

### 本讲主线

Lecture08 原 PDF 分成四个 part，这里合并为一讲：先讲 Web 基础，再讲 Web 环境和会话，再讲文件上传、XSS、SQL 注入、文件包含、反序列化。Web 安全的核心是“用户输入进入解释器或状态管理流程后，被错误信任”。

## Part A Web 基础

### HTTP、HTML、JavaScript

HTTP 是请求/响应模型。客户端发送请求，请求头包含请求方法、URL、协议版本、Header 等，服务端返回响应状态、响应头和响应体。

HTML 是超文本标记语言，页面可包含文本、图片、链接、脚本等。基本结构包括 `<html>`、`<head>`、`<body>`。

JavaScript 是浏览器中的脚本语言，能操作页面、处理事件、发起请求。它既是正常交互能力，也是 XSS 等攻击得以执行的基础。

易考：统一资源标示符是 URL。

HTTP 的几个安全要点：

- HTTP 本身是无状态协议。服务器不会天然记得“上一个请求是谁发的”，所以需要 Cookie/Session/Token 维护登录态。
- HTTP 默认不加密。真正保护传输机密性和完整性的是 HTTPS/TLS。
- GET 参数在 URL 中，容易出现在浏览器历史、服务器日志、Referer 中。
- POST 数据在请求体中，但 POST 不等于加密，也不等于不能伪造。
- Header、Cookie、URL、Body 都可能携带攻击者输入，不能只检查表单字段。

HTML 决定页面结构，JavaScript 决定页面行为。XSS 的本质就是攻击者让自己的 JavaScript 进入别人的页面执行。浏览器不会自动知道“这段脚本是用户评论里的恶意输入”，只要页面把它当脚本解析，它就会执行。

### PHP 与数据库

PHP 常用于服务端动态页面。课程中用 PHP + MySQL 展示：

- 获取用户输入。
- 拼接 SQL 查询。
- 连接数据库。
- 执行 SQL。
- 读取结果。
- 释放资源并关闭连接。

危险模式是把用户输入直接拼入 SQL 或页面输出。

一个典型 Web 请求链路：

```text
浏览器提交请求
-> Web 服务器接收
-> PHP 读取 GET/POST/Cookie/Header
-> 拼接或构造 SQL
-> 数据库执行
-> PHP 把结果拼进 HTML
-> 浏览器解析 HTML/JS
```

安全问题经常发生在“拼接”这个动作上：

- 用户输入拼进 SQL，可能 SQL 注入。
- 用户输入拼进 HTML/JS，可能 XSS。
- 用户输入拼进文件路径，可能文件包含或路径穿越。
- 用户输入传给 `unserialize()`，可能反序列化漏洞。

所以 Web 安全的核心不是某个函数名，而是边界：用户输入什么时候从“数据”变成了“代码、路径、对象或身份”。

### Cookie 与 Session

HTTP 本身无状态。服务器需要通过 Cookie、Session 等机制维护登录状态。

- Cookie 存在客户端，可设置 expires 生命周期。
- Session 数据通常在服务端，客户端通过 SessionID 标识会话。
- 攻击者窃取 SessionID 后用其登录目标账户，属于会话劫持。

GET 与 POST：

- GET 常用于获取/查询资源，参数出现在 URL 中。
- POST 常用于提交或更新资源，数据在请求体中。
- POST 不等于绝对安全，只是比 GET 更不容易直接暴露在 URL 中。
- GET 请求更容易诱发 CSRF，因为构造链接即可触发。

Cookie 和 Session 配合方式：

1. 用户首次登录后，服务器创建一份 Session 数据，保存在服务端。
2. 服务器生成一个 SessionID，通过 `Set-Cookie` 发给浏览器。
3. 浏览器后续请求会自动携带 Cookie。
4. 服务器根据 SessionID 找到对应 Session，从而识别用户身份。

Cookie 在客户端，Session 数据通常在服务端。攻击者如果窃取 SessionID，就可能冒充用户访问系统，这叫会话劫持。

Cookie 常见安全属性：

- `HttpOnly`：降低被 JavaScript 读取 Cookie 的风险。
- `Secure`：只通过 HTTPS 发送。
- `SameSite`：限制跨站请求携带 Cookie，降低 CSRF 风险。
- `Expires/Max-Age`：控制生命周期。

注意：HttpOnly 不能阻止 XSS 脚本“以当前用户身份发请求”，它只是让脚本不容易直接读 Cookie。

### Web 威胁特点

Web 攻击门槛相对低，攻击入口多，逻辑漏洞远多于传统底层漏洞。传统漏洞和 Web 漏洞可以形成攻击链：例如 SQL 注入作为入口，进一步获取系统权限。

OWASP 是 Open Web Application Security Project，开放式 Web 应用程序安全项目。它发布的 OWASP Top 10 常被用来概括常见 Web 风险。课程和课后题里可能提到 SQL 注入、XSS、文件包含、会话劫持等，但注意不同年份 Top 10 名称和分类会变化，考试一般按课件给出的版本判断。

CSRF，Cross-Site Request Forgery，跨站请求伪造，常被拿来和 XSS 混。CSRF 的重点不是注入脚本，而是诱导已登录用户浏览器向目标站点发出请求。因为浏览器会自动带上 Cookie，目标站点可能误以为是用户本人操作。防御通常用 CSRF Token、SameSite Cookie、重要操作二次确认等。

## Part B 文件上传漏洞

文件上传漏洞是攻击者上传可执行文件到服务器并执行。上传内容可以是恶意脚本、WebShell、木马等。

典型链路：

1. 发现上传点。
2. 上传恶意文件。
3. 文件被放入 Web 可访问目录。
4. 通过浏览器访问并执行。

WebShell 是以 ASP、PHP、JSP、CGI 等网页文件形式存在的命令执行环境，也可看作网页后门。

产生原因：

- 缺少上传文件信息的输入检测。
- 只检查后缀，不检查真实内容。
- 保存路径可被访问且脚本可执行。
- 服务端配置允许上传目录执行脚本。
- 文件名、MIME、魔数、大小、内容未综合校验。

安全上传应考虑：

- 白名单后缀。
- 检查 MIME 和文件魔数。
- 重命名文件。
- 上传目录禁止脚本执行。
- 存储到 Web 根目录之外。
- 限制大小。
- 服务端重新编码图片。
- 权限隔离。

文件上传真正危险的点不是“服务器收到了一个文件”，而是上传后的文件被解释执行，或能参与后续攻击链。

常见绕过思路包括：

- 后缀伪装：例如服务端只看 `.jpg`，但内容里混入脚本。
- MIME 伪造：客户端声明的 `Content-Type` 不可信。
- 双后缀或解析差异：不同服务器对 `a.php.jpg`、大小写、特殊后缀解析不同。
- 图片马：图片内容中夹带脚本片段，再配合文件包含执行。
- 上传目录可执行：上传路径位于 Web 根目录下，且服务器允许脚本执行。

防御要分三层：

- 内容层：检查魔数、真实类型，图片重新解码再编码。
- 路径层：随机重命名，禁止用户控制路径，存储到 Web 根目录外。
- 执行层：上传目录禁脚本执行，最小权限，静态资源隔离。

## Part C XSS

XSS 是 Cross-Site Scripting，跨站脚本攻击。攻击者把恶意 JavaScript 放入页面，使其在其他用户浏览器中执行。

XSS 与 SQL 注入区别：

- XSS 主要影响 Web 应用用户。
- SQL 注入主要影响 Web 应用自身和后端数据库。

常见类型：

| 类型 | 特点 |
|---|---|
| 反射式 XSS | 恶意脚本来自 URL 参数等请求内容，非持久 |
| 存储式 XSS | 恶意脚本存入数据库或后台，用户访问页面时触发 |
| DOM 型 XSS | 前端脚本处理 DOM 时引入恶意内容 |

危害：

- 窃取 Cookie/SessionID。
- 冒充用户操作。
- 钓鱼。
- 篡改页面。
- 结合 CSRF 或后端漏洞形成攻击链。

防御：

- 输出编码。
- 输入校验。
- Cookie 设置 HttpOnly、SameSite、Secure。
- CSP。
- 避免危险 DOM API。

XSS 的本质是输出位置不安全。输入进入数据库或请求参数后，最终被输出到页面。如果输出时没有按上下文编码，浏览器就可能把用户输入当 HTML 或 JavaScript 执行。

三类 XSS 的区别：

- 反射式 XSS：恶意输入来自当前请求，例如 URL 参数，服务器原样反射到页面。
- 存储式 XSS：恶意输入被保存到数据库，其他用户访问页面时触发。
- DOM 型 XSS：漏洞主要在前端 JavaScript，前端把 `location.hash`、URL 参数等直接写入 DOM。

输出编码必须看上下文：

- HTML 文本节点：转义 `<`、`>`、`&`。
- HTML 属性：还要处理引号。
- JavaScript 字符串：按 JS 字符串规则转义。
- URL：做 URL 编码，并限制协议如 `javascript:`。

CSP 可以降低 XSS 成功率，但不能替代正确编码。HttpOnly 可以降低 Cookie 被偷的风险，但不能阻止脚本发起转账、改密码等当前用户有权发起的请求。

## Part D SQL 注入

SQL 注入是把 SQL 代码插入或添加到应用输入参数中，再传给后端数据库解析执行。

根本原因：把用户输入直接拼接进 SQL 语句，并交给数据库解释器执行。

常见注入点：

- 登录页面。
- 查询页面。
- 添加/修改页面。
- URL 参数。
- Cookie、Header 等服务端使用的数据。

检测思路：

- 单引号法：通过破坏 SQL 语法观察报错。
- 布尔逻辑法：比较 `and 1=1` 与 `and 1=2` 的页面差异。
- 时间盲注：通过延迟函数判断条件真假。

课堂易考：

- 单引号法测试注入点的原理是检查输入是否破坏 SQL 语法。
- `sqlmap -u url --tables -D "testdb"` 的作用是列举数据库 `testdb` 的所有表。
- 时间盲注中，如果判断数据库名长度为 4 的条件导致明显延迟，则说明当前数据库名长度为 4。

防御：

- 参数化查询/预编译语句。
- 输入校验。
- 最小权限数据库账号。
- 错误信息不直接回显。
- ORM 或安全查询构造器。
- WAF 只能作为补充，不能替代代码修复。

SQL 注入的核心是数据库最终看到的 SQL 语义被用户输入改变。比如程序想要：

```sql
select * from user where name='alice' and pwd='123'
```

如果 `name` 和 `pwd` 直接拼接，攻击者输入 `' or 1=1 --`，就可能让条件恒真并注释掉后续语句。`#` 在 MySQL 中也常用于注释后面的 SQL 内容，课后题喜欢问它的作用。

常见类型：

- 报错注入：通过数据库错误信息泄露结构或数据。
- 联合查询注入：用 `union select` 拼接额外查询结果。
- 布尔盲注：页面只显示真假差异，通过条件逐步猜数据。
- 时间盲注：页面无明显差异，通过延迟函数判断条件真假。
- 堆叠查询：一次提交多条 SQL，取决于数据库和驱动是否支持。

课后题里把盲注分为“基于布尔、基于时间、基于报错”时，按题目口径记即可。更严格地说，报错注入常不算盲注，但考试跟课件走。

首选防御是参数化查询/预编译语句。以 PDO 为例，安全思想是 SQL 结构固定，用户输入作为参数绑定：

```php
$stmt = $pdo->prepare("select * from user where name = ? and pwd = ?");
$stmt->execute([$user, $pwd]);
```

这样数据库不会把用户输入当 SQL 语法解析。输入过滤可以辅助，但黑名单很容易被编码、注释、大小写、数据库差异绕过。

## Part E 文件包含漏洞

文件包含是开发者把重复代码放入单独文件，再在需要处包含。问题在于：如果“包含哪个文件”由用户控制，攻击者就能控制服务器读取或执行的内容。

本地文件包含 LFI：

- 被包含文件在服务器本地。
- 可能读取系统文件、日志文件、上传文件等。
- 可结合上传漏洞、日志污染等形成代码执行。

远程文件包含 RFI：

- 被包含文件可通过 URL 获取。
- 需要配置允许，例如 PHP 中相关 URL include 开关。
- 攻击者可能包含自己服务器上的恶意文件。

PHP 伪协议：

- `php://filter` 可用于读取并编码源码。
- `php://input` 可读取 POST body。
- `file://` 可访问本地文件。
- `zip://` 等可访问压缩包内文件。

文件包含防御：

- 不让用户直接控制文件路径。
- 使用固定映射表。
- 路径规范化并限制目录。
- 禁止远程包含。
- 上传目录不可执行。
- 最小权限。

文件包含漏洞的核心不是 include/require 本身危险，而是“include 的路径被用户控制”。

LFI，Local File Inclusion，本地文件包含：

- 包含服务器本地文件。
- 可能读取 `/etc/passwd`、配置文件、源码、日志。
- 如果日志或上传文件中含有脚本片段，包含后可能执行。

RFI，Remote File Inclusion，远程文件包含：

- 包含外部 URL 上的文件。
- 在 PHP 中通常要求 `allow_url_fopen=On` 且 `allow_url_include=On`。
- 攻击者可让目标包含自己服务器上的恶意脚本。

PHP 伪协议常见作用：

- `php://filter`：读取源码并可做 base64 编码，避免 PHP 直接执行。
- `php://input`：读取 POST body。
- `file://`：访问本地文件。
- `zip://`：访问压缩包内文件。

防御不要只过滤 `../` 或几个关键字。更稳妥的是取消用户对文件路径的直接控制，改用固定映射，例如 `page=home` 只映射到预设的 `home.php`。

## Part F 反序列化漏洞

序列化是把对象或数据结构转成字符串；反序列化是把字符串恢复为对象或数据结构。

反序列化漏洞的关键条件：

- 程序对用户可控数据执行反序列化。
- 目标类已加载。
- 类中存在魔术方法或危险调用链。
- 攻击者能构造满足触发条件的序列化字符串。

PHP 中常见相关魔术方法包括：

- `__wakeup()`：反序列化时触发。
- `__destruct()`：对象销毁时触发。
- `__toString()`：对象当字符串使用时触发。

利用往往需要源码分析，找到 POP chain，即一串可被对象属性控制并最终触发危险操作的方法调用链。

防御：

- 不反序列化不可信数据。
- 使用 JSON 等简单数据格式替代对象反序列化。
- 限制可反序列化类。
- 对数据做完整性签名。
- 避免魔术方法中执行敏感操作。

反序列化漏洞容易被低估，因为入口看起来只是“恢复对象”。真正危险在于：恢复对象时程序可能自动调用魔术方法，而对象属性又可能由攻击者控制。

PHP 中常见魔术方法：

- `__wakeup()`：反序列化时触发。
- `__destruct()`：对象销毁时触发。
- `__toString()`：对象被当字符串使用时触发。

利用通常需要 POP Chain，Property-Oriented Programming，也就是属性导向编程链。攻击者不一定能直接调用危险函数，但可以构造对象属性，让多个类的方法在自动触发过程中一步步传递，最终走到文件删除、命令执行、SQL 查询、模板渲染等危险操作。

核心条件：

1. 用户可控数据进入 `unserialize()`。
2. 相关类在运行时已加载。
3. 存在可被魔术方法或调用链触发的危险操作。
4. 攻击者能控制对象属性影响危险函数参数。

### Web 安全的统一理解

Web 漏洞大多不是因为 HTTP、HTML、SQL 本身邪恶，而是因为应用把边界混淆了：把用户输入当成代码，把客户端状态当成可信事实，把文件名当成安全路径，把序列化字符串当成可信对象。复习 Web 安全时，可以统一问四个问题：

1. 这个数据从哪里来？
2. 这个数据进入了哪个解释器或状态管理机制？
3. 服务端有没有校验、编码、鉴权或隔离？
4. 攻击者能否改变语义、身份、路径或执行内容？

HTTP 是无状态协议，所以登录态需要 Cookie 和 Session 维持。安全问题也常围绕状态展开：SessionID 被窃取会造成会话劫持，Cookie 缺少 HttpOnly 会被 XSS 更容易读取，缺少 SameSite 会增加 CSRF 风险，服务端过度相信客户端参数会造成越权。

### GET、POST 与安全误区

GET 和 POST 是请求语义差异，不是安全等级差异。GET 参数出现在 URL 中，容易进入浏览器历史、日志和 Referer；POST 数据放在请求体里，不等于加密，也不等于不能伪造。真正的保密依赖 HTTPS，真正的授权依赖服务端鉴权。

GET 常用于查询资源，POST 常用于提交或修改资源。若一个修改密码、删除数据、转账等敏感操作用 GET 触发，就更容易被链接、图片标签或跨站请求诱导触发。即使用 POST，如果没有 CSRF token 或 SameSite 等保护，也仍可能被跨站请求利用。

### 文件上传漏洞再理解

文件上传漏洞的危险不只是“上传了坏文件”，而是“上传后的文件被服务器解释执行”或“上传文件能参与后续攻击链”。攻击者上传 PHP/JSP/ASP 脚本，如果服务器把上传目录当作脚本目录执行，就形成 WebShell。即使不能直接执行，也可能通过文件包含、解析差异、图片马、压缩包路径穿越等方式触发。

安全上传要同时检查三个面：

- 内容面：文件魔数、真实类型、重新编码、病毒扫描。
- 路径面：文件名重命名、路径规范化、防止 `../`、存储到 Web 根目录外。
- 执行面：上传目录禁脚本执行，最小权限，静态资源域名隔离。

只检查后缀不可靠，因为后缀可伪造；只检查 MIME 也不可靠，因为 MIME 来自客户端或可被伪造。更安全的做法是服务端重新生成文件，例如图片重新解码再编码。

### XSS 再理解

XSS 的本质是输出位置不安全。用户输入被放回页面时，如果没有根据上下文编码，浏览器会把它当作 HTML 或 JavaScript 执行。

不同上下文需要不同编码：

- HTML 文本节点：转义 `<`、`>`、`&` 等。
- HTML 属性：还要处理引号。
- JavaScript 字符串：按 JS 字符串规则转义。
- URL 参数：做 URL 编码，并限制协议。

反射式 XSS 常见于搜索结果、错误提示、跳转参数；存储式 XSS 常见于留言、评论、个人资料、后台配置；DOM 型 XSS 常见于前端脚本把 `location.hash`、URL 参数等直接写入 DOM。

XSS 的危害取决于用户身份和页面能力。攻击管理员后台的 XSS 可能比普通用户页面严重得多，因为脚本能以管理员身份发请求。HttpOnly 可以降低 Cookie 被读取的风险，但不能阻止脚本冒用当前页面身份发送请求。

### SQL 注入再理解

SQL 注入的关键是数据库看到的最终 SQL 语义被用户改变。比如程序本来想查：

```sql
select * from users where username='bob' and password='abc123'
```

如果输入被直接拼接，攻击者可让条件变成恒真、追加查询、注释掉后续条件，或利用报错、布尔差异、时间延迟推断数据。

常见注入类型：

- 报错注入：利用数据库错误回显信息。
- 联合查询注入：用 `union select` 拼接额外结果。
- 布尔盲注：根据页面真假差异推断。
- 时间盲注：根据响应延迟推断。
- 堆叠查询：一次提交多条语句，取决于数据库和驱动支持。

防御首选参数化查询，因为它让 SQL 结构和参数数据分离。输入过滤可以作为辅助，但不能替代参数化，因为黑名单容易绕过，编码、注释、大小写、函数和数据库差异都会造成漏网。

### 文件包含再理解

文件包含漏洞发生在“包含哪个文件”由用户控制。危害有两类：读取敏感文件，或执行可控内容。

本地文件包含可读取 `/etc/passwd`、配置文件、源码、日志等。如果攻击者能让恶意内容进入日志，再包含日志文件，日志中的代码可能被解释执行。若应用允许上传文件，攻击者可先上传看似合法但含脚本片段的文件，再通过包含漏洞执行。

远程文件包含要求服务器允许包含 URL 资源。其风险更直接：攻击者把恶意脚本放在自己的服务器，目标应用包含后执行。现代环境通常关闭相关配置，但复习时要理解配置项和路径控制的关系。

PHP 伪协议常用于扩展攻击面。`php://filter` 可把源码 base64 编码后返回，帮助读取源码；`php://input` 可读取请求体；`file://` 访问本地文件。防御时不能只过滤几个关键词，而要从设计上取消用户控制 include 路径。

### 反序列化再理解

反序列化漏洞特别容易被低估，因为入口看起来只是“恢复对象”。真正危险在于恢复对象时，程序可能自动调用魔术方法；对象属性又可能被攻击者控制；多个类的方法调用能连成 POP 链，最终走到文件删除、命令执行、SQL 查询、模板渲染等危险操作。

利用条件通常包括：

1. 反序列化数据可控。
2. 相关类在运行时可用。
3. 存在可利用的魔术方法或调用链。
4. 对象属性能影响危险函数参数。

防御反序列化最好的方式是不对不可信数据使用原生对象反序列化。如果必须使用，应限制类型、签名校验、最小化魔术方法副作用，并避免在析构、唤醒、字符串转换等自动触发方法中做敏感操作。

### Web 漏洞对比表

| 漏洞 | 根因 | 典型后果 | 首选防御 |
|---|---|---|---|
| 文件上传 | 上传内容可执行或可被利用 | WebShell、代码执行 | 白名单、重编码、上传目录不可执行 |
| XSS | 输出未按上下文编码 | 会话劫持、冒用身份操作 | 输出编码、CSP、Cookie 安全属性 |
| SQL 注入 | 输入拼接进 SQL | 数据泄露、认证绕过 | 参数化查询 |
| 文件包含 | 用户控制包含路径 | 读文件、代码执行 | 固定映射、禁远程包含、路径限制 |
| 反序列化 | 不可信数据恢复对象 | POP 链、代码执行 | 不反序列化不可信数据、签名和类型限制 |

### 复习自查

1. 为什么 POST 不等于安全？
2. 文件上传漏洞为什么常和 WebShell 联系在一起？
3. XSS 防御为什么要区分 HTML、属性、JS、URL 上下文？
4. 参数化查询为什么比黑名单过滤更可靠？
5. 本地文件包含如何和日志或上传功能形成攻击链？
6. 反序列化漏洞为什么经常需要分析源码？

# Lecture09 渗透测试基础及实践 - 复习手册

> 本文件基于课程 PDF 整理，用于考前复习和人工复核。

### 本讲主线

Lecture09 是渗透测试流程和工具基础。它把前面学到的漏洞和利用技术放入授权测试流程中，包括 PTES、Kali、Metasploit、信息收集、主动扫描和实验环境实践。

### PTES 七阶段

PTES 是 Penetration Testing Execution Standard，将渗透测试标准化为 7 个阶段：

1. 前期交互。
2. 情报搜集。
3. 威胁建模。
4. 漏洞分析。
5. 渗透攻击。
6. 后渗透攻击。
7. 报告。

PTES 的意义是把“会不会攻击”变成“能不能规范地评估风险”。一次渗透测试不是从打开 Metasploit 开始，而是从授权、范围、目标和风险控制开始。七阶段可以理解为：

```text
先谈清楚边界
-> 收集目标信息
-> 判断可能攻击面
-> 分析并验证漏洞
-> 在授权范围内利用
-> 评估拿到入口后的影响
-> 写成防御者能修的报告
```

真实测试中这七步不是绝对线性。比如漏洞利用后发现内网新资产，可能回到情报搜集；扫描结果不可信，可能回到漏洞分析；后渗透发现权限不足，可能重新寻找提权路径。

### 前期交互

目标是确定测试范围、目标、限制条件和合同细节。最重要的是授权边界：哪些系统能测、哪些不能测、测试时间、影响控制、应急联系人、报告形式等。

前期交互决定这次测试是否合法、可控、可交付。至少要确认：

- 目标范围：域名、IP、系统、应用、账号、内网/外网边界。
- 测试时间：是否避开业务高峰。
- 允许手段：是否允许漏洞利用、弱口令尝试、社工、DoS、后渗透。
- 禁止事项：是否禁止数据导出、破坏性测试、持久化。
- 应急联系人：测试导致告警或故障时联系谁。
- 交付要求：报告格式、风险评级、复测方式。

没有这些约定，技术动作再像渗透测试，也可能变成未授权攻击。

### 情报搜集

信息收集是渗透测试中最关键的一环。收集越充分，后续成功率越高。

| 类型 | 是否直接接触目标 | 例子 |
|---|---|---|
| 被动信息收集 | 不直接接触目标系统 | 搜索引擎、whois、DNS、社交媒体、公开泄露信息 |
| 主动信息收集 | 会与目标系统交互 | 主机发现、端口扫描、服务识别、指纹探测 |

社会工程学攻击的是人的弱点，例如伪装邮件、诱导回复、遗留 U 盘等。课程中提到分析邮件头可获得内部邮件服务器信息。

易考：影响渗透测试成功与否的最关键步骤是信息收集。

信息收集关键，是因为漏洞依附在具体资产上。你不知道目标有什么域名、IP、端口、服务、版本、后台、人员和供应商，就很难选择攻击路径。

被动信息收集的特点是尽量不直接接触目标系统，隐蔽性好，但信息可能过时。主动信息收集会产生网络交互，结果更准确，但可能触发日志或告警。

常见信息之间的关系：

- 域名和子域名告诉你入口在哪里。
- DNS 记录可能暴露邮件、内网命名、云服务。
- Whois 和备案可能暴露组织和联系人。
- 开放端口告诉你运行了哪些服务。
- 服务版本决定是否能匹配公开漏洞。
- 页面指纹和 CMS 版本决定 Web 攻击面。
- 泄露的邮箱和员工信息可能影响弱口令、钓鱼和社工风险。

### 威胁建模

在收集情报后，团队分析：

- 哪些资产最关键。
- 哪些入口最可行。
- 哪些攻击路径风险最大。
- 哪些漏洞最值得验证。

它常被忽视，但能决定测试方向。

威胁建模不是画漂亮图，而是排序：在有限时间里，哪些资产最值得测，哪些路径最可能成功，哪些风险最能说明业务影响。

例如你发现：

- 一个公网后台。
- 一个旧版 CMS。
- 一个文件上传点。
- 一批员工邮箱。

威胁建模会把它们连成路径：旧版 CMS 是否有漏洞？后台是否弱口令？上传点是否能执行脚本？邮箱是否能用于重置密码？这比孤立地列“发现 10 个端口”更有价值。

### 漏洞分析

漏洞分析阶段综合前面获得的信息，包括扫描结果、服务查点信息、公开漏洞和利用代码，判断哪些攻击点可实施，并在实验环境中验证。

高水平团队还会针对关键系统做漏洞探测和挖掘，寻找未知漏洞。

漏洞分析要区分“扫描器提示”和“漏洞成立”。扫描器可能因为版本识别、规则过时、配置差异产生误报。一个严谨结论通常要确认：

- 受影响产品和版本是否匹配。
- 目标配置是否满足漏洞条件。
- 是否已有补丁或缓解。
- 是否能用低影响方式验证。
- 验证是否会影响业务。

公开漏洞常通过 CVE、厂商公告、Exploit-DB、NVD、CNVD 等渠道确认。实际利用前要考虑授权边界，不能因为有 exploit 就直接打。

### 渗透攻击与后渗透

渗透攻击阶段利用漏洞获得访问控制权。后渗透阶段关注进入系统后的进一步行为，例如：

- 获取敏感信息。
- 权限提升。
- 横向移动。
- 跳板攻击。
- 内网拓展。
- 评估业务影响。

报告阶段要站在防御者角度，总结漏洞、攻击路径、影响、修复建议和安全体系薄弱点。

渗透攻击阶段的目标通常不是“破坏系统”，而是证明风险存在。例如证明 SQL 注入能读取敏感表，证明文件上传能执行受控命令，证明弱口令能登录后台。验证要尽量最小化影响。

后渗透回答的是“如果攻击者已经进来了，还能做什么”。它可能包括：

- 当前权限是什么。
- 能否提权。
- 能否访问敏感文件或数据库。
- 能否横向移动到其他主机。
- 能否获得凭证或 Token。
- 监控是否发现了这次行为。

后渗透必须严格受授权限制。比如客户不允许导出数据，就只能证明可访问，不应复制真实敏感数据。

### Kali Linux

Kali 是专门用于渗透测试的 Linux 发行版，由 BackTrack 发展而来，基于 Debian。常见基础命令包括文件管理、系统信息、进程管理、网络配置、软件包管理等。

易考：Kali 属于 Debian 类型的 Linux 系统。

Kali 的定位是工具箱，不是自动攻击系统。它集成了扫描、枚举、Web 测试、密码测试、漏洞利用、取证等工具。学习 Kali 的重点不是背每个命令，而是知道每类任务需要什么工具：

- 主机发现和端口扫描：找在线资产和开放服务。
- 服务识别：判断服务类型和版本。
- Web 目录扫描：找隐藏路径、备份文件、管理入口。
- 漏洞扫描：发现已知风险。
- 抓包和代理：分析请求响应。
- 密码测试：验证弱口令风险。

命令输出也不能机械相信，要结合上下文判断。例如端口开放不等于有漏洞，服务版本旧也不等于一定可利用。

### Metasploit 与 Meterpreter

Metasploit 是开源渗透测试框架，也是漏洞研究和渗透代码开发平台，支持情报搜集、漏洞分析、渗透攻击、后渗透和报告。

Meterpreter 是 Metasploit 中常用于后渗透阶段的 payload/控制通道，具备信息获取、系统监控、权限提升、跳板和扩展模块等能力。

常见概念：

- exploit：利用模块。
- payload：攻击载荷，通过网络传送到目标并执行。
- auxiliary：辅助模块，如扫描、Fuzz、DoS 测试等。
- target：受影响目标系统类型。
- options：模块参数。

易考：通过网络传送到攻击目标进行执行的针对特定平台的攻击代码是 payloads。

Metasploit 把渗透测试中的许多动作模块化：

- exploit：触发目标漏洞的模块。
- payload：漏洞触发后在目标上执行的载荷。
- auxiliary：辅助模块，例如扫描、枚举、fuzzing、DoS 验证。
- post：后渗透模块。
- encoder：对 payload 做编码，绕过坏字符或简单检测。
- target：特定系统版本、语言、补丁级别或架构。
- options：模块参数，例如 RHOSTS、RPORT、LHOST、LPORT。

使用逻辑通常是：搜索模块，查看说明，设置目标参数，选择 payload，检查配置，在授权环境中运行。课程中的 MS08-067 示例重点是流程：发现服务、识别漏洞、选择模块、设置 payload、获得会话、做后渗透演示。

Meterpreter 是一种功能丰富的 payload/会话，不只是“一个 shell”。它可以加载扩展、收集系统信息、上传下载文件、端口转发、截屏、权限提升等。真实测试中要谨慎使用，因为后渗透动作更容易触碰授权边界。

### 被动信息收集

被动信息收集不直接与目标服务器交互，通常包括：

- 搜索引擎查询。
- Google Hacking。
- Whois 信息。
- DNS 信息。
- 组织架构与人员信息。
- 邮件头分析。
- 公开泄露数据。

它的优点是不容易被目标记录；缺点是信息可能过时或不完整。

### 主动信息收集

主动信息收集会与目标系统交互，可能被日志记录。

常见任务：

- 主机发现：识别哪些主机在线。
- 端口扫描：识别开放端口。
- 服务识别：识别端口背后的服务和版本。
- 操作系统指纹探测：判断系统类型和版本。
- Web 指纹探测：判断 Web 容器、CMS、框架版本。
- 目录扫描：寻找敏感路径、备份文件、上传目录等。

易考：对目标主机系统版本、服务版本进行检测叫指纹探测。

主动信息收集常见顺序：

1. 主机发现：哪些 IP 在线。
2. 端口扫描：哪些端口开放。
3. 服务识别：端口背后是什么服务。
4. 版本识别：服务版本、Web 容器、CMS、框架。
5. 目录和接口发现：隐藏路径、备份文件、上传点、管理后台。
6. 漏洞初筛：根据版本、配置和响应特征匹配风险。

主动扫描要控制速率和范围。过高频率可能触发 IDS/IPS，甚至影响业务。授权测试里，温和、可解释、可复现比“扫得猛”更重要。

### 实践案例的理解

课程中的 XP/MS08-067 实践用于演示完整流程：

1. 扫描目标。
2. 用漏洞扫描器发现可能漏洞。
3. 在 Metasploit 中查找对应模块。
4. 设置目标、参数和 payload。
5. 在授权实验环境中验证利用。
6. 获得 Meterpreter 会话后进行后渗透演示。

复习时重点理解流程和角色分工，不需要把课堂截图中的每条命令背成攻击手册。

MS08-067 这种实验的学习价值在于把前面概念串起来：

- 信息收集发现目标是 Windows XP，开放 SMB 服务。
- 漏洞分析确认该版本可能受 MS08-067 影响。
- Metasploit 中 exploit 模块负责触发漏洞。
- payload 负责在目标上建立会话。
- Meterpreter 展示后渗透能力。

考试不会要求你把每条命令背成攻击流程，但会问 exploit、payload、target、options 分别是什么，或者问信息收集、漏洞分析、渗透攻击在流程里分别解决什么问题。

### 渗透测试不是工具演示

渗透测试的目标是帮助系统所有者理解真实风险，而不是简单跑一遍扫描器。一次合格的渗透测试必须有授权、范围、过程记录、风险控制和报告输出。工具只是执行手段，真正重要的是测试者能否把信息收集、漏洞分析、利用验证、业务影响和修复建议串成闭环。

PTES 七阶段可以理解成一条工作流：

1. 前期交互：明确能测什么、不能测什么、什么时候测、出问题联系谁。
2. 情报搜集：尽可能了解目标资产、人员、域名、服务、技术栈。
3. 威胁建模：从攻击者视角选择最可能、最有价值的攻击路径。
4. 漏洞分析：把情报转成可验证的安全问题。
5. 渗透攻击：在授权范围内验证漏洞能否造成实际影响。
6. 后渗透：评估取得立足点后还能造成什么业务后果。
7. 报告：把发现、证据、影响、修复建议交付给防御者。

这七步不是永远线性。实际测试中，发现新信息后可能回到情报搜集，利用失败后可能重新做漏洞分析，后渗透时也可能发现新的攻击面。

### 信息收集为什么最关键

信息收集决定攻击面。漏洞不是凭空存在的，它依附在具体资产、服务、版本、配置和人员流程上。收集越充分，后续越容易找到真实薄弱点。

被动信息收集的优势是隐蔽，不直接接触目标系统。常见来源包括搜索引擎、备案和 whois、DNS 记录、公开代码仓库、社交媒体、招聘信息、历史泄露、邮件头等。它适合在不触发目标日志的情况下建立初始画像。

主动信息收集会直接与目标交互，因此可能留下日志。常见任务包括主机发现、端口扫描、服务识别、操作系统识别、Web 指纹、目录扫描。主动扫描必须严格遵守授权范围，因为扫描本身可能造成告警或影响服务。

被动和主动不是二选一。被动阶段告诉你“可能有什么”，主动阶段验证“实际开着什么”。例如公开信息显示目标使用某 CMS，主动指纹探测再确认版本；DNS 发现子域名，端口扫描再确认开放服务。

### 威胁建模如何思考

威胁建模是在有限时间内选择优先级。不是所有发现都值得深入利用。要考虑：

- 资产价值：是否涉及核心业务、敏感数据、管理员入口。
- 暴露程度：是否公网可达，是否需要内网条件。
- 利用难度：是否有公开 exploit，是否需要复杂前置条件。
- 影响范围：能否导致数据泄露、权限提升、横向移动。
- 检测与风险：验证是否可能影响业务。

好的威胁建模会把“漏洞列表”变成“攻击路径”。例如：子域名暴露后台，后台存在弱口令，登录后有文件上传，上传目录可执行，最终形成 WebShell。这比单独报告每个小问题更能体现业务风险。

### Kali 的定位

Kali 是集成大量安全测试工具的 Debian 系发行版。它不是“会自动黑掉目标”的系统，而是给授权测试者提供工具箱。复习时不必死背每个 Linux 命令，但要熟悉基本能力：

- 文件与目录操作：查看、复制、移动、删除。
- 进程管理：查看和结束进程。
- 网络配置：查看 IP、接口、路由。
- 软件包管理：`apt`、`dpkg`。
- 帮助系统：`man`、`--help`。

工具环境只是基础。真正的能力是知道什么时候用什么工具、结果如何解释、误报如何确认。

### Metasploit 的定位

Metasploit 把漏洞利用过程模块化。它的基本组件包括：

- exploit：触发漏洞的模块。
- payload：漏洞触发后执行的载荷。
- auxiliary：辅助模块，如扫描、枚举、fuzzing。
- target：目标系统类型和版本。
- options：模块参数。

Metasploit 的使用逻辑是：选择模块，查看信息，设置参数，选择目标和 payload，在授权环境中验证。课程中的 MS08-067 示例重点是展示这种流程，而不是让人机械背命令。尤其要注意目标版本，因为很多 exploit 依赖特定系统版本、语言、补丁级别和地址布局。

Meterpreter 是一个功能丰富的后渗透 payload。它的意义在于说明“获得初始控制权”不是测试结束，后渗透阶段还要评估攻击者能否获取敏感信息、扩大权限、横向移动、建立跳板。实际测试中这些操作必须在授权范围内，并控制影响。

### 漏洞扫描与漏洞验证

扫描器能帮助发现潜在漏洞，但扫描结果不等于真实漏洞。原因包括：

- 版本识别可能错误。
- 服务可能打了补丁但版本号未变。
- 漏洞需要特定配置才可触发。
- 扫描器规则可能过时。
- 某些漏洞验证可能有破坏性。

因此渗透测试报告不能只贴扫描器输出。需要人工确认影响、复现条件、证据和修复建议。对于高风险漏洞，通常应在测试环境或低影响方式下验证。

### 报告应该写什么

报告是渗透测试交付的核心。它应该让管理者看懂风险，让技术人员能修复问题。

一条漏洞报告通常包括：

- 漏洞名称和风险等级。
- 影响资产和范围。
- 发现方式。
- 复现证据。
- 业务影响。
- 根本原因。
- 修复建议。
- 参考资料。

好的报告不只是“存在 SQL 注入”，而是说明哪个参数、为什么可注入、能访问什么数据、如何修、修完怎么验证。

报告通常要有两种视角：

- 管理视角：风险等级、业务影响、优先级、整改成本。
- 技术视角：复现步骤、证据、根因、修复代码或配置建议。

例如同样是“后台弱口令”，差报告只写“存在弱口令”；好报告会写清楚影响资产、测试账号、登录入口、可访问功能、可能造成的数据影响、密码策略建议、多因素认证建议、复测方法。

报告阶段重要，是因为渗透测试最终目的不是证明测试者厉害，而是帮助被测方降低风险。没有清晰报告，前面的利用成功对防御改进帮助有限。

### 法律和边界

渗透测试必须有明确授权。授权至少应覆盖目标范围、测试时间、允许的技术手段、禁止事项、数据处理要求和应急联系人。超出授权范围的扫描、利用、后渗透、数据导出，都可能从测试变成违法攻击。

课程里的实验环境，如 XP/MS08-067、DVWA、OWASP Broken Web Applications VM，都是为了在可控环境中理解原理。复习时要把“实验验证能力”和“现实授权边界”同时记住。
