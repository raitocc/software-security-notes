# Lecture08 Web 安全 - 复习手册

> 本文件基于课程 PDF 整理，用于考前复习和人工复核。

### 本讲主线

Lecture08 原 PDF 分成四个 part，这里合并为一讲：先讲 Web 基础，再讲 Web 环境和会话，再讲文件上传、XSS、SQL 注入、文件包含、反序列化。Web 安全的核心是“用户输入进入解释器或状态管理流程后，被错误信任”。

## Part A Web 基础

### HTTP、HTML、JavaScript

HTTP 是请求/响应模型。客户端发送请求，请求头包含请求方法、URL、协议版本、Header 等，服务端返回响应状态、响应头和响应体。

HTML 是超文本标记语言，页面可包含文本、图片、链接、脚本等。基本结构包括 `<html>`、`<head>`、`<body>`。

JavaScript 是浏览器中的脚本语言，能操作页面、处理事件、发起请求。它既是正常交互能力，也是 XSS 等攻击得以执行的基础。

易考：统一资源标示符是 URL。

HTTP 的几个安全要点：

- HTTP 本身是无状态协议。服务器不会天然记得“上一个请求是谁发的”，所以需要 Cookie/Session/Token 维护登录态。
- HTTP 默认不加密。真正保护传输机密性和完整性的是 HTTPS/TLS。
- GET 参数在 URL 中，容易出现在浏览器历史、服务器日志、Referer 中。
- POST 数据在请求体中，但 POST 不等于加密，也不等于不能伪造。
- Header、Cookie、URL、Body 都可能携带攻击者输入，不能只检查表单字段。

HTML 决定页面结构，JavaScript 决定页面行为。XSS 的本质就是攻击者让自己的 JavaScript 进入别人的页面执行。浏览器不会自动知道“这段脚本是用户评论里的恶意输入”，只要页面把它当脚本解析，它就会执行。

### PHP 与数据库

PHP 常用于服务端动态页面。课程中用 PHP + MySQL 展示：

- 获取用户输入。
- 拼接 SQL 查询。
- 连接数据库。
- 执行 SQL。
- 读取结果。
- 释放资源并关闭连接。

危险模式是把用户输入直接拼入 SQL 或页面输出。

PHP 基础语法在本讲里不是重点，但几个细节会影响后面的漏洞理解：

- PHP 代码通常写在 `<?php ... ?>` 中，课件也提到短标签、脚本标签等写法；实际开发中以标准标签最常见。
- PHP 语句一般以分号结束，注释可以用 `//`、`#` 或 `/* ... */`。
- 变量以 `$` 开头，双引号字符串中会解析变量。例如：

```php
<?php
$username = "liuzheli";
$SQLStr = "$username=123";
echo $SQLStr;
?>456
```

输出结果相当于 `liuzheli=123456`。这里要看懂两层含义：第一，PHP 会把双引号字符串里的 `$username` 替换成变量值；第二，PHP 标签外的 `456` 会被当作普通页面内容输出。后面讲 SQL 注入时，`"$username"` 这类字符串拼接就是风险来源之一。

一个典型 Web 请求链路：

```text
浏览器提交请求
-> Web 服务器接收
-> PHP 读取 GET/POST/Cookie/Header
-> 拼接或构造 SQL
-> 数据库执行
-> PHP 把结果拼进 HTML
-> 浏览器解析 HTML/JS
```

安全问题经常发生在“拼接”这个动作上：

- 用户输入拼进 SQL，可能 SQL 注入。
- 用户输入拼进 HTML/JS，可能 XSS。
- 用户输入拼进文件路径，可能文件包含或路径穿越。
- 用户输入传给 `unserialize()`，可能反序列化漏洞。

所以 Web 安全的核心不是某个函数名，而是边界：用户输入什么时候从“数据”变成了“代码、路径、对象或身份”。

### Cookie 与 Session

HTTP 本身无状态。服务器需要通过 Cookie、Session 等机制维护登录状态。

- Cookie 存在客户端，可设置 expires 生命周期。
- Session 数据通常在服务端，客户端通过 SessionID 标识会话。
- 攻击者窃取 SessionID 后用其登录目标账户，属于会话劫持。

Cookie 是服务器写给浏览器、之后由浏览器自动带回服务器的小段文本。一个 Cookie 通常包含名称、值、过期时间、路径、域等字段，其中 path 和 domain 决定它在哪些 URL 范围内生效。PHP 中常见设置形式是：

```php
setcookie(name, value, expire, path, domain, secure)
```

其中 `secure` 表示通过 HTTPS 连接传输；`expire` 没设置时通常是会话 Cookie，浏览器会话结束后消失；设置了过期时间就是持久 Cookie，可能保存在客户端直到过期。课件示例中 `time()+3600*24*30` 表示约 30 天有效期。

Session 是服务端机制。服务端保存会话数据，客户端只保存或传递 SessionID。SessionID 的传递通常依赖 Cookie；如果浏览器禁用 Cookie，也可能通过 URL 重写或隐藏表单字段传递，但这样更容易暴露在地址栏、日志或页面源码中。

GET 与 POST：

- GET 常用于获取/查询资源，参数出现在 URL 中。
- POST 常用于提交或更新资源，数据在请求体中。
- POST 不等于绝对安全，只是比 GET 更不容易直接暴露在 URL 中。
- GET 请求更容易诱发 CSRF，因为构造链接即可触发。

Cookie 和 Session 配合方式：

1. 用户首次登录后，服务器创建一份 Session 数据，保存在服务端。
2. 服务器生成一个 SessionID，通过 `Set-Cookie` 发给浏览器。
3. 浏览器后续请求会自动携带 Cookie。
4. 服务器根据 SessionID 找到对应 Session，从而识别用户身份。

Cookie 在客户端，Session 数据通常在服务端。攻击者如果窃取 SessionID，就可能冒充用户访问系统，这叫会话劫持。

围绕会话还有几个课件点要能区分：

- 会话劫持：攻击者拿到有效 SessionID 后，用它冒充目标账户。如果 SessionID 保存在 Cookie 中并被窃取，也常称为 Cookie 劫持。
- 会话保持攻击：攻击者拿到 Session 后周期性访问或刷新，使服务器认为会话仍活跃，从而延长可用时间。
- 会话固定：攻击者让受害者使用攻击者预先知道的 SessionID，受害者登录后攻击者再复用该会话。
- 传输层保护不足：HTTP 明文传输时，用户名、密码或 SessionID 可能被嗅探；HTTPS/TLS 的价值就在于保护传输机密性和完整性。

Cookie 常见安全属性：

- `HttpOnly`：降低被 JavaScript 读取 Cookie 的风险。
- `Secure`：只通过 HTTPS 发送。
- `SameSite`：限制跨站请求携带 Cookie，降低 CSRF 风险。
- `Expires/Max-Age`：控制生命周期。

注意：HttpOnly 不能阻止 XSS 脚本“以当前用户身份发请求”，它只是让脚本不容易直接读 Cookie。

### Web 威胁特点

Web 攻击门槛相对低，攻击入口多，逻辑漏洞远多于传统底层漏洞。传统漏洞和 Web 漏洞可以形成攻击链：例如 SQL 注入作为入口，进一步获取系统权限。

OWASP 是 Open Web Application Security Project，开放式 Web 应用程序安全项目。它发布的 OWASP Top 10 常被用来概括常见 Web 风险。课程和课后题里可能提到 SQL 注入、XSS、文件包含、会话劫持等，但注意不同年份 Top 10 名称和分类会变化，考试一般按课件给出的版本判断。

CSRF，Cross-Site Request Forgery，跨站请求伪造，常被拿来和 XSS 混。CSRF 的重点不是注入脚本，而是诱导已登录用户浏览器向目标站点发出请求。因为浏览器会自动带上 Cookie，目标站点可能误以为是用户本人操作。防御通常用 CSRF Token、SameSite Cookie、重要操作二次确认等。

课件引用的 OWASP 2010 Top 10 中还出现了几类容易混淆的风险：

- 遭破坏的身份认证和会话管理：重点是认证凭据、SessionID、登录态生命周期管理不当。
- 不安全的直接对象引用，IDOR，Insecure Direct Object Reference：服务端把数据库编号、文件名、账户号等对象引用直接暴露给用户，又没有做对象级授权检查。典型例子是用户把 URL 中的 `acct=1001` 改成 `acct=1002` 后看到别人的账户信息。
- 不安全的加密存储：密码、密钥、敏感数据存储时保护不足。
- 传输层保护不足：登录、会话、敏感操作未通过安全信道传输。
- 未验证的重定向和转发：跳转目标受用户控制，可能被用于钓鱼或绕过访问控制。

## Part B 文件上传漏洞

文件上传漏洞是攻击者上传可执行文件到服务器并执行。上传内容可以是恶意脚本、WebShell、木马等。

典型链路：

1. 发现上传点。
2. 上传恶意文件。
3. 文件被放入 Web 可访问目录。
4. 通过浏览器访问并执行。

WebShell 是以 ASP、PHP、JSP、CGI 等网页文件形式存在的命令执行环境，也可看作网页后门。

课件中的一句话木马可以这样理解：

```php
<?php eval($_POST['uname']); ?>
```

这段代码从 POST 参数 `uname` 中取内容，再交给 `eval()` 当 PHP 代码执行。它本身很短，但一旦被上传到可执行目录，攻击者就能通过 HTTP 请求发送命令。WebShell 的价值不在于文件长得复杂，而在于它把“访问网页”变成了“让服务器执行攻击者给出的代码”。

PHP 上传文件时，`$_FILES` 数组里常见字段包括：

| 字段 | 含义 |
|---|---|
| `name` | 上传文件在客户端的原文件名 |
| `type` | 浏览器声明的 MIME 类型 |
| `size` | 文件大小 |
| `tmp_name` | 服务端临时文件路径 |
| `error` | 上传错误码 |

风险代码通常是直接取 `$upfile["name"]`，再用 `move_uploaded_file($tmp_name, "up/".$name)` 移到 Web 可访问目录。如果没有校验真实类型、随机重命名、限制执行权限，攻击者就可能上传脚本并访问执行。

产生原因：

- 缺少上传文件信息的输入检测。
- 只检查后缀，不检查真实内容。
- 保存路径可被访问且脚本可执行。
- 服务端配置允许上传目录执行脚本。
- 文件名、MIME、魔数、大小、内容未综合校验。

安全上传应考虑：

- 白名单后缀。
- 检查 MIME 和文件魔数。
- 重命名文件。
- 上传目录禁止脚本执行。
- 存储到 Web 根目录之外。
- 限制大小。
- 服务端重新编码图片。
- 权限隔离。

文件上传真正危险的点不是“服务器收到了一个文件”，而是上传后的文件被解释执行，或能参与后续攻击链。

常见绕过思路包括：

- 后缀伪装：例如服务端只看 `.jpg`，但内容里混入脚本。
- MIME 伪造：客户端声明的 `Content-Type` 不可信。
- 双后缀或解析差异：不同服务器对 `a.php.jpg`、大小写、特殊后缀解析不同。
- 图片马：图片内容中夹带脚本片段，再配合文件包含执行。
- 上传目录可执行：上传路径位于 Web 根目录下，且服务器允许脚本执行。

课件把上传文件分成几类风险：病毒或木马可能被其他用户下载运行；WebShell 可形成服务器端命令执行入口；其他恶意脚本可能影响 Web 页面或业务逻辑；恶意图片、SVG、图片元数据或伪装图片可能结合浏览器解析、文件包含等漏洞触发。特别要记住：PHP 的 `include` 关注内容是否含 PHP 代码，不按文件后缀判断执行性，所以“上传一个看似合法的图片，再被本地文件包含执行”是典型攻击链。

防御要分三层：

- 内容层：检查魔数、真实类型，图片重新解码再编码。
- 路径层：随机重命名，禁止用户控制路径，存储到 Web 根目录外。
- 执行层：上传目录禁脚本执行，最小权限，静态资源隔离。

## Part C XSS

XSS 是 Cross-Site Scripting，跨站脚本攻击。攻击者把恶意 JavaScript 放入页面，使其在其他用户浏览器中执行。

XSS 与 SQL 注入区别：

- XSS 主要影响 Web 应用用户。
- SQL 注入主要影响 Web 应用自身和后端数据库。

常见类型：

| 类型 | 特点 |
|---|---|
| 反射式 XSS | 恶意脚本来自 URL 参数等请求内容，非持久 |
| 存储式 XSS | 恶意脚本存入数据库或后台，用户访问页面时触发 |
| DOM 型 XSS | 前端脚本处理 DOM 时引入恶意内容 |

危害：

- 窃取 Cookie/SessionID。
- 冒充用户操作。
- 钓鱼。
- 篡改页面。
- 结合 CSRF 或后端漏洞形成攻击链。

防御：

- 输出编码。
- 输入校验。
- Cookie 设置 HttpOnly、SameSite、Secure。
- CSP。
- 避免危险 DOM API。

XSS 的本质是输出位置不安全。输入进入数据库或请求参数后，最终被输出到页面。如果输出时没有按上下文编码，浏览器就可能把用户输入当 HTML 或 JavaScript 执行。

三类 XSS 的区别：

- 反射式 XSS：恶意输入来自当前请求，例如 URL 参数，服务器原样反射到页面。
- 存储式 XSS：恶意输入被保存到数据库，其他用户访问页面时触发。
- DOM 型 XSS：漏洞主要在前端 JavaScript，前端把 `location.hash`、URL 参数等直接写入 DOM。

输出编码必须看上下文：

- HTML 文本节点：转义 `<`、`>`、`&`。
- HTML 属性：还要处理引号。
- JavaScript 字符串：按 JS 字符串规则转义。
- URL：做 URL 编码，并限制协议如 `javascript:`。

CSP 可以降低 XSS 成功率，但不能替代正确编码。HttpOnly 可以降低 Cookie 被偷的风险，但不能阻止脚本发起转账、改密码等当前用户有权发起的请求。

课件中的 `htmlspecialchars()` 是理解 XSS 防御的好例子：

```php
echo 'Hello ' . htmlspecialchars($_GET['name']);
```

如果用户输入包含 `<script>`，函数会把关键字符转成实体，浏览器更倾向于把它显示成文本，而不是当脚本执行。这里的核心不是“某个函数万能”，而是把用户输入留在“数据”身份里，不让它进入 HTML 或 JavaScript 语法结构。

存储式 XSS 通常比反射式 XSS 风险更高，原因有三点：脚本保存在服务器端，访问正常业务 URL 即可触发；受害者不一定需要点击含攻击代码的异常链接；一些基于 URL 的浏览器过滤或插件更难发现跨页面触发的脚本。现实危害也不只是弹窗，还可能包括窃取 Cookie、键盘记录、页面跳转、伪造表单、篡改页面内容、插入广告或恶意链接。

过滤绕过要从原理上理解：黑名单如果只删除某个固定字符串，攻击者可能通过大小写、事件属性、标签拆分、编码、属性上下文等方式改变表现形式。因此考试中看到“仅过滤 `<script>` 是否足够”时，重点答“不足够，必须按输出上下文编码并配合 CSP、Cookie 安全属性等措施”。

## Part D SQL 注入

SQL 注入是把 SQL 代码插入或添加到应用输入参数中，再传给后端数据库解析执行。

根本原因：把用户输入直接拼接进 SQL 语句，并交给数据库解释器执行。

常见注入点：

- 登录页面。
- 查询页面。
- 添加/修改页面。
- URL 参数。
- Cookie、Header 等服务端使用的数据。

检测思路：

- 单引号法：通过破坏 SQL 语法观察报错。
- 布尔逻辑法：比较 `and 1=1` 与 `and 1=2` 的页面差异。
- 时间盲注：通过延迟函数判断条件真假。

课堂易考：

- 单引号法测试注入点的原理是检查输入是否破坏 SQL 语法。
- `sqlmap -u url --tables -D "testdb"` 的作用是列举数据库 `testdb` 的所有表。
- 时间盲注中，如果判断数据库名长度为 4 的条件导致明显延迟，则说明当前数据库名长度为 4。

SQLMap 是课程中用于演示 SQL 注入自动化利用的工具。常见命令含义如下：

| 命令 | 含义 |
|---|---|
| `sqlmap -u url --users` | 枚举数据库用户 |
| `sqlmap -u url --dbs` | 枚举数据库 |
| `sqlmap -u url --current-db` | 查看当前数据库 |
| `sqlmap -u url --current-user` | 查看当前数据库用户 |
| `sqlmap -u url --tables -D "testdb"` | 枚举 `testdb` 中的表 |
| `sqlmap -u url --columns -T "user" -D "testdb"` | 枚举指定表字段 |
| `sqlmap -u url --dump -C "username,password" -T "user" -D "testdb"` | 导出指定列数据 |

防御：

- 参数化查询/预编译语句。
- 输入校验。
- 最小权限数据库账号。
- 错误信息不直接回显。
- ORM 或安全查询构造器。
- WAF 只能作为补充，不能替代代码修复。

SQL 注入的核心是数据库最终看到的 SQL 语义被用户输入改变。比如程序想要：

```sql
select * from user where name='alice' and pwd='123'
```

如果 `name` 和 `pwd` 直接拼接，攻击者输入 `' or 1=1 --`，就可能让条件恒真并注释掉后续语句。`#` 在 MySQL 中也常用于注释后面的 SQL 内容，课后题喜欢问它的作用。

常见类型：

- 报错注入：通过数据库错误信息泄露结构或数据。
- 联合查询注入：用 `union select` 拼接额外查询结果。
- 布尔盲注：页面只显示真假差异，通过条件逐步猜数据。
- 时间盲注：页面无明显差异，通过延迟函数判断条件真假。
- 堆叠查询：一次提交多条 SQL，取决于数据库和驱动是否支持。

课后题里把盲注分为“基于布尔、基于时间、基于报错”时，按题目口径记即可。更严格地说，报错注入常不算盲注，但考试跟课件走。

布尔盲注的典型比较是：`1' and 1=1 #` 与 `1' and 1=2 #`。前者条件为真，后者条件为假，如果页面响应不同，说明输入影响了 SQL 条件。这里单引号用于闭合原来的字符串，`#` 在 MySQL 中用于注释后续语句。

时间盲注的典型形式是：`1' and if(length(database())=4,sleep(5),3) #`。如果响应明显延迟，说明条件成立，即当前数据库名长度为 4。它不依赖页面直接回显数据库内容，而是把“真假”转换成“快慢”来观察。

首选防御是参数化查询/预编译语句。以 PDO 为例，安全思想是 SQL 结构固定，用户输入作为参数绑定：

```php
$stmt = $pdo->prepare("select * from user where name = ? and pwd = ?");
$stmt->execute([$user, $pwd]);
```

这样数据库不会把用户输入当 SQL 语法解析。输入过滤可以辅助，但黑名单很容易被编码、注释、大小写、数据库差异绕过。

## Part E 文件包含漏洞

文件包含是开发者把重复代码放入单独文件，再在需要处包含。问题在于：如果“包含哪个文件”由用户控制，攻击者就能控制服务器读取或执行的内容。

正常包含和文件包含漏洞要分清。开发者固定包含公共数据库连接文件、页头页脚模板，这是正常代码复用；如果参数 `file`、`page`、`func` 等来自用户并直接进入 `include "$file"`，才会形成“用户决定包含谁”的危险边界。

本地文件包含 LFI：

- 被包含文件在服务器本地。
- 可能读取系统文件、日志文件、上传文件等。
- 可结合上传漏洞、日志污染等形成代码执行。

远程文件包含 RFI：

- 被包含文件可通过 URL 获取。
- 需要配置允许，例如 PHP 中相关 URL include 开关。
- 攻击者可能包含自己服务器上的恶意文件。

PHP 伪协议：

- `php://filter` 可用于读取并编码源码。
- `php://input` 可读取 POST body。
- `file://` 可访问本地文件。
- `zip://` 等可访问压缩包内文件。

文件包含防御：

- 不让用户直接控制文件路径。
- 使用固定映射表。
- 路径规范化并限制目录。
- 禁止远程包含。
- 上传目录不可执行。
- 最小权限。

文件包含漏洞的核心不是 include/require 本身危险，而是“include 的路径被用户控制”。

LFI，Local File Inclusion，本地文件包含：

- 包含服务器本地文件。
- 可能读取 `/etc/passwd`、配置文件、源码、日志。
- 如果日志或上传文件中含有脚本片段，包含后可能执行。

本地文件包含常见两条链：

1. 包含上传文件：攻击者先上传含 PHP 片段的“合法文件”，再通过包含漏洞把该文件纳入 PHP 解释流程。即使文件名后缀是 `.jpg`，只要内容中有 PHP 代码并被 `include` 处理，就可能执行。
2. 包含日志文件：攻击者先发起一条带恶意代码的请求，使其进入 Web 服务器访问日志，再通过 LFI 包含日志文件。课件中提到 Apache 常见日志路径如 `/var/log/apache2/access.log`。

RFI，Remote File Inclusion，远程文件包含：

- 包含外部 URL 上的文件。
- 在 PHP 中通常要求 `allow_url_fopen=On` 且 `allow_url_include=On`。
- 攻击者可让目标包含自己服务器上的恶意脚本。

PHP 伪协议常见作用：

- `php://filter`：读取源码并可做 base64 编码，避免 PHP 直接执行。
- `php://input`：读取 POST body。
- `file://`：访问本地文件。
- `phar://`、`zip://`：访问 PHP 归档或压缩包内文件。`zip://hack.jpg%23shell.php` 中 `%23` 对应 `#`，用于定位压缩包内成员。

注意 `php://input` 读取的是原始 POST 请求体，本身不是“读某个本地文件内容”的协议；在远程包含相关配置开启时，它可把请求体中的代码交给 `include` 处理。`php://filter` 则常用于读取本地源码并做 base64 编码输出。

防御不要只过滤 `../` 或几个关键字。更稳妥的是取消用户对文件路径的直接控制，改用固定映射，例如 `page=home` 只映射到预设的 `home.php`。

## Part F 反序列化漏洞

序列化是把对象或数据结构转成字符串；反序列化是把字符串恢复为对象或数据结构。

反序列化漏洞的关键条件：

- 程序对用户可控数据执行反序列化。
- 目标类已加载。
- 类中存在魔术方法或危险调用链。
- 攻击者能构造满足触发条件的序列化字符串。

PHP 中常见相关魔术方法包括：

- `__wakeup()`：反序列化时触发。
- `__destruct()`：对象销毁时触发。
- `__toString()`：对象当字符串使用时触发。

利用往往需要源码分析，找到 POP chain，即一串可被对象属性控制并最终触发危险操作的方法调用链。

防御：

- 不反序列化不可信数据。
- 使用 JSON 等简单数据格式替代对象反序列化。
- 限制可反序列化类。
- 对数据做完整性签名。
- 避免魔术方法中执行敏感操作。

反序列化漏洞容易被低估，因为入口看起来只是“恢复对象”。真正危险在于：恢复对象时程序可能自动调用魔术方法，而对象属性又可能由攻击者控制。

PHP 中常见魔术方法：

- `__sleep()`：序列化时触发。
- `__wakeup()`：反序列化时触发。
- `__destruct()`：对象销毁时触发。
- `__toString()`：对象被当字符串使用时触发。

PHP 反序列化对象还有一个前提：类定义在反序列化前必须已经加载。也就是说，攻击者不能凭空让程序拥有任意类；他通常是在目标应用已有类中寻找可拼接的调用链。

利用通常需要 POP Chain，Property-Oriented Programming，也就是属性导向编程链。攻击者不一定能直接调用危险函数，但可以构造对象属性，让多个类的方法在自动触发过程中一步步传递，最终走到文件删除、命令执行、SQL 查询、模板渲染等危险操作。

核心条件：

1. 用户可控数据进入 `unserialize()`。
2. 相关类在运行时已加载。
3. 存在可被魔术方法或调用链触发的危险操作。
4. 攻击者能控制对象属性影响危险函数参数。

课件中的 Typecho 示例可以概括为：程序从 `$_GET['__typecho_config']` 读取用户输入，先 `base64_decode()`，再传给 `unserialize()`，于是满足“反序列化参数用户可控”。后续构造函数、字符串拼接、`__toString()`、`__get()` 等方法被串起来，构成 POP 链。复习时不需要背完整源码，但要能说出链路分析方法：找入口、找自动触发方法、看属性是否可控、追踪危险函数参数。

### Web 安全的统一理解

Web 漏洞大多不是因为 HTTP、HTML、SQL 本身邪恶，而是因为应用把边界混淆了：把用户输入当成代码，把客户端状态当成可信事实，把文件名当成安全路径，把序列化字符串当成可信对象。复习 Web 安全时，可以统一问四个问题：

1. 这个数据从哪里来？
2. 这个数据进入了哪个解释器或状态管理机制？
3. 服务端有没有校验、编码、鉴权或隔离？
4. 攻击者能否改变语义、身份、路径或执行内容？

HTTP 是无状态协议，所以登录态需要 Cookie 和 Session 维持。安全问题也常围绕状态展开：SessionID 被窃取会造成会话劫持，Cookie 缺少 HttpOnly 会被 XSS 更容易读取，缺少 SameSite 会增加 CSRF 风险，服务端过度相信客户端参数会造成越权。

### GET、POST 与安全误区

GET 和 POST 是请求语义差异，不是安全等级差异。GET 参数出现在 URL 中，容易进入浏览器历史、日志和 Referer；POST 数据放在请求体里，不等于加密，也不等于不能伪造。真正的保密依赖 HTTPS，真正的授权依赖服务端鉴权。

GET 常用于查询资源，POST 常用于提交或修改资源。若一个修改密码、删除数据、转账等敏感操作用 GET 触发，就更容易被链接、图片标签或跨站请求诱导触发。即使用 POST，如果没有 CSRF token 或 SameSite 等保护，也仍可能被跨站请求利用。

### 文件上传漏洞再理解

文件上传漏洞的危险不只是“上传了坏文件”，而是“上传后的文件被服务器解释执行”或“上传文件能参与后续攻击链”。攻击者上传 PHP/JSP/ASP 脚本，如果服务器把上传目录当作脚本目录执行，就形成 WebShell。即使不能直接执行，也可能通过文件包含、解析差异、图片马、压缩包路径穿越等方式触发。

安全上传要同时检查三个面：

- 内容面：文件魔数、真实类型、重新编码、病毒扫描。
- 路径面：文件名重命名、路径规范化、防止 `../`、存储到 Web 根目录外。
- 执行面：上传目录禁脚本执行，最小权限，静态资源域名隔离。

只检查后缀不可靠，因为后缀可伪造；只检查 MIME 也不可靠，因为 MIME 来自客户端或可被伪造。更安全的做法是服务端重新生成文件，例如图片重新解码再编码。

### XSS 再理解

XSS 的本质是输出位置不安全。用户输入被放回页面时，如果没有根据上下文编码，浏览器会把它当作 HTML 或 JavaScript 执行。

不同上下文需要不同编码：

- HTML 文本节点：转义 `<`、`>`、`&` 等。
- HTML 属性：还要处理引号。
- JavaScript 字符串：按 JS 字符串规则转义。
- URL 参数：做 URL 编码，并限制协议。

反射式 XSS 常见于搜索结果、错误提示、跳转参数；存储式 XSS 常见于留言、评论、个人资料、后台配置；DOM 型 XSS 常见于前端脚本把 `location.hash`、URL 参数等直接写入 DOM。

XSS 的危害取决于用户身份和页面能力。攻击管理员后台的 XSS 可能比普通用户页面严重得多，因为脚本能以管理员身份发请求。HttpOnly 可以降低 Cookie 被读取的风险，但不能阻止脚本冒用当前页面身份发送请求。

### SQL 注入再理解

SQL 注入的关键是数据库看到的最终 SQL 语义被用户改变。比如程序本来想查：

```sql
select * from users where username='bob' and password='abc123'
```

如果输入被直接拼接，攻击者可让条件变成恒真、追加查询、注释掉后续条件，或利用报错、布尔差异、时间延迟推断数据。

常见注入类型：

- 报错注入：利用数据库错误回显信息。
- 联合查询注入：用 `union select` 拼接额外结果。
- 布尔盲注：根据页面真假差异推断。
- 时间盲注：根据响应延迟推断。
- 堆叠查询：一次提交多条语句，取决于数据库和驱动支持。

防御首选参数化查询，因为它让 SQL 结构和参数数据分离。输入过滤可以作为辅助，但不能替代参数化，因为黑名单容易绕过，编码、注释、大小写、函数和数据库差异都会造成漏网。

### 文件包含再理解

文件包含漏洞发生在“包含哪个文件”由用户控制。危害有两类：读取敏感文件，或执行可控内容。

本地文件包含可读取 `/etc/passwd`、配置文件、源码、日志等。如果攻击者能让恶意内容进入日志，再包含日志文件，日志中的代码可能被解释执行。若应用允许上传文件，攻击者可先上传看似合法但含脚本片段的文件，再通过包含漏洞执行。

远程文件包含要求服务器允许包含 URL 资源。其风险更直接：攻击者把恶意脚本放在自己的服务器，目标应用包含后执行。现代环境通常关闭相关配置，但复习时要理解配置项和路径控制的关系。

PHP 伪协议常用于扩展攻击面。`php://filter` 可把源码 base64 编码后返回，帮助读取源码；`php://input` 可读取请求体；`file://` 访问本地文件。防御时不能只过滤几个关键词，而要从设计上取消用户控制 include 路径。

### 反序列化再理解

反序列化漏洞特别容易被低估，因为入口看起来只是“恢复对象”。真正危险在于恢复对象时，程序可能自动调用魔术方法；对象属性又可能被攻击者控制；多个类的方法调用能连成 POP 链，最终走到文件删除、命令执行、SQL 查询、模板渲染等危险操作。

利用条件通常包括：

1. 反序列化数据可控。
2. 相关类在运行时可用。
3. 存在可利用的魔术方法或调用链。
4. 对象属性能影响危险函数参数。

防御反序列化最好的方式是不对不可信数据使用原生对象反序列化。如果必须使用，应限制类型、签名校验、最小化魔术方法副作用，并避免在析构、唤醒、字符串转换等自动触发方法中做敏感操作。

### Web 漏洞对比表

| 漏洞 | 根因 | 典型后果 | 首选防御 |
|---|---|---|---|
| 文件上传 | 上传内容可执行或可被利用 | WebShell、代码执行 | 白名单、重编码、上传目录不可执行 |
| XSS | 输出未按上下文编码 | 会话劫持、冒用身份操作 | 输出编码、CSP、Cookie 安全属性 |
| SQL 注入 | 输入拼接进 SQL | 数据泄露、认证绕过 | 参数化查询 |
| 文件包含 | 用户控制包含路径 | 读文件、代码执行 | 固定映射、禁远程包含、路径限制 |
| 反序列化 | 不可信数据恢复对象 | POP 链、代码执行 | 不反序列化不可信数据、签名和类型限制 |
